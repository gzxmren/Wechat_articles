graphify：把 Karpathy 的想法做成工具的开源项目

最近看到一个开源项目叫 graphify[1]，做的事情挺让我眼前一亮——它把 Andrej Karpathy 公开说过想要的一个工作流，直接做出来了。

Karpathy 的想法

Karpathy 之前提过，他自己有一个 /raw 文件夹，里面什么都往里扔——论文 PDF、推文截图、随手记的笔记、代码片段。但问题是：这些东西丢进去容易，想找的时候很难。LLM 的上下文窗口再大，也不可能每次把整个文件夹读一遍。

graphify 就是来解决这个问题的。

它干了什么

简单说，把任意文件夹变成可查询的知识图谱。

你把代码、文档、论文、截图、白板照片往一个文件夹一丢，运行一条命令，它就帮你：

  1. 提取里面的概念、实体、关系
  2. 用图的结构把它们连起来
  3. 自动聚类，找到核心节点（它叫 "God Nodes"）
  4. 输出一个可交互的 HTML 图谱 + 一份结构化报告

然后你就可以直接向 AI 提问，AI 不用再从原始文件里搜，直接从图谱里找答案。官方数据：查询时的 token 消耗降低 71.5 倍。

为什么这项目有意思

几个点让我觉得它不只是又一个 "AI wrapper"：

代码不走 LLM。  代码文件用 tree-sitter 做 AST 解析，提取类、函数、调用关系、docstring，完全本地处理，不花 token 也不泄露代码。

每条关系都有置信度。  标记为 EXTRACTED（直接从源文件提取）、INFERRED（推理得出，带置信度分数）、AMBIGUOUS（不确定）。你知道哪些是实打实找到的，哪些是模型猜的。

增量更新。  有 SHA256 缓存机制，改了哪些文件就只重新处理哪些，不用每次全量跑。

支持图片。  截图、流程图、白板照片、甚至其他语言的图片，都能用 Claude 的视觉能力提取信息。对"什么都往文件夹里扔"的用法特别友好。

怎么用

pip install graphifyy && graphify install

然后在 Claude Code 里：

/graphify ./raw

就完了。它会生成一个 graphify-out/ 文件夹，里面有：

  • graph.html — 交互式图谱，可以点、搜、按社区筛选
  • GRAPH_REPORT.md — 核心节点、意外关联、建议问题
  • graph.json — 持久化的图谱数据，下次查询直接用

还支持 --watch 模式，文件改了自动更新图谱；支持 git hooks，每次 commit 自动重建。

跟手动整理知识库比

我现在用笔记工具 + Claude 维护知识库，三层结构：Raw Sources → Wiki → Schema。手工摄入、手工建概念页、手工维护索引。好处是控制力强，适合长期积累。

graphify 的思路更暴力：不用手动整理，它自动从原始材料里抽概念和关系，自动聚类，自动发现关联。代价是对结构的控制力没那么细，但省事非常多。

两种思路各有场景。需要精细控制和长期积累，手动模式更好；需要快速理解一个陌生代码库或一堆散乱资料，graphify 更直接。

值不值得试

如果你有以下任何一个场景，我觉得值得试：

  • 手上有一堆论文/文档/代码，想快速理清它们之间的关系
  • 经常需要让 AI 理解一个大型代码库的架构
  • 习惯把各种资料丢到一个文件夹，但每次找东西都很痛苦
