导读： 一个人，不写一行代码，每天交付10到30次产品更新。他做的工具现在贡献了全球4%的代码量。这不是科幻，是 Anthropic Claude Code 负责人 Boris Cherny 的日常。如果你想过"我有个好想法但不会编程"，这篇会告诉你：这个障碍正在消失，而且消失的速度比所有人预想的都快。更重要的是，当编程不再是门槛，做产品的整套逻辑都反过来了——你以为对的，几乎全错了。

大多数人对"编程被AI取代"的理解是错的。

他们以为这是一个渐进的过程——AI先帮你写写小脚本，再帮你改改小毛病，最后可能在某些简单任务上替代初级程序员。一个缓慢的、可预期的、给你足够时间调整的过渡。

不是这样的。

Boris Cherny，Anthropic 旗下编程工具 Claude Code 的负责人，从2024年11月起没有亲手写过一行代码。每天往公司代码仓库里提交10到30次代码更新，全部由AI完成。

这个人在 Instagram 时期就是公司里产出最高的几个工程师之一。他不是不会写代码。恰恰相反——他太会写了，所以才知道自己可以不写了。

如果你觉得这只是一个顶级工程师的个人体验，再看数据：行业分析机构统计，全球最大的代码托管平台 GitHub 上，4%的公开代码提交来自 Claude Code，企业内部的比例更高。Anthropic 工程团队扩大约4倍，人均产出提升了两倍。Spotify 声称其最优秀的开发者自2024年12月以来没写过一行代码。

这不是趋势。这是已经发生的事。

但"已经发生"这几个字太轻了。让我说说这件事更深的含义。

一、能力越强的人，越容易被过去锁定

有一次团队碰到一个棘手的程序故障——系统占用的内存越来越多，迟早会崩。Boris 按老办法上手：一步步排查、定位、分析。一个新来的工程师走了另一条路：直接把问题丢给 Claude Code。

Claude 自己做了排查、写了分析工具、找到问题根源、提交了修复方案。比 Boris 更快。

老手的经验反而成了包袱——你已经知道怎么解决问题了，所以压根不会想到让AI试试。

这是一个残酷的悖论：你在旧世界里越成功，在新世界里越危险。

Boris 说"编程已经被解决了"。这不是一句口号，他有三层论证。

第一层，自己的亲身体验。 这不是AI帮新手写几行简单脚本。是一个顶级工程师的全部日常编程工作被接管。他加了限定——产品开发、工具制作、系统搭建这些方向。但他也预测"几个月内行业大部分编程工作都会越来越被AI覆盖"。

第二层，数据在加速。 2025年2月 Claude Code 写了大约20-30%的公司代码，5月约30%，11月超过100%。不只是在涨，涨的速度本身也在加快。

第三层，放到编程的历史里看。 Boris 的祖父是苏联第一批程序员，在纸带上打孔编程。编程这件事的演变是：人工计算→拨开关→打孔卡→写代码→用人话指挥AI。每一次转变，上一代的"编程方式"都变成了看不见的底层零件。

亲手写代码即将变成AI生成代码的底层零件，就像今天没人需要知道CPU里每个开关怎么拨。

这不是第一次发生。但可能是最后一次。

不过有一条线得画清楚。"被解决"不是什么都解决了。代码的写被解决了，代码的验还在路上。目前 Anthropic 所有的代码更新都由 Claude 做第一道检查，但人类审查仍然是最后一关。

Boris 说了一句值得记住的话：靠"懂底层技术"建立优势的窗口正在关闭，而且是按月在收缩。

但更有意思的是情感层面。Boris 说他比以往任何时候都更享受编程——那些琐碎的部分（版本管理、装依赖库、查bug细节）终于可以甩给AI了，留下的是真正有意思的部分：想出要造什么、跟用户聊、思考大方向。

团队里有个叫 Lena 的工程师，周末仍然自己手写底层代码。Boris 的态度是"永远有空间享受手艺本身"。

这里有一个同时成立的双重事实：编程作为手艺不会消失，但编程作为一种必须由人来干的苦活正在消失。就像今天仍然有人手写毛笔字，但没人因此觉得打印机没用。区别在于：选择手写是热爱，被迫手写是劳动。

好，编程被解决了。但 Claude Code 是怎么走到今天这一步的？这个故事本身就说明了AI时代做产品的逻辑和过去完全不同。

二、两个赞到全球4%的代码：产品增长靠的不是功能，是AI本身在变强

Boris 加入 Anthropic 后先花一个月做各种原型试验，再花一个月研究AI模型到底能干什么、不能干什么。然后他做了一个极其简陋的命令行工具——就是那种黑底白字、打字交互的界面。

关键时刻是他给AI一个能执行电脑命令的工具，随口问了句"我现在在听什么音乐？"AI在没有任何提示的情况下，自己摸索出了获取答案的路径——它去查了电脑上正在播放的内容。没人教它，它自己想出来了。

他在公司内部发布这个原型时，只得到了2个赞。

所有人觉得编程工具应该像 VS Code 那样有漂亮界面的编辑器，谁会想在一个黑底白字的窗口里让AI写代码？

但 Boris 坚持这条路。理由只有一个：AI进步的速度太快了，任何花哨的界面都跟不上。漂亮的编辑器每次AI升级都得重新适配界面，而黑底白字的命令行几乎不需要改——产品的核心就是AI本身，壳子越薄越好。

后面发生的事证明了他是对的。

第一阶段：冷启动。 2025年2月对外发布，不是一夜爆红。很多人不知道这东西是干嘛的、怎么用。

第二阶段：AI换代，一切起飞。 2025年5月，Anthropic 发布了新一代AI模型 Opus 4。之前的模型只能独立干15-30秒就跑偏，Opus 4 可以独立运行10-30分钟不出错。这不是快了一点，是性质变了——AI从"需要你一直盯着"变成"你交代完就可以走开了"。用户量在这个节点直线上升。

第三阶段：加速度本身在加速。 日活用户最近一个月翻了一倍。预测到2025年底，全球五分之一的代码提交会来自这个工具。

每次AI模型升级，用户量就跳一个台阶。

这就是AI产品和传统软件产品最根本的区别：传统软件靠加功能、做运营来增长；AI产品靠AI本身变强来增长。 产品团队的工作不是"做更多功能"，而是确保产品能把AI的能力原原本本地递给用户——AI每强一分，产品自动强一分。

这也是为什么 Boris 选了最简陋的命令行而不是漂亮的编辑器：壳子越薄，AI升级时需要改的东西越少，升级红利就越大。

但光靠AI变强还不够。Boris 从最早期就建立了极快的反馈循环——用户报一个问题，他几分钟内就修好。用户觉得"被听到了"→愿意提更多反馈→产品改得更快→吸引更多用户。像滚雪球。

现在 Claude Code 甚至能自己读用户反馈、找出bug、提出修复方案、直接提交代码更新。产品在用自己改进自己。

这个滚雪球的领先优势是时间积累出来的——内部用了4-5个月才对外发布，加上一年多的用户反馈数据。这不是对手能一夜追上的。

Boris 用了一个关键词："指数思维"。Anthropic 三位联合创始人是AI领域最重要的"规模定律"论文前三作者——他们就是那群最早用数据证明"AI规模越大能力越强"的人。指数思维刻在公司骨子里。

2025年5月 Boris 预测"年底大家就不需要传统代码编辑器了"，全场倒吸一口气。他说"我只是把已有的趋势线往前画了画而已"。

用直线思维去理解指数增长，就像站在海边看地平线——眼前是平的，但你站错了尺度。地球是圆的，只是你站得太低看不出来。

那么，当编程被AI搞定、产品增长靠AI自己变强之后，做产品这件事的逻辑怎么变？

三、六条反常识：放手比控制更有效

Boris 的产品哲学，和传统做产品的思路几乎完全反着来。六条原则，每一条都违反直觉。

第一条：产品就是AI本身，不是AI的包装。

传统做法是把AI当成系统里的一个零件，在外面搭一堆交互流程、操作步骤、引导界面。Boris 反过来——产品就是AI，外面只加一层最薄的壳。

他的解释很形象：AI有自己擅长的做事方式，你非要强迫它按你设计的步骤走，就像逼一个厨师必须先切葱再切姜——他被束缚住了，反而做不好菜。让AI按它自己最自然的方式干活，效果最好。

精心设计的多步骤流程能提升10-20%的效果，但下一代AI模型一出来，这点提升直接被抹平。Claude Code 只给AI几个基础工具让它自己决定怎么组合，反而每次AI升级都自动变强。

第二条：故意给团队不够的人。

人手不够不是为了省钱，是一种筛选机制。如果一个想法需要20个人花半年才能验证，这想法本身可能就不好。人不够用的时候，唯一的出路就是让AI多干活。Claude Code 最早就是 Boris 一个人写的，因为团队就他一个人。

第三条：为6个月后的AI能力做产品。

故意接受产品现在和市场不太匹配，赌的是6个月后AI够强了就对上了。Boris 原话："头六个月你会很难受，因为产品和市场需求对不上。"

他能这么赌，是因为在 Anthropic 内部能看到AI的进步曲线。但他也给了一个所有人都能用的判断方向：AI会越来越擅长使用各种工具，独立工作时间会越来越长。2025年2月发布时AI只能独立跑15-30秒，Boris 已经在为能跑10-30分钟的AI设计产品——等到新模型发布，产品和AI能力刚好对上。

第四条：看用户和AI的"不正经"用法。

用户拿你的产品做了你压根没设计过的事——那就是最强烈的需求信号。

第五条：给工程师不限量的AI使用额度。

最有价值的创新来自那些看起来太疯狂的想法。如果用AI还得打报告审批，工程师就不会去试那些"大概率没用但万一有用就改变世界"的实验。

第六条：觉得没准备好就对了，赶紧发布。

AI产品有个特殊之处：你连自己的产品能干什么都不完全知道，因为AI的行为本身就有不可预测性。在实验室里你永远无法穷尽用户会怎么用它。Claude Code 内部用了4-5个月才对外发布，那4-5个月不是在打磨功能，是在确保安全——安全过关后立刻推出，管它粗不粗糙。

六条原则的共同逻辑是放手而不是控制——放手让AI决定怎么干、让小团队自己摸索、接受产品粗糙、让用户用你想不到的方式折腾。

这和过去20年做产品的套路完全对立。过去的核心假设是"我们比用户更懂产品应该是什么样"。AI时代倒过来了——AI比你想的更能干，用户比你想的更有创造力。

但"放手"有三条不能放的线：安全（Anthropic 存在的根本原因）、听用户声音的速度（滚雪球的核心齿轮）、对用户行为的观察（放手的是"怎么做"，抓紧的是"做什么"和"做得对不对"）。

那第四条——看用户的"不正经用法"——到底在实际中长什么样？这可能是六条里最有颠覆性的一条。

四、用户在"滥用"，AI在"脱缰"——这才是真正的需求信号

Facebook 有两个经典案例。40%的群组帖子其实是买卖交易——用户自己把社交工具硬生生改造成了二手市场，在帖子标题写价格、评论里留联系方式。60%的陌生异性主页浏览其实是在相亲。于是有了 Facebook Marketplace 和 Dating。

不是用户调研告诉你的。是用户用脚投票的。

Boris 在 Claude Code 上看到了同样的事，但分三层。

第一层：非程序员闯了进来。 数据分析师 Brendan 自己动手装了一堆程序员才用的工具，下载 Claude Code，硬是学会了打开那个黑底白字的窗口——就为了让AI帮他做数据查询。下一周，整个数据团队都在做同样的事。一个人跳过障碍是好奇心，一群人跳过障碍就是刚需。

第二层：外部用户的奇葩用法。 种番茄、分析基因、恢复坏硬盘里的婚纱照、看核磁共振片子——全都在用一个本来是给程序员写代码用的工具。用户用行为说的是：我不想要一个聊天伙伴，我想要一个能替我干活的东西。

第三层：半年下来，数据说话了。 大量用户持续用编程工具做非编程的事。这不是噪音，是信号。

三层信号叠在一起，产品方向一目了然：做一个不需要程序员背景、但能替你操作电脑的产品。最终方案出人意料地简单——"把 Claude Code 放进一个普通人能用的桌面应用"。10天做出来，上线后增长远超 Claude Code 早期。

但 Boris 在这里提出了一个更深层的洞察——不只是用户有"你没想到的需求"，AI自己也有你没想到的用法。

传统做AI产品的思路是先设计好流程，再把AI塞进去。Claude Code 反过来——给AI一堆工具，不告诉它怎么用，然后看它自己会怎么做。

就像那个"你在听什么音乐"的例子：没人教AI该怎么查，但你给它一个能执行电脑命令的工具，它自己就摸索出了答案。因为AI在海量数据中见过无数人类使用工具的方式，给它工具时它会综合出最"顺手"的用法——往往比产品经理提前设计好的流程更管用。

如果说用户的隐藏需求是通过"滥用"暴露的，那AI的隐藏能力就是通过"脱缰"暴露的。少给约束，你才能看见它真正想做什么、擅长做什么。

传统做产品的顺序是：先想清楚用户是谁→他们有什么痛点→设计方案→找技术实现。Boris 的顺序倒过来：先看AI现在能干什么→看它自己最自然的做法是什么→再去匹配人的需求→套上最薄的壳。

这里有一个后续问题：当产品越做越大，观察谁还能做？Boris 其实已经走到了下一步——让 Claude 自动阅读用户反馈、识别问题规律、提出修复方案。但最有价值的信号，往往最不像信号。

Brendan 在那个黑窗口里做数据查询——他是怎么绕过那些障碍的？当时他脸上是沮丧还是兴奋？——这种东西你从数据报表里看不到。你能看到用户在增长，但看不到一个数据分析师为了用程序员工具做自己的活儿，愿意花半天自学安装各种东西时那股较真劲儿。

判断什么东西值得当成信号，这件事本身还是得靠人。

而 Boris 描述的 Cowork 使用场景——交罚款、管项目进度、回邮件——更像是"私人助理"而不是"软件"。但AI助理和真人助理有个本质区别：它可以同时帮无数人干活，而且干得越多成本越低。Boris 自己同时跑5个AI助理，真人助理模式下想都不敢想。

更准确地说，这是一个全新的东西：像管家一样懂你要什么、灵活应变，又像水电一样可以无限供应、随用随开。

这意味着付费方式可能更像交电费（用多少付多少）而不是像买软件（按年订阅）——你不是在为一个"产品"买单，而是在为一种持续运转的"能力"买单。

编程被解决了，产品逻辑变了，一种全新的东西出现了。那在这个新世界里，人变成什么角色？

五、Builder：当"会写代码"不再是入场券

Boris 说到今年底"每个人都是产品经理，每个人都能编程"。"软件工程师"这个头衔会被"Builder"（建造者）取代。

但"人人都是产品经理"不等于产品经理消失了。产品经理过去的专属领地是"把用户需求翻译成工程师能懂的语言"——这个活被AI干掉了。剩下的是比较谁还有别的本事。设计、产品、工程之间的界限越来越模糊。

Boris 描述的最好用的 Builder 长什么样：一个工程师同时有审美、有商业嗅觉、喜欢跟用户聊天、能跨领域思考。Claude Code 团队里所有人都写代码——产品经理写、工程主管写、设计师写、连财务都写。

更关键的是，编程被解决之后，Claude Code 自己正在进入一个新阶段：帮人想出应该做什么。Boris 把 Claude 指向一个用户反馈频道，Claude 会自动读反馈、找出问题、提出修复方案，甚至直接提交代码。

这不只是"AI帮你写代码"了，而是"AI理解问题、提出方案、动手解决"的完整闭环。AI正在从工具变成同事。

这对产品经理来说是一个大信号：如果AI能直接从用户反馈中挖出需求、排好优先级、自己去实现，那产品经理的核心价值就不再是"翻译需求"，而是"拍方向"——决定做什么不做什么，以及为什么。

Builder 需要四项核心本事：

  1. 对用户的直觉——从行为中嗅到还没被说出口的需求。你能不能看到 Brendan 费劲装各种工具只为用AI做数据分析时，那股掩饰不住的兴奋？
  2. 对AI能力的手感——知道AI现在能干什么、6个月后大概能干什么。Boris 花了一个月纯研究AI模型才开始做产品。了解AI不是看说明书，是自己动手跟它磨合。
  3. 看大局的能力——不是"这个功能怎么做"，而是"这个方向走下去会怎样"。
  4. 自己动手验证的能力——自己用AI把想法做成原型。不用等工程师排期，你自己就能试。

其中"了解AI能干什么"是个不断贬值的技能——因为AI的边界在飞速移动。"理解人要什么"才是持续增值的技能——人的根本需求变化很慢。

并非所有原则都适合每个团队。观察用户的"滥用"行为、故意少给人、早发布快迭代——这些到哪儿都能用。为6个月后的AI做产品——可以看公开的技术路线图替代内部信息，没那么精确但方向差不多。但"产品就是AI的薄壳"——这对用别人家AI的公司来说很危险，因为AI供应商自己做个同类产品就能干掉你。

"别把AI关在笼子里"（使用态度）和"产品就是AI的薄壳"（商业策略）是两回事。前者谁都适用，后者只有自己拥有AI的公司才成立。
