从构建Claude Code中汲取的教训

引言：智能体工程的核心挑战

在人工智能领域，构建一个真正有效的智能体（Agent）系统远比想象中复杂。当我们谈论大语言模型（LLM）时，往往容易忽视一个根本性问题：如何为模型设计一套恰当的动作空间（Action Space）——即模型能够执行的操作集合。这正是 Anthropic 工程师 Thariq 在其关于 Claude Code 的深度分享中所揭示的核心命题。

Claude Code 是 Anthropic 推出的 AI 编程助手，它不仅仅是一个简单的代码补全工具，而是一个能够自主执行复杂编程任务的智能体系统。在构建这个系统的过程中，团队经历了一系列艰难的取舍和迭代，最终形成了一套独特的设计哲学。本文将深入剖析这篇文章的技术细节，为读者呈现智能体工程的核心要义。

第一部分：动作空间设计的基本框架

1.1 什么是动作空间？

在强化学习和智能体系统中，动作空间指的是智能体能够执行的所有可能动作的集合。对于基于大语言模型的智能体而言，动作空间主要体现为工具（Tools）或函数调用（Function Calling）的能力。

以 Claude 为例，它通过“工具调用”机制与外部世界交互。开发者可以在 Claude API 中定义各种工具，这些工具本质上是对外部功能的封装——可能是执行一段 Bash 命令、运行 Python 代码、读取文件系统，或是调用某个外部 API。关键问题在于：我们应该如何设计这些工具？应该提供多少工具？每个工具应该具备怎样的粒度和功能？

1.2 作者的比喻：数学问题的求解

作者提出了一个极为精妙的类比：假设你面对一道困难的数学题，你会希望拥有什么工具来解决问题？

  • 纸笔：这是最基本的要求，你可以通过手动计算来解题，但效率受限

  • 计算器：比纸笔更强大，能够处理更复杂的运算，但你需要了解高级功能的使用方法

  • 计算机：最强大且高效的选择，但你必须懂得编写和执行代码

这个比喻揭示了工具设计的核心悖论：工具越强大，对使用者的能力要求就越高。同样地，为 AI 智能体设计工具时，我们需要考虑模型本身的能力边界——给予它过于简单的工具会限制其能力，而过于复杂的工具则可能导致模型无法有效利用。

1.3 设计的核心原则

作者指出，设计智能体工具的关键在于：给予与其能力相匹配的工具。这听起来简单，但实际操作中需要通过持续观察模型的输出、进行大量实验来逐步了解模型的能力边界。正如作者所说：“你需要学会像智能体一样去观察。”

第二部分：AskUserQuestion 工具的演进之路

2.1 背景：提升智能体的“征询”能力

在 Claude Code 的设计中，一个重要的挑战是如何让模型更好地向用户提问——这在 AI 领域被称为征询（Elicitation）。当智能体遇到不确定的情况时，它应该能够向用户请求额外的信息或确认。然而，如果这个过程过于繁琐，用户体验将会大打折扣。

2.2 尝试一：修改 ExitPlanTool

工程师们的第一个尝试是在 **ExitPlanTool **中添加一个参数，使其能够同时接收计划（plan）和问题列表。ExitPlanTool 原本是 Claude Code 中用于在计划模式下让用户确认或退出的工具。

这个方案虽然实现简单，但存在根本性的逻辑问题：模型同时被要求提供计划和对计划的质询，这两者之间可能存在冲突。如果用户的回答与原计划相矛盾，模型应该怎么办？是否需要调用两次工具？这种设计上的模糊性导致该方案被放弃。

2.3 尝试二：修改输出格式

第二个尝试是修改模型的输出指令，使其能够以特定的 Markdown 格式来呈现问题。例如，要求模型输出带选项的列表，用户可以在方括号中选择。

这种方法的优点是通用性强——模型基本能够按照要求输出。然而，问题在于这种输出方式无法被严格保证：模型可能会添加额外的句子、遗漏某些选项，或者使用完全不同的格式。对于需要精确解析的 UI 交互来说，这种不可预测性是致命的。

2.4 尝试三：专门的 AskUserQuestion 工具

最终方案是创建一个专门的 AskUserQuestion 工具，模型可以在任何时候调用它，特别是在计划模式下。当工具被触发时，系统会显示一个模态窗口来呈现问题，并暂停智能体的工作循环，直到用户回答为止。

这个工具的成功在于几个关键因素：

  • 结构化输出：通过工具定义，模型被迫产生结构化的输出

  • 多选项保证：工具设计确保模型会提供多个选项供用户选择

  • 灵活性：用户可以在 Agent SDK 中调用此工具，或在 skills 中引用它

  • 模型友好：最重要的是，模型似乎乐于调用这个工具，其输出效果也很好

作者特别强调了一个重要教训：即使是最精心设计的工具，如果模型不理解如何使用它，也是徒劳的。工具的设计必须与模型的认知能力和使用习惯相匹配。

第三部分：从 Todo 到 Task：能力的演进与工具的淘汰

3.1 Todo 的诞生

在 Claude Code 最初发布时，团队意识到模型需要一个待办事项列表（Todo List）来保持任务的追踪。通过 TodoWrite 工具，模型可以创建、更新待办事项，并在完成工作时将其勾选。

这个设计符合直觉：人类在处理复杂任务时尚且需要待办清单，AI 模型自然也需要。

3.2 系统的困境

然而，实践中的效果并不理想。即使有了 TodoWrite 工具，模型仍然经常忘记自己需要完成的任务。团队不得不在每 5 轮交互后插入系统提醒，来重新强调模型的目标。

这反映出一个深层问题：工具的频繁干预实际上可能干扰模型的自主思考。

3.3 能力的跃升与工具的过时

随着模型能力的提升，情况发生了戏剧性的变化。新一代模型不仅不再需要 Todo 提醒，反而觉得这种提醒是一种限制。当系统不断提醒它待办事项时，模型会产生一种“必须严格遵守清单”的错觉，而不是根据实际情况灵活调整任务。

同时，团队观察到 Opus 4.5 版本在子代理（subagent）协调方面有了显著提升。但这里出现了一个新问题：子代理之间如何在共享的 Todo List 上协调工作？

3.4 Task 工具的崛起

基于这些观察，团队用 Task 工具取代了 TodoWrite 工具。两者的核心理念不同：

3.5 关键洞见

这一部分揭示了一个重要原则：随着模型能力的增长，曾经必要的工具可能反而成为限制。设计师需要不断回顾和重新评估之前的假设，根据模型的实际表现调整工具设计。这也是为什么作者建议使用能力谱系相近的少量模型——这样可以更有效地管理工具集的兼容性。

第四部分：搜索接口的设计与渐进式披露

4.1 上下文构建的重要性

对于 Claude Code 这样的编程助手，能够搜索和构建上下文是核心能力之一。模型需要理解代码库的结构、找到相关的函数和类、了解项目的依赖关系，才能提供有效的编程辅助。

4.2 RAG 的局限

Claude Code 最初发布时，团队使用了 RAG（检索增强生成）向量数据库来为模型提供上下文。RAG 是一种成熟的技术，它通过将文档转换为向量并在向量空间中搜索，来找到与查询最相关的内容。

尽管 RAG 功能强大且速度快，但它存在几个问题：

  • 需要预先建立索引，增加了系统复杂性

  • 在不同环境中可能不够稳定

  • 最关键的是：上下文是被“给予”模型的，而不是模型自己“找到”的

4.3 赋予模型搜索能力

团队意识到：如果模型能够在互联网上搜索，为什么不能让它搜索代码库？通过给予模型 Grep 工具（一种 Unix/Linux 下的文本搜索工具），模型可以自行搜索文件、定位代码片段、自主构建上下文。

这是一个范式转变：从被动接收上下文，到主动寻找上下文。作者观察到，随着模型变得愈发智能，只要给予恰当的工具，它在构建自身上下文方面的能力会显著提升。

4.4 渐进式披露模式

基于这一洞见，团队引入了渐进式披露（Progressive Disclosure）的概念。这一设计模式的核心思想是：不一次性向模型展示所有信息，而是让它通过探索逐步发现相关信息。

具体实现方式是：模型可以读取“skill”文件，而这些 skill 文件又可以引用其他文件，模型可以递归地读取这些被引用的文件。这种设计带来几个好处：

  • 避免一次性输入过多无关信息

  • 允许模型按需获取相关信息

  • 减少上下文窗口的浪费

一个常见的 skill 用法是赋予模型额外的搜索能力——例如，告诉模型如何使用某个 API 或如何查询数据库。随着时间推移，Claude 从“基本无法自主构建上下文”进化到“能够在多层文件中进行嵌套搜索，找到精确所需的上下文”。

4.5 无需新工具的功能扩展

渐进式披露的一个重要优势是：可以在不增加工具的情况下扩展功能。这是因为增加任何新工具都会增加模型的认知负担——模型需要在众多工具中做出选择。因此，团队设定了很高的“新增工具门槛”。

第五部分：Claude Code Guide Agent 案例分析

5.1 问题背景

团队注意到一个有趣的问题：Claude 不够了解 Claude Code 本身。如果你问它如何添加一个 MCP（Model Context Protocol）或某个斜杠命令的作用，它无法给出答案。

5.2 最初的尝试

一个显而易见的解决方案是将所有这些信息都放入系统提示（system prompt）中。然而，这个方案被否决了，原因如下：

  • 用户很少问这类问题

  • 这些信息会污染主要的上下文，影响模型的核心工作——编写代码

  • 系统提示会变得臃肿，增加“上下文腐化”的风险

5.3 渐进式披露的再次应用

团队采用了另一种形式的渐进式披露：给予模型一个指向文档的链接，让它可以在需要时加载并搜索这些文档。

然而，这个方案的效果不够理想：模型往往会加载大量搜索结果到上下文中，只是为了找到那个“正确的答案”，这造成了资源浪费。

5.4 Guide Agent 的诞生

最终方案是构建了一个专门的 Claude Code Guide 子代理。当用户向 Claude 询问关于 Claude Code 本身的问题时，模型会被提示调用这个子代理。这个子代理拥有详尽的指令，指导它如何高效地搜索文档以及应该返回什么样的内容。

虽然这个方案并非完美无缺——模型在回答关于自身设置的问题时仍然可能感到困惑——但效果已经显著改善。团队成功地实现了功能扩展，而没有增加新的工具。

第六部分：核心设计哲学总结

6.1 艺术与科学的结合

通读全文，最重要的洞见或许是：设计智能体工具既是一门艺术，也是一门科学。作者坦言，如果读者期望获得一套刚性的规则来指导工具设计，这篇文章并不能满足这个期望。

工具设计高度依赖于：

  • 所使用的具体模型

  • 智能体的目标

  • 模型的运行环境

6.2 实践导向的方法论

尽管没有放之四海而皆准的规则，作者还是提供了一套实用的方法论：

  1. 持续实验：频繁尝试不同的工具设计和配置

  2. 仔细观察：认真阅读模型的输出，理解它如何与工具交互

  3. 保持迭代：不断尝试新方法，根据效果调整设计

6.3 像智能体一样观察

作者最后强调的那句话点明了整篇文章的核心：“学会像智能体一样去观察。”这意味着设计师需要放下人类的直觉假设，真正去理解模型是如何“看待”和“使用”这些工具的。这种观察者视角的转变，是构建优秀智能体系统的关键所在。

结语

Claude Code 的实践经验为 AI 智能体工程领域提供了宝贵的参考。从工具设计到能力演进，从搜索接口到渐进式披露，每一个决策都体现了对模型能力的深入理解和持续观察的重要性。这些教训不仅仅是技术细节，更是一种设计哲学的体现：成功的智能体系统不是一次性设计出来的，而是在与模型的持续互动中迭代演进的。

正如作者所言，核心能力不在于掌握某种“秘诀”，而在于培养一种持续的观察和学习能力——学会像智能体一样去观察世界。
