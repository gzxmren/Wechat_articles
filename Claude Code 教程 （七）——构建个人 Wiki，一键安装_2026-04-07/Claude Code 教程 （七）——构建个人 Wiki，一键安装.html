Claude Code 教程 （七）——构建个人 Wiki，一键安装

点关注不迷路


Karpathy 上周二发了条推文，谈他怎么用 LLM 维护个人知识库。开头那句挺扎眼——"我现在很大一部分 token 消耗，从写代码转到了维护知识。"

我自己也搭了一个。目前里面有 34 个公司条目、31 个主题概念、111 篇结构化研报摘要，加上 12 条经过反复验证的规则和 5 条被数据推翻的"传统智慧"。脱敏之后我把整套架构整理成了一个空模板扔到 GitHub 上：karpathy-claude-wiki，MIT 协议，fork 即用。

这是我第一次用 GitHub 放 demo，自己用着没问题，但没法保证你那边能跑通。所以这篇主要写它能干什么——两个真实场景给你看。具体怎么搭、命令怎么敲、目录结构长什么样，全部写在仓库 README 里了，想试的人自己去那边看。

下面讲四件事：它和普通本地知识库差在哪、它能做到什么效果、为什么是今年才能做、国产模型能不能跑通。

一、它和我之前的 knowledge/ 差在哪

粗看对比，有点眼花缭乱，其实就是从普通知识库——二层的线性架构，变成个人Wiki——多层的漏斗型+网状架构。

左边是我之前的 knowledge/：按公司分子目录，每篇研报摘要追加进 reports.md。文件越积越长，公司之间没有任何连接，AI 想取一条具体信息要么找不到、要么把整个文件塞进上下文。

右边是现在的 wiki：每篇研报是一个独立的 sources/ 页面，公司放 entities/，主题放 concepts/，两边用双向链接（wikilinks）互相引用。再往上是几个结晶层：explorations/ 存查询沉淀、decisions/ 存交易决策、rules.md 存验证 3 次以上的规则、false-beliefs.md 存被数据推翻的"传统智慧"。所有页面按变化速度分层——raw/ 永远不动，sources/ 偶尔改，entities/ 每周改，rules.md 每季度改。

撑起这个架构的是三条原则：

  • 按变化速度分层——慢的和快的不放一起，否则快的会淹没慢的
  • 单一事实来源——同一个事实只在一个地方维护，其他地方引用它
  • 单一入口——所有素材都走统一的"摄入协议"，不靠你"记得规矩"

这样的知识库是一个漏斗型，从快到慢，从离散到总结，长期积累的rule和错误都是最宝贵的经验，值得你反复细品。而且还是网状结构，最初的“原材料”并不会丢失，随时通过link供你查验。不只是你的wiki，也是AI的大脑。



二、它现在能做到什么效果

讲架构没意思，看两个真实场景。

场景一：一份新闻自动改写两家持仓 + 重写一家的核心叙事

昨天我把 Anthropic 4 月 6 日的官方公告（与 Google/Broadcom 签 3.5 GW TPU 合同、run-rate 突破 $30B）丢进 wiki。

AI 自动做了三件事：在 sources/ 创建了一个新页面；跳到我持仓里某存储龙头的追踪表（tracker.md），把这条数据点加进"正面证据"列表；跳到某 CPU 龙头的资料卡（profile.md），把"Anthropic 三栈不含这家公司的 AI 加速器"作为反向证据加进去——并触发了一次核心论点的重写。

旧论点和新论点的对比长这样：

旧论点里有一条线是"这家 CPU 龙头的 AI 加速器抢 GPU 龙头的份额"。Anthropic 三栈选型直接把这条路封死——头部实验室的算力清单里完全没有它。AI 读到这条之后做的不是简单加一条"负面证据"完事——它把那条旧叙事整个砍掉，重写成两条独立成立的腿（CPU 腿 + 代工腿），并在资料卡里建了一个"防错栏"，明确写"以下叙事不构成核心论点的一部分，未来的我看到类似新闻不要再被带回去"。

这一连串动作我没指挥。它读完 wiki 的结构规则之后自己判断的。一份新闻、三个文件被改写、一个核心持仓的论点结构被重组——这是普通知识库永远做不到的。

场景二：一次有价值的讨论变成永久资产

上周我和 AI 讨论了一个问题："光伏 / 锂电 / 新能源车这一轮周期的时间节奏、预警信号和估值规律是什么？" 这是我做了好几年的活儿，所有的观察和教训散落在 flomo 的几百条笔记里，从来没人替我整理过。

它综合了我前几年的 flomo 笔记 + 几篇行业研报 + 几个关键公司的关键时间点，给了一个完整的复盘框架。我只截一段它最后产出的"周期顶部预警信号"清单：

  1. ROIC 异常高——正常 10-20%，爆发期到 40-50%，必然吸引扩产
  2. 设备订单暴增——丝网印刷从 110GW 涨到 300GW 只用了 4 个月，设备商改收款条件要 70% 现金
  3. 龙头主动降速——某光伏龙头取消 HJT 规划、分批投产而非一把猛扩
  4. 买方集体转空——HF想空锂电、多家基金减仓组件逆变器
  5. 企业开始裁员——二线的龙头裁员降capex = 行业参与者认怂
  6. 资金极度抱团 → 新能源无人问津，可能是反转信号

这六条不是它编的。每一条都能在我 flomo 里找到对应的原始笔记，只是我自己从来没有把它们排成一张清单。

我说"存成一篇探索（exploration）"，它就把这次讨论沉淀成了 wiki/explorations/2026-04-cycle-case-study.md，带原始问题、引用的所有出处、完整答案。下次我或者 AI 再遇到任何"周期性行业"的问题，会先调这个文件——直接引用，不重复推演。

每一次有价值的讨论，都在为未来的你存钱。


为什么以前做不了，现在做得了

这个想法其实不新。Andy Matuschak 谈"常青笔记"（evergreen notes）谈了好多年，Roam Research 火过一阵子，Obsidian 的反向链接已经存在好几年。但那一波都没真正跑出来——为什么现在突然成了？

因为三件东西同时齐了，缺一个都跑不起来：

第一是长上下文。百万 token 级别的窗口意味着 AI 一次能塞进整个 wiki 的索引 + 几个相关公司的全量内容 + 你的问题。它做矛盾扫描的时候，必须能同时看到你的假设、你的持仓、你最近的研报摘要。窗口只有几千 token 这事就做不了。

第二是文件系统原生操作。ChatGPT 让你做这事很难——你得不停地复制粘贴，AI 没法直接读写本地文件。Claude Code 这类智能体把这个限制打掉了：AI 直接读写、搜索 markdown 文件，像开发者改代码一样。这意味着摄入、交叉引用、重新整理这些操作可以真正自动化，不靠你自觉。

第三是协议自动化。光有上面两个还不够，你还需要"什么时候触发什么动作"的规则。比如每次摄入新素材都自动检查是否需要更新主题综述，每次研究新公司前都自动读一遍"被推翻的传统智慧"。这些自动行为靠的是 Claude Code 的 Skill（技能）体系——你定义一次规则，它每次都执行。

这三件事 2024 年都还不齐。今年才齐。


三、装这个 wiki：4 个问题，几分钟

最简单的做法：打开 Claude Code，把下面这段话直接复制进去——

请读 https://github.com/Benboerba620/karpathy-claude-wiki 的 README，并按里面的 "AI Auto-Install Protocol" 帮我装一个 wiki。

它会自己读取 README、复制模板、问你 4 个问题：

  1. wiki 装在哪？ 默认 ./wiki/，回车即可
  2. 你的主要领域是什么？ investing / research / reading / writing 四选一（投资人选 investing，无悬念）
  3. 项目根目录有现成的 CLAUDE.md 吗？ 用 Claude Code 的人多半都有
  4. 第一个想追踪的对象？ 公司、书、人都行

整个过程 1-2 分钟，全程中文，不用自己敲命令行。装完之后你的目录会多出这些东西——

项目根/
├── CLAUDE.md              ← 原本的 CLAUDE.md，末尾被追加一段 wiki 协议
├── wiki/
│   ├── _index.json        ← 看到这个文件就说明装对了
│   ├── _schema.md         ← wiki 的"宪法"
│   ├── _log.md            ← 操作日志
│   ├── overview.md        ← 自动生成的总览
│   └── entities/<NAME>/profile.md   ← 等你填的第一个空模板
└── scripts/
    └── wiki_index.py      ← 索引生成脚本

怎么验证装对了：三件事都齐——wiki/_index.json 存在；你的 CLAUDE.md 末尾多了 ## Wiki Protocols (from karpathy-claude-wiki) 这段；wiki/entities/<NAME>/profile.md 里有一个空的模板等你填。

中间 Claude 会问你要不要清掉模板自带的几个 EXAMPLE-*.md 占位文件——这是模板里用来展示页面结构长啥样的示例页，留着不影响后续操作（Claude 会自动识别为占位跳过），删掉更干净，看你偏好。

装好之后，你日常就做这三件事：

剩下的工程细节（结构规则怎么定义、4 个核心协议是什么、卫生检查脚本怎么用），全部都写在仓库 README 里，想搭的去那里看，30 分钟读完。

我把这套安装协议端到端跑过一次完整测试——清空环境，按 README 字面意思走最难的那条路（"已有 CLAUDE.md + 非默认 research 领域 + 第一个示例是个人名"），发现并修了 7 个真实的卡点（比如"模板临时目录被删早导致后续步骤找不到文件"、"粗暴的查找替换让单复数前后矛盾"这种）。所以默认 investing 路径成功率大概 90%，非默认领域 80% 上下，剩下的小问题 README 里都标了怎么手工救一步。如果你装的时候卡在某一步走不下去，把那一步的原文告诉我——我把协议再补一刀，比 star 实在。


四、用国产模型也能跑

国内的研究员很多人不能稳定用 Claude API。所以我专门用 Kimi K2.5 做了一次摄入测试——把同一份 Anthropic 公告原素材丢进去，让它产出一篇结构化摘要页。结果直接看图：

能跑：文件头格式合规、双向链接自动维护、矛盾扫描真的触发了（它主动指出 Anthropic 三栈不含某 CPU 龙头的 AI 加速器）。一次摄入大约 0.05 元人民币，一篇研报 5 分钱。

它也有问题：会发明不存在的公司条目、偶尔轻度幻觉、风格 emoji 化。加固提示词的三条规则 + 我没测的部分（Cline 集成稳定性、长程一致性、批量任务）都写在仓库 README 的"国产模型版本"一节里。谁跑通了告诉我，我把结果加进去。

维基百科用了二十多年、几百万人协作，才建起一座公共图书馆。而我一个人、一个 AI、加上几年攒下的研究笔记，几个晚上就搭出了一座只属于自己的——它知道我读过什么、写过什么、判断过什么、犹豫过什么。

未来某一天你问它一个问题，它从你自己的笔记深处掏出一段你已经忘了写过的话，那一刻你就会明白——这座wiki只属于你。
