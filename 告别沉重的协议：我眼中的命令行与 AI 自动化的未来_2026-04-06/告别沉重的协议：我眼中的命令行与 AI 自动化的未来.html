告别沉重的协议：我眼中的命令行与 AI 自动化的未来

 


这几天，我对着 draw.io 的画布发了很久的呆。

为了画一张“快速排序”的算法流程图，我机械地拖拽着矩形框、连接线，反复调整箭头的指向。这种工作在图形界面（GUI）里显得极其琐碎且低效。那一刻我突然意识到，这种专为人类视觉设计的交互方式，可能正在成为 AI 替我们工作的阻碍。人类不擅长记忆枯燥的代码，所以发明了图标；但 AI 天生就是读着代码和文档长大的，对它而言，图形界面反而是一层隔靴搔痒的翻译件。

于是，我尝试换了一种方式，回到了那个漆黑的、只有光标闪烁的命令行窗口。

把沉重的软件装进一行命令里

在我的测试环境下，我跑通了一个叫 CLI-Anything 的方案。它的逻辑非常简洁：如果你有一个图形化软件（比如 draw.io 或者是 Blender），它能通过 AI 的理解，把软件背后的 API 逻辑硬生生地“拽”出来，封装成命令行。

我记得在尝试给 draw.io 进行“CLI 化”时，后台的 AI 代理花了整整 46 分钟进行代码编写和自动测试。在那大半个小时里，我只是安静地喝着咖啡，看着屏幕上一行行日志翻滚。当最后显示“100% 通过”时，我输入了一行简单的指令，让它画出那个流程图。

结果很有意思。AI 不再需要像人类一样去“寻找”那个矩形按钮，它直接在底层生成了工程文件，然后导出了一个完美的 SVG。这种感觉就像是你不再需要教一个翻译官如何用手势比划，而是直接给了他一份母语对照表。

顺便提一句，不只是本地软件，连网页也在被“命令行化”。我用 OpenCLI 试着在不打开浏览器的情况下，去搜索招聘网站上的职位，或者在 Google

上提问。当我输入 opencli boss search --city 北京 软件开发 时，数据以整齐的 JSON 格式直接跳了出来。没有广告遮挡，没有弹窗干扰，只有核心的信息。

为什么说 CLI 是 AI 的母语

说到这里，我不得不去思考一个很现实的问题：为什么我们转了一大圈，又回到了几十年前的交互方式？

其实逻辑很简单。GUI 是为了弥补人类记忆力的缺陷，而 CLI 则是为了发挥 AI 的长处。我对比过很多数据，发现 AI 在处理命令行时的 Token 消耗量，往往比处理复杂的、嵌套的协议要低得多。

在一个典型的任务测试里，如果使用现在很流行的 MCP（模型上下文协议），AI 需要把工具的名字、参数格式、调用示例全部塞进对话窗口里。每增加一个工具，对话成本就翻倍增长。

其实不是。CLI 聪明的地方在于它的“自解释性”。AI 并不需要一次性学会所有命令，它只需要学会如何使用 --help。当它需要某个功能时，自己去查一下帮助文档，然后即学即用。这种“渐进式披露”的逻辑，让我在实操中节省了大量的 Token 开销。

MCP 与 CLI：黑盒与流水线的对决

最近很多人在争论，到底是 MCP 更好，还是 CLI 更具生命力。在我看来，MCP 更像是一个“黑盒”。所有的调用细节都隐藏在 AI 代理的内部，如果哪一步出错了，我很难在本地复现和调试。

而 CLI 让我最着迷的，是它对“管道符（|）”的天然支持。

这就很有意思了。我可以用一行命令获取 GitHub 上的 Issue 列表，接着用管道符传给过滤程序，筛选出包含“Bug”字样的内容，再传给排序工具，最后直接导出一份 CSV 报表。这就像是一条透明的流水线，每一节都在我的掌控之中。这种组合能力，是目前任何封装好的 AI 协议都难以企及的。

当然，我必须承认，MCP 在权限控制和标准化安装包方面确实有它的优势，特别是在多租户的云端环境下。但对于我这种追求实战效率的人来说，CLI 这种“即插即用”的力量感，实在难以抗拒。

我现在的习惯

折腾了这么多工具后，我并没有陷入非此即彼的选择题。

目前的 AI 工具链进化得很快，像 Claude Code 这样的工具已经开始融合两者的长处。它们不再要求你一次性注入所有协议，而是学会了按需搜索。

如果你也想优化自己的工作流，我有一些很小的建议，不需要什么大刀阔斧的改变，只需要一点习惯的挪移：

  • • 习惯查阅官方 CLI： 像 GitHub 提供的 gh 工具，比在网页上点来点去要强大得多。试着先从登录和查看列表开始。
  • • 尝试一次“管道”操作： 哪怕只是把一个命令的搜索结果保存到文本文件里。
  • • 给 AI 读文档，而不是喂操作手册： 当你让 AI 帮你写自动化脚本时，直接给它 CLI 的 --help 输出，它会比你想象中更聪明。

有些东西虽然古老，但那是因为它们已经足够接近底层逻辑。

在这个满是精美图标的世界里，有时候，最快的那条路，其实就在那个跳动的光标后面。
