如何为Claude Code设计一个好的记忆系统？

Claude Code已经成为我工作中的主力AI工具，我在前文《Coding之外：Claude Code如何充当我的工作助手》也有过介绍。目前，越来越丰富的MCP、SKILLS生态也让Claude Code的能力越来越强大。

但同时，有一个根本性的问题始终存在，Claude Code没有跨会话记忆能力，每次都是从零开始。这意味着，每次新的会话我们都需要通过一种方式去输入想让它了解的上下文，非常繁琐。

我们常采用的解决方案是用本地文件存储，比如放在Claude.md中。但这种方式的缺点也很明显，需要手动维护，可能会遗漏一些重要信息，也无法按需提取所需要的上下文知识，还有文件越写越长会导致token浪费，同时也可能导致注意力分散影响输出的准确性。

正因如此，开始出现一些为Claude Code设计自动化持续记忆能力的开源方案。今天这篇文章，我想基于Claude-Mem（https://github.com/thedotmack/claude-mem）项目来聊聊：如何设计一个好的记忆系统？

记忆系统的核心问题

设计记忆系统，其实核心点是要解决三个问题：

  1. 记什么：需要记忆什么内容？
  2. 何时记：什么时候触发记忆？
  3. 何时查：什么时候检索记忆？

这三个问题虽然看起来简单，但每个背后都有不少设计考量。接下来，我会逐一展开Claude-Mem是怎么解决这些问题的。

记什么：四类数据

首先是记什么的问题。

Claude-Mem把需要记忆的内容分成了四类。

数据类型
触发时机
内容
用途
observations（观察）
每次工具执行后
事实、叙述、概念
细粒度知识记录
session_summaries（会话摘要）
会话结束时
请求、完成、学习
高层次会话总结
user_prompts（用户提示词）
用户提交消息时
原始输入
记录用户意图
sdk_sessions（会话元数据）

项目、状态
会话管理

其中最核心的是observations（观察）。

每次Claude Code执行工具调用（例如读文件、访问网络、运行命令等），都会记录这次工具调用的输入和输出详情，然后把它压缩成一个结构化的observation。

一个典型的observation的结构如下：

<observation>
  <type>discovery</type>
<title>身份验证使用带 Redis 缓存的 JWT 令牌</title>
<facts>
    <fact>JWT 令牌存储在 Redis 中，1 小时过期</fact>
    <fact>令牌刷新逻辑在 src/auth/token-manager.ts:45</fact>
    <fact>Redis 连接池在 src/config/redis.ts:12 配置</fact>
</facts>
<narrative>
    在调试登录问题时，发现身份验证系统使用缓存在 Redis 中的 JWT 令牌。
    令牌管理器实现在过期前 5 分钟自动刷新，防止用户在活动会话期间登出。
</narrative>
<concepts>
    <concept>authentication</concept>
    <concept>caching</concept>
    <concept>jwt</concept>
</concepts>
<files_read>
    <file>src/auth/token-manager.ts</file>
    <file>src/config/redis.ts</file>
</files_read>
</observation>

可以看到这里的设计。它并不是把原始的工具输出直接存下来，而是提取出了几个关键的要素：

  • fact（事实）：高度信息浓缩的知识点

  • narrative（叙述）：和实时相关的完整上下文和因果关系

  • concept（概念）：事实相关的标签，用于分类和搜索

  • files（文件）：涉及的文件路径

那什么样式的事实才是好事实呢？

facts是整个记忆系统里最重要的数据。Claude-Mem对事实的要求非常严格。

首先，每个事实必须是原子化的、可独立理解的陈述。比如上例中 “JWT 令牌存储在Redis中，1小时过期” 就是一个好的事实，而如果提炼成 “系统使用了缓存” 就太模糊了。

其次，事实不能用代词，必须自包含。如 “Redis 连接池在 src/config/redis.ts:12 配置”可以独立理解， 但如果是 “它在第12行配置” 就不行。什么是它？脱离上下文就看不懂了。

最后，事实需要足够的具体，尽量包含文件名、行号、函数名这些细节。 “refresh() 方法在 TokenManager.ts:45 实现自动令牌刷新” 就比 “代码处理令牌刷新” 有用得多。

这几个要求加起来，就让事实具有高压缩比。根据项目中的统计，一个5000字符的工具输出，压缩成5条事实后大约300字符，10 倍以上的压缩比，同时尽量保留了语义的完整性。

会话摘要：更高层次的记忆

observations记录的是每次工具执行的细节，粒度很细。但有时当我们新开启一个会话时，也想知道之前的会话做了什么，所以Claude-Mem也设计了会话层级的记忆。

这就是前文的四种数据类中，session_summaries的作用。每次会话结束时，Claude-Mem会基于整个会话的完整上下文生成一份摘要，主要包括：用户想要什么、研究了什么、学到了什么、完成了什么、下一步的计划是什么。

<summary>
  <request>用户想要实现什么功能</request>
  <investigated>调查了哪些代码和文档</investigated>
  <learned>学到了什么关键信息</learned>
  <completed>完成了哪些工作</completed>
  <next_steps>下一步计划做什么</next_steps>
  <notes>可选的补充说明</notes>
</summary>

observations和session_summaries是独立生成的两种数据类型，互为补充。前者是细粒度的工具执行记录，后者是高层次的会话总结。查询时可以根据需要选择合适的粒度。

何时记：依赖Claude Code Hooks系统

知道了记什么，接下来是何时记的问题。

Claude-Mem依赖了Claude Code Hooks系统，每一次会话的完整生命周期内，它会在不同的关键时刻（Hook点）自动触发记忆操作。会话启动时有SessionStart，用户发消息时有UserPromptSubmit，发生工具调用时有PostToolUse，会话结束时有Stop。

真正关键的是两个：PostToolUse和Stop。

PostToolUse负责捕获每一次工具的执行，将工具详情以及工具调用之后的结果发送到本地启动的一个Worker服务处理。

Stop则负责在会话结束时生成摘要。它会从完整的会话记录中总结出上文讲过的会话摘要。

为了不影响Claude Code主会话的流畅性，Claude-Mem启动了一个后台Worker服务，并初始化了一个独立的Claude Agent实例来异步处理Hook点捕获到的记忆数据。

关于Claude Agent SDK的使用，可以翻阅之前的文章《Claude Agent SDK实战：打造开源版DeepWiki》

这个设计比较关键。主工作流需要全速执行用户任务，如果每次工具调用都同步等待处理observations记录，体验会很差。Claude Agent运行在后台，完全不会阻塞我们和Claude Code的交互。

另外，这个Claude Agent被设计成纯只读模式。它禁用了所有的可能产生副作用的工具，只能分析数据，综合知识，不能执行任何操作。这样就既保证了效率，也确保了运行的安全。

const disallowedTools = [
'Bash',           // Prevent infinite loops
'Read',           // No file reading
'Write',          // No file writing

'Grep',           // No code searching
'Glob',           // No file pattern matching

];

通过PostToolUse捕获到工具调用的数据后，怎么将其处理成事实并存储呢？

事实提取的完整流程

大概的流程是这样的。

PostToolUseHook点捕获到工具执行信息（工具名、输入、输出、项目目录），然后把这些数据转成XML格式发送给Claude Agent。Agent收到后会判断这次操作是否有意义，若有意义就生成结构化的观察，也就是前文讲述过的observation xml结构。

Agent有明确的记录重点，也就是前文讲述过的关于事实定义的一系列原则：原子化、自包含、具体。

接着，返回的observation xml会被解析成结构化对象，存储在SQLite数据库中。同时，每个observation还会被拆分成多个向量文档同步到Chroma向量库中。关键点是，其中的每个 fact 都成为独立的向量文档。

obs_123_narrative  → 完整叙述作为一个向量
obs_123_fact_0     → 第一个事实作为独立向量
obs_123_fact_1     → 第二个事实作为独立向量
obs_123_fact_2     → 第三个事实作为独立向量

这种细粒度的设计可以让后续基于语义的查询更准确。例如查询"Redis 配置"可以直接匹配到 obs_123_fact_2，而不是整个observation。

何时查：两种检索路径

observations和session_summaries在合适的Hook点存入SQLite和Chroma向量库后，什么时候使用呢？这就是何时查的问题。

Claude-Mem为此设计了两种检索路径，各有侧重。

第一种情况是在SessionStartHook点自动注入。

每当启动新会话时，SessionStartHook会自动触发，从SQLite查询相关的历史observations和``session_summaries`，注入到会话上下文中。

这种方式不需要使用Claude Agent，完全自动化。直接用SQLite来过滤，按项目、时间、类型等条件筛选，几十毫秒就能完成。

注入到Claude Code上下文中的内容采用混合模式：大部分observations只显示索引信息（ID、时间、类型、标题），而最近5条observations会显示完整的narrative或facts。这样设计的好处是既能让Claude Code在新会话中快速了解项目历史脉络，又不会因为注入太多内容而浪费Token。如果Claude Code对某条索引感兴趣，可以通过后面的MCP工具获取完整详情。

第二种是通过MCP工具主动搜索。

当Claude Code在会话中需要查找特定历史信息时，可以主动调用设置的MCP搜索工具。比如当我们问 “之前认证功能是怎么实现的？”，Claude Code会调用MCP工具并传入search({query: "认证实现"})。

MCP会根据是否有query选择不同的策略。如果有query参数，就使用Chroma（它也是通过MCP来提供服务）向量搜索，找到top-100的结果后再应用90天的过滤窗口，最后用返回的ID去SQLite查询完整数据。如果没有query，就直接用SQLite过滤，支持按type、date等参数过滤。

const tools = [
  {
    name: '__IMPORTANT',  // 元工具：注入工作流指令
    description: `3-LAYER WORKFLOW (ALWAYS FOLLOW):
1. search(query) → Get index with IDs (~50-100 tokens/result)
2. timeline(anchor=ID) → Get context around interesting results
3. get_observations([IDs]) → Fetch full details ONLY for filtered IDs
NEVER fetch full details without filtering first. 10x token savings.`,
  },
  {
    name: 'search',
    description: 'Step 1: Search memory. Returns index with IDs...',
  },
  {
    name: 'timeline',
    description: 'Step 2: Get context around results...',
  },
  {
    name: 'get_observations',
    description: 'Step 3: Fetch full details for filtered IDs...',
  }
]

注意看MCP定义的第一个工具__IMPORTANT。它其实是个虚拟工具，没有实际功能，只返回工作流描述。

这个设计比较巧妙，用__前缀确保在在字母排序时排在最前面，这样Claude Code连接MCP服务器，请求工具列表时会先读到它的描述。通过这种方式，工作流指令就被注入到了Claude Code的上下文中。

以后我们在设计自己的MCP服务器时也可以学习这种技巧。

最终的搜索结果会包含三类数据：observations（工具执行观察）、session_summaries（会话摘要）、user_prompts（用户提示词）。和第一种情况的自动注入不同，语义搜索是Claude Code通过MCP主动触发的，适合查找特定的历史信息。

渐进式信息披露

MCP这里还有一个很重要的设计理念：人类不会在开始工作前阅读所有资料，AI 也不应该。

Claude-Mem的MCP搜索被设计成渐进式的。先用search获取简要索引，只有ID、时间、类型、标题，消耗的token较少。看到感兴趣的结果后，可以用timeline查看这个observation前后发生了什么，了解当时的工作流程。最后，只对真正需要的observations调用get_observations获取完整内容。

这种渐进式的设计具有非常明显的Token节省效果，同时又不影响Claude Code的输出效果。

以上两种路径在不同的场景互为补充，SessionStart确保每次新开一个会话都有基础上下文，MCP搜索满足即时的查询需求。

存储架构：SQLite + Chroma双存储

聊完了记什么、何时记、何时查，再来看看底层的存储架构。

前文也有提过，Claude-Mem采用的是双存储架构：SQLite负责结构化存储，Chroma负责语义搜索。

SQLite是主存储，保存所有的观察、摘要、提示词、会话元数据。

核心表observations结构如下：

CREATE TABLE observations (
idINTEGER PRIMARY KEY AUTOINCREMENT,
  session_id TEXTNOTNULL,
projectTEXTNOTNULL,

-- 渐进式披露元数据
  title TEXTNOTNULL,
  subtitle TEXT,
typeTEXTNOTNULL,

-- 内容
  narrative TEXTNOTNULL,
  facts TEXT,           -- JSON 数组

-- 可搜索性
  concepts TEXT,        -- JSON 数组
  files_read TEXT,      -- JSON 数组
  files_modified TEXT,  -- JSON 数组

  created_at TEXTNOTNULL,
  created_at_epoch INTEGERNOTNULL
);

同时，还建了FTS5全文搜索索引，支持关键词快速匹配。

而Chroma是向量数据库，专门用于语义搜索。前文也提到，每个observation会被拆分成多个向量文档，用于精确匹配。

实际查询时，系统会智能选择策略。没有查询词、只有过滤条件时，直接用SQLite查询。有查询词且Chroma可用时，走Chroma语义搜索。如果既有查询词又有过滤条件，就走混合搜索：SQLite先过滤，Chroma再排名。

这里就不再赘述，对细节感兴趣的朋友可以直接读源码。

写在最后

回到开头的问题：如何设计一个好的记忆系统？

Claude-Mem给出的答案其实就围绕三个核心问题展开：

记什么：不是记原始数据，而是提取结构化的observations（观察）。其中核心是facts（事实），它被要求是原子化、自包含、具体、可压缩的知识点。10倍压缩比，既节省空间又尽量保留语义。

何时记：通过Claude Code Hooks系统在合适的Hook点触发。PostToolUse捕获每次工具执行，Stop生成会话摘要。异步Worker设计让知识提取和主工作流解耦，互不干扰。

何时查：两种路径互补。SessionStart自动注入确保每次新开会话都有基础上下文，MCP搜索满足即时查询需求。渐进式信息披露极大的提升了Token的利用率。

这个思路我觉得值得关注。目前大多数AI工具都没有跨会话记忆，每次对话都从零开始。但开发者的工作是连续的，项目知识是累积的。记忆系统的价值不仅是技术实现，更是AI工具体验的根本性提升。
