看到一个 6000+ 星的 agent 仓库，我没收藏，顺手把它做成了一个技能

同样是看到一个 6000+ 星的 GitHub agent 项目，很多人第一反应是先 star、先收藏。我没这么干。

我更关心一件事：它能不能今天就接进我的工作流。看完 agency-agents-zh 之后，我没有停在"这个项目真强"，而是顺手把它封装成了一个可调用的 skill。

这里面真正值钱的，不是它有多少角色，而是这些角色能不能被你稳定调出来。很多人刷 GitHub，最后只多了几个书签；但一旦你把开源项目接进 workflow，它就不再是"资料库"，而是"能力源"。

我这次的方法很简单，只有 4 步。

第一步：先找一个持续更新的上游能力源

像 agency-agents-zh 这种仓库，价值不在"角色多"，而在目录清楚、内容持续更新、能被抓取。

第二步：不要急着全量搬运，先做一层路由

也就是先定义：什么任务该调用它、去哪里找 source、怎么搜到合适角色。这样做的好处是，你不是把仓库原样搬进来，而是先把"调用规则"搭起来。

第三步：补一个最小抓取脚本

只要能搜索、拉取、做基础容错，就已经能把"开源仓库"变成"可用工具"。

第四步：用输出合同约束交付

不是把一堆 prompt 甩给用户，而是明确：这次用了哪些角色、为什么这么选、最终交付什么结果。

比如我这次做出来的 xiancai-agent-router，本质上就不是新建一个 agent，而是把 GitHub 上现成的 agent 库，封成一个可按需调用的路由层。

你只要丢进一个任务，它就会先搜索匹配角色，再抓取对应角色文件，再按预设工作流执行。它最值钱的地方，不是"又多了一个 agent"，而是把原本得手动翻仓库、人工挑角色的过程自动化了。

所以我现在看开源项目，已经不是"值不值得收藏"，而是"能不能顺手做成自己的 skill"。做 skill 最快的方法，往往也不是从零写 prompt，而是先给现成能力源加一层路由。
