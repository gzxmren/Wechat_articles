工具选型：为什么需要 Everything Claude Code？

有了 MVP，很多人这时候会忍不住直接开始改代码——让 AI 优化界面、添加功能。

别急。

在动手改一行代码之前，我们需要先理解一个问题：Superpower 和 Everything Claude Code 到底有什么区别？

H3两者的定位差异

Superpower 擅长 0→1 的规划和执行：

  • 快速搭建项目骨架
  • write-plan 和 execute-plan 能快速把想法变成可运行的代码
  • 适合新项目启动、大规模重构

Everything Claude Code 擅长 1→N 的迭代和质量保障：

  • 内置 100+ 个工作流 Skill（TDD、代码审查、性能优化...）
  • 12 个专家 Agent（规划师、代码审查员、安全审计员...）
  • 自动化 Hook（编辑时自动检查、提交前自动审查...）
  • 适合日常开发、代码审查

H3我的使用策略

  • 新项目启动：Superpower write-plan + execute-plan
  • 日常迭代：Everything Claude Code 的 TDD + code-review
  • 复杂重构：两者结合，Superpower 规划 + ECC 质量保障

最重要的是：这两套工作流都是开源的、可定制的。你完全可以根据自己的习惯，裁剪、组合、改造。

"工具服务于你，不是你适应工具。"

H3什么是 Everything Claude Code？

这是一个在 GitHub 上有 3.6 万 Star 的开源配置包。

一次性获得：

  • 12 个专家 Agent（规划师、代码审查员、安全审计员...）
  • 100+ 个工作流 Skill（TDD、代码审查、性能优化...）
  • 自动化 Hook（编辑时自动检查、提交前自动审查...）
  • 8 套规则约束（代码规范、安全规则、最佳实践...）

以前每次都要告诉 Claude 的事情，现在一行命令就位。

H3三个核心概念：Agent / Skill / Hook

1. Agent（专家代理）

Agent就是一个有专门职责的AI角色。

从命令列表中可以看到的Agent：

  • planner：规划师，负责制定实施计划
  • code-reviewer：代码审查员，负责检查代码质量
  • security-reviewer：安全审计员，负责发现安全漏洞
  • build-error-resolver：构建错误解决专家
  • go-reviewer / python-reviewer：语言专家

每个Agent都有自己的专业知识和工作流程。就像你团队里的不同角色，各司其职。

2. Skill（工作流技能）

Skill是一套标准化的工作流程。

常用的Skill：

  • /everything-claude-code:plan：生成详细的实施计划
  • /everything-claude-code:tdd：测试驱动开发流程
  • /everything-claude-code:code-review：代码审查流程
  • /everything-claude-code:go-review：Go代码专项审查
  • /everything-claude-code:python-review：Python代码专项审查

每个Skill背后都是一套经过验证的最佳实践。你不需要记住所有细节，只需要知道什么时候调用哪个Skill。

3. Hook（自动化钩子）

Hook 是在特定时机自动触发的检查。

比如：

  • 编辑 JS/TS 文件时，自动检查 console.log
  • 编辑 TS 文件时，自动跑 TypeScript 类型检查
  • 执行 git push 前，自动提醒 review 改动

Hook 的价值在于：把质量检查变成自动化流程的一部分，而不是事后补救。

三者的关系

Agent 是"谁来做"，Skill 是"怎么做"，Hook 是"自动做"。

举例：

  • 你调用 /everything-claude-code:tdd 这个 Skill（怎么做）
  • 它会启动 tdd-guide 这个 Agent（谁来做）
  • 完成后，Hook 会自动检查代码格式和类型（自动做）

"Agent、Skill、Hook 三位一体，构成了 Everything Claude Code 的核心工作流。"

H3命令简化改造：从 /everything-claude-code: 到 /ecc:

问题：/everything-claude-code: 这个前缀太长（23个字符），输入效率低， 我经常打字一快就输错。

如何改造： 修改插件配置文件中的 command 字段。具体方法可以参考我之前的文章：

实战演示：从设计稿到代码的完整工作流

H3场景设定

现状：

  • 一个能跑的MVP项目（上一篇完成的）
  • 一套完整的Pencil UI设计稿（第5篇生成的）

目标： 根据Pencil设计稿，重新实现主界面的UI和功能

要求：

  • 严格按照设计稿还原UI
  • 保证代码质量（测试覆盖、类型安全）
  • 全程自动化，不手动写业务代码

H3步骤一：读取 Pencil 设计稿

操作：

利用 pencil mcp 工具，
读取 Pencil 设计稿 @docs/界面设计/UI 设计/main-opus-4.6.pen，
分析里面主编辑器画板的布局结构、组件层级和交互逻辑

输出示例：

主界面结构分析：

1. 顶部工具栏（Toolbar）
   - 左侧：文件操作按钮（新建、打开、保存）
   - 中间：格式化按钮（加粗、斜体、标题）
   - 右侧：导出按钮（复制到微信、导出PDF）

2. 主编辑区（Main Area）
   - 左侧：Markdown编辑器（CodeMirror）
   - 右侧：实时预览区（Preview）
   - 分隔条：可拖拽调整左右比例

3. 底部状态栏（Status Bar）
   - 左侧：字数统计
   - 右侧：自动保存状态

关键点：

不是简单地"看一眼"设计稿，而是让 AI 理解：有哪些组件、组件之间的层级关系、每个组件的功能职责。这些信息会成为后续架构设计的输入。

为什么要先读设计稿？

因为设计稿是"规格"（Spec）的一部分。在 Spec Coding 的理念中，代码是规格的产物。

  • 设计稿定义了"长什么样"
  • PRD 定义了"做什么"
  • 架构设计定义了"怎么做"

三者结合，才能生成高质量的代码。

H3步骤二：架构设计

操作：

/ecc:plan 
 1.界面设计： 根据 Pencil 设计稿 @docs/界面设计/UI 设计/main-opus-4.6.pen 里面的主编辑器界面
 2.功能清单： @docs/需求清单/requirement-list.md
 3.PRD: @docs/PRD.md
 4.技术架构：docs/技术架构/architecture.md
要求：要基于当前 MVP 项目，设计主界面重构的技术方案

输出示例：

## 主界面重构实施计划

### 任务拆解

Task 1: 重构顶部工具栏组件
- 创建 Toolbar.vue 组件
- 实现文件操作逻辑（新建、打开、保存）
- 实现格式化按钮逻辑
- 实现导出功能

Task 2: 重构编辑器组件
- 升级 CodeMirror 配置

- 优化 Markdown 渲染性能

- 创建 Resizer 组件

### 技术选型

- 组件库：继续使用 Vue 3 Composition API
- 状态管理：使用 Pinia

关键点：

这份计划基于：Pencil 设计稿的结构 + 现有 MVP 项目的代码 + 最佳实践和技术约束。

为什么要用 /ecc:plan 而不是直接写代码？

因为计划阶段可以发现很多问题：

  • 技术选型是否合理
  • 任务拆解是否合理
  • 是否有遗漏的功能

在计划阶段改，成本是代码阶段的 1/10。

这就是"代码只是副产物"的第二层含义：好的计划自然会产出好的代码。

H3步骤三：TDD 开发

操作：

/ecc:tdd 实现 Task 1：重构顶部工具栏组件

这一步失败了，它思考过程提示，根据我上面 plan 生成的计划，Sonnet 4.6建议先执行/superpower:write-plan命令，制定详细的实施计划，再执行 TDD 开发。

在这里很明显就发现了，superpower和everything-claude-code在某个环节形成了互补关系。

为什么要用 TDD？

因为 TDD 强制你先思考"这个组件应该做什么"，而不是"这个组件怎么实现"。

这样写出来的代码：

  • 接口清晰（测试定义了接口）
  • 职责单一（测试覆盖了所有职责）
  • 易于重构（有测试保护）

这就是"代码只是副产物"的第三层含义：测试定义了规格，代码只是满足规格的实现。

H3步骤四：代码审查

操作：

/ecc:code-review 审查 Toolbar 组件的代码质量

审查报告示例：

✅ 通过项（5）
1. 组件结构清晰，职责单一
2. 测试覆盖率 100%
3. TypeScript 类型定义完整
4. 无安全漏洞
5. 符合 Vue 3 最佳实践

⚠️ 建议改进（2）
1. 建议添加按钮禁用状态的处理
2. 建议添加键盘快捷键支持

🔴 必须修复（0）

关键点：

代码审查不是"挑刺"，而是"保证质量"。AI 会从多个维度检查：代码规范、安全性、性能、可维护性。

为什么要自动化代码审查？

因为人工审查容易遗漏，而且不一致。

AI 审查的优势：

  • 标准统一（基于配置的规则）
  • 覆盖全面（检查所有维度）
  • 即时反馈（写完就能审）

这就是 Hook 的价值：把质量检查变成自动化流程的一部分。

H3步骤五：功能验证

操作：

npm run dev

验收清单：

  • [ ] 工具栏布局与设计稿一致
  • [ ] 文件操作按钮功能正常
  • [ ] 格式化按钮功能正常
  • [ ] 导出功能正常
  • [ ] 响应式布局正常
  • [ ] 无控制台错误

全部通过，Task 1 完成。

关键点：

自动化测试保证了"代码逻辑正确"，但最终还是要人工验证"用户体验正确"。

为什么不能完全依赖自动化测试？

因为有些东西测试覆盖不了：

  • UI 是否美观
  • 交互是否流畅
  • 细节是否到位

所以完整的工作流是： 自动化测试 + 人工验收

  • 自动化保证质量下限
  • 人工验收保证质量上限

实战反思：AI 编程的三个真实瓶颈

走完整个工作流后，我发现实际开发时间远超预期。下面是某个对话窗口的优化案例：

H3瓶颈一：AI 生成代码的排查困境

现象：作为资深开发者，如果代码是我自己写的，我能快速定位问题。但全程由 AI 产出的代码，我反而不好排查错误。

现状：

  • 只能让 AI 排查完，我再基于经验判断
  • 即使用 Sonnet 4.6 这种一线模型，对话轮次依然很多
  • 如果用国产模型，对话轮次会更多，时间成本更高

对代码小白的影响：

  • 完全依赖 AI 的修改建议
  • 缺乏判断能力，对话轮次比懂代码的人更多
  • 排查错误的时间会更长

H3瓶颈二：90/10 法则——最后 10% 的质量困境

核心问题：无论是 Superpower 还是 Everything Claude Code，都很难解决代码最后调优的 10% 问题。

90/10 法则：

  • 90% 的代码：AI 能规范产出，质量合格
  • 10% 的调优：才是上线标准，却要花费 90% 的时间

我的判断：

未来的 Vibe Coding 时代，谁能掌握技巧把这 90% 的时间压缩到最低，谁就能脱颖而出。

H3瓶颈三：Pencil 工具的精度局限

问题描述：

  • Pencil 生成的设计再好看，一比一还原成代码难度很大
  • 与 Figma 设计稿相比，颗粒度差很远

具体痛点：

  1. 缺少标注：无法通过 .pen 源文件知道两个盒子之间的精确像素
  2. 调优困难：明知道 padding 不够，但不知道需要多少
  3. 适用范围：只能做到 MVP 项目标准，复杂界面的后期调优时间很长

H3瓶颈四：Code Review 的时间成本困境

核心矛盾：界面调优和功能调优的度比较难拿捏。

实战发现：

  • 调优内容很多不在任何 Spec 文件里
  • 不是 Spec 写得不够好，而是规范文件的颗粒度根本不会那么细

两难选择：

  1. 不认真 Code Review：代码距离屎山不远
  2. 全程认真 Code Review：
    • 需要阅读代码的能力
    • 占用大量时间
    • 无法同时开多个 Agents 处理不同任务

现状：这个工作模式，我还没有找到比较好的解决方案。

总结与展望

H3代码确实只是副产物

计划的质量决定代码的质量。没有好的 Spec，再强的 AI 也只能给你一坨跑起来的屎山。

这篇文章，我们完整演示了 Everything Claude Code 的工作流：

读设计稿 → 架构设计 → TDD 开发 → 代码审查 → 功能验证

全程自动化，没有手写一行业务代码。

但如果你仔细看，我做的决策并不少：

  • 选择哪个任务先做
  • 如何拆解复杂任务
  • 审查报告中哪些建议要采纳

这才是"超级个体"和"会用 AI 写代码"的区别：

你在做架构师和质量把关人的活，代码只是 AI 替你产出的副产物。

H3主界面重构完成了，但这只是开始。

真正让产品可用，还有三个关键环节：

  1. 性能优化：首屏加载要控制在 2 秒内，大文件拖拽不能卡顿，这需要懒加载、虚拟滚动、Web Worker 等技术组合拳。

  2. 体验打磨：快捷键要符合直觉（Cmd+S 保存、Cmd+Z 撤销），拖拽要有视觉反馈，错误提示要友好——这些细节决定用户会不会持续使用。

  3. 容错处理：网络断了怎么办？文件损坏了怎么恢复？这些边界场景，AI 不会主动考虑，需要我们补充测试用例，让 AI 按规范实现。

这些打磨工作，我会在接下来一周完成。完成后，我们就进入最后一个环节：让产品真正上线。
