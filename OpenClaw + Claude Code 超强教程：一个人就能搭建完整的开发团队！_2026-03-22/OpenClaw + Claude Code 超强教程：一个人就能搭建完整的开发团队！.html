第一部分：核心概念解析

1.1 为什么需要双层系统？

问题：Codex/Claude Code 单独用不够好

根本限制：上下文窗口是固定的

你必须二选一：

塞满代码 → 没空间放业务上下文
     OR
塞满客户历史 → 没空间放代码库

实际遇到的问题：

问题
影响
不知道功能为哪个客户做
可能违反客户约束
不知道上次类似需求为什么失败
重复踩坑
不知道产品定位和设计原则
代码风格不一致
只看当前代码 + prompt
无法全局优化

解决方案：双层架构

OpenClaw（编排层）的角色：

  • 🧠 持有所有业务上下文
  • 📋 把业务上下文翻译成精确 prompt
  • 🤖 调度不同的 Agent
  • 📊 监控进度，失败自动重试
  • 🔔 完成后通知你

Agent（执行层）的角色：

  • 💻 读写代码库
  • 🧪 运行测试和构建
  • 📝 提交代码和创建 PR
  • 🔍 响应 code review

1.2 类比：餐厅厨房

传统方式（直接用 Codex）：
顾客 → 厨师（只知道食材，不知道客人口味）

双层架构（OpenClaw + Agent）：
顾客 → 主厨（OpenClaw，知道客人口味、库存、菜单）
       → 专业厨师（Agent，专注做菜）

主厨（OpenClaw）需要知道：

  • 👤 客人口味偏好
  • 🍳 食材库存
  • 📜 菜单定位
  • ⏰ 哪道菜应该优先做

厨师（Agent）只需要：

  • 🔪 精确的做菜指令
  • 🥘 食材清单
  • 📋 装盘规范

#!/bin/bash
# 启动 Agent 脚本

TASK_ID="$1"
AGENT_TYPE="$2"
PRIORITY="${3:-normal}"

if [ -z "AGENT_TYPE" ]; then
echo"用法: $0 <task_id> <agent_type> [priority]"
exit 1
fi

PROJECT_ROOT=~/medialyst
WORKTREE_ROOT=~/ai-agent-system/workspace/worktrees
TASK_RECORD=~/ai-agent-system/workspace/tasks/active/${TASK_ID}.json

echo"🚀 启动 Agent: PRIORITY)"

# 检查任务是否存在
if [ ! -f "$TASK_RECORD" ]; then
echo"❌ 任务记录不存在: $TASK_RECORD"
exit 1
fi

# 创建 git worktree
BRANCH_NAME=<<MATH_INLINE_ki0v1s>>TASK_RECORD" | jq -r '.branch')
WORKTREE_NAME="$TASK_ID"
WORKTREE_PATH="WORKTREE_NAME"

if [ ! -d "$WORKTREE_PATH" ]; then
    echo "📁 创建 worktree: $WORKTREE_PATH"
    cd "$PROJECT_ROOT"
    git worktree add "BRANCH_NAME" origin/main
fi

# 安装依赖（如果需要）
if [ -f "$WORKTREE_PATH/package.json" ]; then
    echo "📦 安装依赖..."
    cd "$WORKTREE_PATH"
    pnpm install
fi

# 启动 tmux 会话
SESSION_NAME="agent-$TASK_ID"
tmux has-session -t "$SESSION_NAME" 2>/dev/null

if [ $? -ne 0 ]; then
    echo "🖥️  创建 tmux 会话: $SESSION_NAME"
    tmux new-session -d -s "WORKTREE_PATH"

    # 根据不同 Agent 类型执行不同命令
    case "$AGENT_TYPE" in
        codex)
            tmux send-keys -t "HOME/.codex-agent/run-agent.sh PRIORITY" Enter
            ;;
        claude)
            tmux send-keys -t "HOME/.claude-agent/run-agent.sh PRIORITY" Enter
            ;;
        gemini)
            tmux send-keys -t "HOME/.gemini-agent/run-agent.sh PRIORITY" Enter
            ;;
        *)
            echo "❌ 未知的 Agent 类型: $AGENT_TYPE"
            exit 1
            ;;
    esac

    # 更新任务状态
    TMP=$(mktemp)
    jq '.status = "running" | .tmuxSession = "'TASK_RECORD" > "$TMP"
    mv "TASK_RECORD"

    echo "✅ Agent 已启动: $SESSION_NAME"
else
    echo "⚠️  tmux 会话已存在: $SESSION_NAME"
fi
EOF

        TASK_ID=task_file" .json)

        echo "" | tee -a "$LOG_FILE"
        echo "检查任务: LOG_FILE"

        # 读取任务信息
        TMUX_SESSION=task_file")
        AGENT_TYPE=task_file")
        REPO=task_file")
        BRANCH=task_file")

        # 检查 tmux 会话是否存活

            if tmux has-session -t "$TMUX_SESSION" 2>/dev/null; then
                echo "✅ tmux 会话存活: LOG_FILE"

                mv "FAILED_DIR/${task_id}.json"

                # 发送通知
                notify_weixin "❌ <b>任务失败</b>: $task_id\n\n原因: 达到最大重试次数"
            fi
        fi
    fi
}

# 主循环
log "🔍 开始监控任务..."

if [ -d "$ACTIVE_DIR" ]; then
    for task_file in "$ACTIVE_DIR"/*.json; do
        if [ -f "$task_file" ]; then
            check_task "$task_file"
        fi
    done
else
    log "没有活跃任务"
fi

log "✅ 监控完成"

附录：常见问题与解决方案

Q1: Agent 经常失败怎么办？

可能原因：

  1. Prompt 不够详细
  2. 缺少必要的上下文
  3. 任务拆解太粗糙

解决方案：

  1. 增加业务上下文（会议记录、客户历史）
  2. 细化任务拆解
  3. 提供明确的成功标准

Q2: 内存不够用怎么办？

可能原因：

  1. 同时运行太多 Agent
  2. 每个worktree都有独立的 node_modules

解决方案：

  1. 限制并发 Agent 数量（4-5 个）
  2. 使用符号链接共享 node_modules
  3. 升级硬件（128GB RAM 可跑 30+ Agent）

Q3: Code Review 太严格怎么办？

可能原因：

  1. Claude Reviewer 过度谨慎
  2. 没有配置好豁免规则

解决方案：

  1. 禁用 Claude Reviewer（只标记 critical 时使用）
  2. 配置豁免规则
  3. 调整严重性阈值

Q4: 如何防止生成垃圾代码？

关键措施：

  1. 自动化测试
    单元测试、集成测试、E2E 测试
  2. Code Review
    3 个 Reviewer 交叉验证
  3. 人工审核
    所有 PR 必须人工 review
  4. CI 拦截
    测试失败无法合并
