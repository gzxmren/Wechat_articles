AI Coding 防屎山：给Claude加护栏，让修改不再失控

断断续续搞了1年的AI coding，发现对于小白来讲，他们面前最大的山是“不可控”。无法精准的告诉AI怎么修改，无法控制AI哪里改，哪里不改。


明明在拿AI精修网站，改一次像中彩票，改三次开始互相打架，改五次项目变“屎山”。

更让人崩溃的是不知道从哪里开始追责，因为“错”不是一次改坏，是每次改一点点，最后你也说不清到底是哪一次把项目带沟里了。

类比一下，你觉得自己脸形不好看，去整形医院和AI医生沟通，你说：我的脸型太难看了，我想要一个鹅蛋脸。 AI一顿操作，你发现自己不仅下巴变成了鹅蛋，头顶也尖了，整个头成为了3维立体的鹅蛋。


AI coding 的恐怖就在这：它不是不听话，它是太能干了。

最简单的办法：和AI深度沟通你的需求，把你的“边界”写成规则文档，每次让 AI 改代码前，必须先读这份文档。

比如我最近在做一个带后台的客单，接口比较多，又是多语言，害怕AI改到后面，后端都接不起来。于是我写了一个文档，将技术栈、总原则、组件、结构、逻辑、绝对不能做的内容写在里面。

有了这个文档后，我每次用Claude/codex改代码简单且准确了无数倍，只需要告诉它：先阅读并严格执行CODING_RULES。 不需要冗长的提示词，不需要如履薄冰的修改，不会有一层层的bug。AI还是很强，但它终于有了“护栏”。


CODING_RULES 防屎山工程规范（核心节选）

适用项目：XXX公司双语官网（EN/RU）

技术栈：Next.js（App Router）+ TypeScript + Tailwind

目标：保证项目结构清晰、可扩展、可维护；支持大量牌号数据页面；避免复制粘贴式堆代码。

0. 总原则（必须牢记）

  1. 内容与代码分离：文案/数据必须放 content 文件里，页面只负责展示。
  2. 组件可复用：重复出现的 UI 必须抽组件，禁止复制粘贴。
  3. 单一数据入口：读取 content 的逻辑只能写在 `lib/content/*`，其他地方不要直接读文件。
  4. 页面文件要短：页面只组合组件，禁止堆业务逻辑。
  5. 每次改动可运行：改完必须能 `npm run dev` 启动并访问对应页面。
