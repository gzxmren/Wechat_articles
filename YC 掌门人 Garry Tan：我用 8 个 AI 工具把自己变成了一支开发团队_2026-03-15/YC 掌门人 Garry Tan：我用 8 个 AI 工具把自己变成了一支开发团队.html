YC 掌门人 Garry Tan：我用 8 个 AI 工具把自己变成了一支开发团队

从0到AI，每天为你拆解 AI 工具的实战用法与部署技巧。全文约 4000 字，建议收藏慢读。

"规划不是审查。审查不是发布。创始人的品位不是工程师的严谨。把它们混在一起，你只会得到一个平庸的混合体。"

"我不想要一个万能的 AI 助手。我想要能随时切换认知模式的工具——创始人模式、工程师模式、偏执审查者模式、发布机器模式。"

"一个人，十个并行 AI，每个都有正确的认知模式在工作。这不是增量改进，这是一种完全不同的软件构建方式。"

Garry Tan 是谁？为什么他说的值得听

Garry Tan 是 Y Combinator 的总裁兼 CEO，全球最顶级的创业加速器的掌门人。YC 孵化了 Airbnb、Stripe、Dropbox、Coinbase……那些改变了互联网格局的公司，大多数都经过 YC。

他本人也是工程师出身，在成为职业投资人之前，曾是 Palantir 的早期员工，后来联合创办了 Posterous（被 Twitter 收购）。

所以当他说"我每天写一万行代码、每天合并十个 PR"的时候，这不是吹牛——这是一个真正在用 AI 工具重新武装自己的工程师的实战经验。

最近，他公开了自己完整的 Claude Code 使用方式，打包成一个叫 gstack 的开源工具集。

一个 AI 助手的根本局限

在讲 gstack 做了什么之前，先说它为什么存在。

很多人用 AI 编程助手，用着用着就发现了一个问题：这个工具太"万能"了，反而不够用。

你问它"帮我规划一个功能"，它给你一个设计方案。 你问它"帮我审查代码"，它又给你一个方案。 你问它"帮我发布"，它还是给你一个方案。

每次都是一个差不多的"方案"。没有真正的产品直觉，没有真正的工程严谨，没有真正的发布纪律。所有认知模式被压缩在一起，结果是一个什么都会、什么都不精的通才。

Garry 的判断是：AI 助手需要"档位"。

规划时你需要的是创始人的视野——什么是真正的产品？ 设计时你需要的是工程经理的严谨——架构、边界、边缘情况。 审查时你需要的是偏执的高级工程师——什么东西能通过测试但在生产中炸掉？ 发布时你需要的是不废话的发布工程师——同步、测试、推送、PR，执行就行了。

gstack 就是这套"档位系统"的实现。

创始人模式：永远先问"这是对的产品吗？"

/plan-ceo-review 是 Garry 几乎每个功能都会用的第一步。

他举了一个例子：他在做一个类 Craigslist 的二手交易应用，需求是"让卖家上传商品图片"。

一个普通的 AI 助手会怎么做？添加一个文件选择器，保存图片，完成。

/plan-ceo-review 会问不同的问题："图片上传"真的是这个功能吗？

也许真正的功能是"帮卖家生成一个能卖出去的商品列表"。如果是这样，整个方向就变了：

  • 能从图片里识别商品吗？
  • 能自动搜索规格和竞品定价吗？
  • 能自动生成标题和描述吗？
  • 能判断哪张图片更有吸引力吗？
  • 能检测出图片太暗、太乱、让人不信任吗？

"上传图片"和"帮卖家卖出商品"是两个完全不同的产品。前者是一个功能，后者是一个有竞争力的差异化体验。

这就是他说的 Brian Chesky 模式——不是实现那个明显的需求单，而是从用户视角重新审视，找到那个"感觉必然、令人愉悦、甚至有点神奇"的版本。

工程师模式：用图来逼出隐藏的假设

产品方向确定之后，切换到 /plan-eng-review。

这个模式不要更多的发散思维，它要的是可以落地的技术骨架：

  • 上传、识别、丰富内容、生成草稿的架构是什么？
  • 哪些步骤同步执行，哪些进后台任务队列？
  • 如果上传成功但内容丰富失败，怎么降级？
  • 重复任务怎么防止？
  • 数据什么时候持久化，什么可以安全重算？

Garry 特别强调图表的重要性：架构图、状态图、数据流图、测试矩阵。

他的理由很直接：LLM 在被强迫画出系统的时候，会变得更完整。图表把模糊的假设强制变成明确的决定，让"说得含糊"变得很难。

审查模式：想象生产事故，在它发生之前

代码写完，测试通过。你觉得可以发布了吗？

/review 的存在是因为有整整一类 bug，能通过所有测试，然后在生产环境里把你打得措手不及。

Garry 希望这个模式找的是结构性问题，不是风格问题：

  • N+1 查询
  • 过时读取、竞态条件
  • 信任边界问题（有没有相信客户端传来的元数据？）
  • 遗漏的索引
  • 转义 bug
  • 错误的重试逻辑
  • 能通过测试但在真实失败场景下完全失效的测试

还是那个商品上传的例子。测试都绿了，但 /review 还应该问：

两个标签页同时操作，会不会互相覆盖封面图选择？上传失败的文件会不会永远留在对象存储里？从网页抓取内容再塞进草稿生成，会不会产生提示注入攻击面？

这不是要夸你做得好。这是在生产事故发生之前，先想象一遍它会怎么发生。

发布模式：该发的时候就发，不要再聊了

/ship 是为已经做完的分支准备的，不是为了决定做什么。

这个模式的设计哲学是：动作比讨论重要。

很多分支死在有趣的工作做完之后，只剩枯燥的发布工作——同步主分支、跑测试、更新变更日志、推送、开 PR。人会拖延这些事，AI 不应该。

/ship 接管这些步骤：同步 main，检查分支状态，跑对应的测试，处理版本号和 changelog（如果项目有要求），推送，开或更新 PR。

它还和 Greptile 集成——这是一个 YC 投资的自动代码审查工具。它分析 PR 里的真实 bug，包括竞态条件、安全问题、能通过 CI 但在生产中爆炸的那些。

集成之后，每次 /ship，都会自动拉取 Greptile 的评论，分类处理：

  • 真实问题：加入关键发现清单，修完再发布
  • 已修复的问题：自动回复确认
  • 误报：让你确认后，自动回复说明原因

还有学习机制——每次你确认的误报都会被记录，下次自动跳过同类模式。

让 AI 有眼睛：/browse 和 /qa

这两个工具是 gstack 里最有创意的部分。

在此之前，AI 编程助手是半盲的。它能写代码，但看不到 UI 状态、认证流程、重定向、控制台报错、空状态、布局问题。测完之后，还是得人工打开浏览器点一圈。

/browse 解决的就是这个问题。

底层是一个编译好的二进制文件，连接到持久运行的 Chromium 实例，基于 Playwright 构建。第一次调用约 3 秒启动，之后每次调用只要 100-200 毫秒。浏览器一直在运行，Cookie、标签页、localStorage 全部保留。

实际使用是这样的：

你说"去测 staging 环境的注册流程，还有这个分支改到的所有页面"。AI 自己去：

  • 打开注册页面
  • 填表单
  • 点提交
  • 截图，用视觉分析确认跳转是否正确
  • 挨个打开所有修改过的页面
  • 检查控制台有没有报错
  • 验证 API 端点
  • 汇报结果

大概 60 秒，18 次工具调用，一次完整的 QA 。你没有打开过浏览器。

/qa 在此基础上更进一步，加了方法论：它会读取 git diff，识别哪些文件改动可能影响哪些页面和路由，只测受影响的部分——不需要你指定测什么。

四种模式：

  • 自动模式（在功能分支上默认）：读 diff，找受影响的页面，精准测试
  • 完整模式：系统性探索整个应用，5-15 分钟，记录 5-10 个有据可查的问题
  • 快速模式（--quick）：30 秒烟雾测试，首页+主导航，看加载、控制台、链接
  • 回归模式（--regression baseline.json）：跑完整测试，对比上次基线，哪些修了，哪些是新问题

还有一个 /setup-browser-cookies：从你真实浏览器（Chrome、Arc、Brave、Edge）导入 Cookie 到 Playwright 会话，这样 /qa 就能测登录后的页面，不用每次手动登录。

复盘模式：不要凭感觉，要看数据

/retro 是每周结束时用的。

它分析提交历史、工作模式、发布速度，生成一份有态度的复盘报告——不是感觉，是数据。

它知道谁在跑这个命令，对你自己做最深入的分析，然后逐一点评每个贡献者：具体的夸奖，具体的成长机会——就像你在一对一谈话里会给的那种反馈。

指标包括：提交次数、代码行数、测试比例、PR 大小、修复比例。它还能从提交时间戳推断出编码会话，找到最常改动的热点文件，追踪连续发布天数，找出这周最大的一次发布。

结果会存成 JSON 快照，下次 /retro compare 可以看这周和上周的对比。

一个人，十个 AI，每个都在用正确的模式工作

gstack 的独特价值，在于和 Conductor 结合之后的可能性。

Conductor 可以同时跑多个 Claude Code 会话，每个会话有独立的工作区和独立的 Chromium 实例。

这意味着什么？

同时运行：一个会话在对 staging 跑 /qa，一个在对 PR 跑 /review，一个在实现新功能，另外七个在其他分支上工作。互相不干扰，每个都在正确的认知模式里。

Garry 的原话是："一个人，十个并行 AI，每个都有正确的认知模式在工作。这不是增量改进，这是一种完全不同的软件构建方式。"

写在最后

gstack 值得关注，不只是因为它的功能清单，而是因为它背后的一个判断：

AI 助手最大的问题，不是能力不够，而是模式不清。

把规划、审查、发布全混在一起，每次都是一个差不多的通才答案。把它们分开，给每种任务正确的认知框架，效果是质变。

这也是 Garry 公开这套工具的原因——不是炫技，是分享一种工作方式。对于认真用 Claude Code 的人来说，这套方法论值得认真研究。

几个核心要点：

  1. AI 助手需要"档位"——规划、审查、发布是三种完全不同的认知模式
  2. /plan-ceo-review 先问"这是对的产品吗"，再开始建
  3. /plan-eng-review 用图表把模糊假设变成明确决定
  4. /review 不找风格问题，找能在生产中爆炸的结构性 bug
  5. /browse 和 /qa 让 AI 真正"看见"你的应用，替代手工点击测试
  6. 与 Greptile 集成，自动分类代码审查评论，误报也能反馈学习

安装一行命令：在 Claude Code 里粘贴 git clone https://github.com/garrytan/gstack.git ~/.claude/skills/gstack && cd ~/.claude/skills/gstack && ./setup，其余的让 Claude 来做。
