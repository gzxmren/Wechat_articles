为什么你的 Multi-Agent 越拆越乱？

在做 Agent 落地的时候，经常被问到下面几个问题：

“这个场景是不是该上 Multi-Agent？”、“为什么要用 Multi-Agent？”、 “你的 SubAgent 是按什么标准拆的？”

一开始我还会一本正经地画架构图、讲设计原则，
被问得多了之后，反而开始意识到一个问题：

大家真正困惑的，可能不是怎么拆 Agent，
而是——到底什么时候「不该拆」，什么时候「不该拆」。

这篇文章不是一份 Multi-Agent 教程，
是慢慢总结出来的一套偏工程判断的经验。

一、我什么时候开始意识到：单 Agent 已经不太行了

一开始，我也很贪心。希望一个 Agent 能：

  • 理解业务背景
  • 记住完整历史上下文
  • 调用各种工具
  • 做分析、给结论，最好还能顺便写点总结

结果就是——
它什么都会一点，但什么都不稳定。

后来我意识到一个关键问题：

Agent 出问题，往往不是能力不够，而是上下文太吵。

当一个 Agent 的上下文里同时混着：

  • 历史对话
  • 中间推理
  • 工具返回结果
  • 临时规则和补丁

你以为它在「综合判断」，
实际上它是在 用概率硬抗混乱。

也是从这里开始，我才真正理解：
引入 Multi-Agent 的核心目的，并不是“分工协作”，而是 隔离上下文污染源。

二、判断要不要拆 Agent，我只看一个指标  ——「上下文耦合度」

后来我基本不再问：

“这个能力是不是该一个 Agent？”

而是改问：

“这个判断，到底需要知道多少上下文？”

我现在常用一个很土、但极其好用的判断方式：

  • 如果一个子任务
    • 不需要完整对话历史
    • 不依赖其它任务的中间推理
    • 只对一个非常窄的问题负责

那它就不该被硬塞在主 Agent 里。我把这个标准叫做：

上下文耦合度

  • 耦合度高 → 放在一个 Agent 里
  • 耦合度低 → 拆出去，单独活

有意思的是，后来我在 Anthropic 的一篇文章里看到几乎一模一样的结论：
Agent 的拆分，本质上是上下文边界的拆分，而不是功能拆分。

这也让我更加确信：
我之前踩的那些坑，并不是“姿势不对”，
而是判断维度一开始就错了。

三、多 Agent 真正的成本，不是 Token,而是你没算清的「通信税」

很多人聊 Multi-Agent 时，第一反应都是算账：

  • 多跑几次模型
  • 多花点 token
  • 成本会不会上升

但在真实工程里，我越来越觉得：
这些都不是最贵的。

真正昂贵的，是另一种你很难量化的成本：

通信税

只要你拆了 Agent，就一定会发生三件事：

  1. 上下文被转述
  2. 信息被压缩
  3. 决策被分段

每多一个 Agent，你就在赌一件事：
关键信息不会在传递过程中丢失或变形。

而这个赌局，随着 Agent 数量增加，
胜率是指数下降的。

我见过最典型的后果是：

  • 每个 SubAgent 的输出都“看起来很合理”
  • 但整体决策却开始漂移
  • 出问题时，没人说得清是哪一步出了问题

这类现象，在 Cognition 的那篇
《Don’t Build Multi-Agents》里被总结成一句非常狠的话：

隐式决策链断裂（Implicit Decision Chain Break）

你把原本连续完成的思考，
拆成了多段独立的判断，
但并没有任何机制保证它们还能拼回原来的逻辑。

四、什么样的 SubAgent，才值得被拆出来？

如果一定要拆，我现在对 SubAgent 的要求其实非常苛刻。

一个合格的 SubAgent，在我眼里应该像个「偏执狂」：

1️⃣ 上下文极度克制

  • 只知道完成任务所需的信息
  • 看不到的东西，反而更安全

2️⃣ 职责单一到有点无聊

  • 不分析全局
  • 不做综合判断
  • 只回答一个明确问题

3️⃣ 输出必须结构化

  • 不是“人类看着顺”的那种
  • 而是“机器解析零歧义”的那种

我现在更愿意把 SubAgent 理解成：

一个被人格化的函数，
而不是一个缩小版的大模型。

五、写在最后：

Multi-Agent 不是升级路线，而是一种止损手段

如果你让我用一句话总结经验，那就是：

Multi-Agent 解决的不是“能力不足”，
而是“复杂度失控”。

它不是默认选项，而是：

  • 单 Agent 已经被榨干
  • 上下文开始互相污染
  • 再怎么调 prompt 都救不回来的时候

才值得考虑的架构手段。

所以，如果你现在也在犹豫要不要上 Multi-Agent，
我反而建议你先问自己一句：

我是真的需要拆 Agent，
还是只是觉得“这样看起来更高级”？

很多架构，输的不是能力，
而是输在太早相信了复杂度本身。
