一、先搞懂：多Agent协作，到底是个啥？

很多人一听到“多Agent”“路由”“子Agent”这些词就头大，其实不用怕，我用最通俗的话给你掰碎了说，保证一听就懂。

OpenClaw本身就是个开源的AI Agent框架，它的多Agent协作，核心就是“多个人、多种人格”一起干活。你可以创建多个独立的AI Agent，每个Agent都有自己的“小空间”（工作区）、“小记忆”（会话存储）、“身份证”（认证配置），就像公司里的不同岗位，各干各的活，互不打扰。

举个例子：你可以创建5个Agent，分别当需求分析师、开发者、代码审查员、测试工程师，还有一个项目调度员，专门负责分配任务、汇总结果。这样一来，你再也不用一个人切换5个角色，每个Agent都专注自己的领域，效率直接翻倍。

这里有几个关键概念，不用死记硬背，听我唠一遍就记住：

✅ Agent：就是一个独立的AI“员工”，有自己的工作空间、记忆和配置，只干自己擅长的事；

✅ 多Agent路由：多个AI“员工”同时干活，通过规则分配任务，谁该干啥，一目了然；

✅ 子Agent：一个Agent可以调用另一个Agent，比如调度员调用需求分析师，形成主从关系，不用你手动干预；

✅ 绑定（Bindings）：就是制定规则，比如飞书群里的消息，自动交给调度员Agent处理，不用你手动转发。

简单说，多Agent协作，就是用AI搭建一个“虚拟研发团队”，让AI替你分担所有重复、繁琐的工作，你只需要专注核心需求，省时又省力。

三、手把手教你：搭建属于自己的虚拟研发团队（超简单，零复杂操作）

很多人觉得，搭建多Agent肯定很复杂，需要懂很多代码、会配置各种参数。其实真的不用！我一步步教你，全程复制命令、粘贴配置，新手也能轻松搞定，全程不绕弯、不踩坑。

咱们的目标是搭建一个完整的研发协作系统，包含5个AI“员工”：Director（项目调度员）、需求分析师、开发者、代码审查员、测试工程师，再配合飞书群组，实现从需求到代码的全自动化流程。

3.1 先搞清楚架构：三层结构，简单好记

不用看那些复杂的架构图，我用一句话给你说清楚咱们的虚拟研发团队架构：

① 交互层：飞书群组（你和AI“员工”沟通的地方，你在群里提需求，AI就开始干活）；

② 调度层：Director（项目调度员，相当于团队leader，负责接收你的需求、分配任务、汇总结果）；

③ 执行层：4个专业AI“员工”（需求分析师、开发者、代码审查员、测试工程师），各司其职，专注单一任务。

工作流程也特别简单，就像流水线一样：你提需求 → 调度员分析需求 → 分配任务给各个AI → 各个AI完成任务 → 调度员汇总结果 → 反馈给你，全程自动流转，不用你手动干预。

3.2 第一步：创建5个AI“员工”（Agent）

创建Agent特别简单，不用手动建文件夹、写配置，只需要在命令行敲几条命令，OpenClaw就会自动帮你创建好，每个Agent都有自己独立的工作空间和配置，特别省心。

打开你的命令行，依次敲这5条命令，复制粘贴就行，不用修改任何内容：

openclaw agents add director
openclaw agents add analyst
openclaw agents add developer
openclaw agents add reviewer
openclaw agents add tester

敲完这5条命令，OpenClaw就会自动创建5个Agent，分别对应：调度员（director）、需求分析师（analyst）、开发者（developer）、代码审查员（reviewer）、测试工程师（tester）。

每个Agent创建成功后，都会自动生成3样东西：独立的工作空间（放自己的配置文件）、独立的状态目录（存认证信息）、独立的会话存储（记自己的工作记录），不用你手动操作，特别方便。

这里提醒一句：创建的时候，如果系统让你绑定模式，不用管它，默认不绑定就行，后面咱们可以再修改配置，不影响使用。

3.3 第二步：配置主配置文件（openclaw.json）

创建好Agent之后，咱们需要简单配置一下主配置文件，告诉OpenClaw这5个Agent怎么配合工作、怎么对接飞书。

配置文件的路径是：~/.openclaw/openclaw.json，你只需要把下面的配置复制粘贴进去，替换掉原来的内容，再把里面的飞书AppID和AppSecret换成你自己的，就可以了，不用懂任何代码。

{
  agents: {
    list: [
      {
        id: "director",
        name: "项目调度",
        default: true,
        workspace: "~/.openclaw/workspace-director",
        agentDir: "~/.openclaw/agents/director/agent"
      },
      {
        id: "analyst",
        name: "需求分析师",
        workspace: "~/.openclaw/workspace-analyst",
        agentDir: "~/.openclaw/agents/analyst/agent"
      },
      {
        id: "developer",
        name: "开发者",
        workspace: "~/.openclaw/workspace-developer",
        agentDir: "~/.openclaw/agents/developer/agent"
      },
      {
        id: "reviewer",
        name: "代码审查员",
        workspace: "~/.openclaw/workspace-reviewer",
        agentDir: "~/.openclaw/agents/reviewer/agent"
      },
      {
        id: "tester",
        name: "测试工程师",
        workspace: "~/.openclaw/workspace-tester",
        agentDir: "~/.openclaw/agents/tester/agent"
      }
    ],
    defaults: {
      subagents: {
        maxSpawnDepth: 2,          // 允许调度员调用其他Agent，最多2层嵌套
        maxChildrenPerAgent: 5,    // 每个Agent最多同时处理5个任务，避免卡壳
        maxConcurrent: 8,          // 全局最多同时运行8个任务
        model: "anthropic/claude-sonnet-4-5",  // 子Agent用便宜的模型，省成本
        runTimeoutSeconds: 900     // 单个任务最多15分钟，避免无限运行
      }
    }
  },

  bindings: [
    {
      agentId: "director",
      match: { channel: "feishu", accountId: "dev-team" } // 飞书消息自动交给调度员
    }
  ],

  channels: {
    feishu: {
      enabled: true,
      dmPolicy: "pairing",  // 私聊需要配对，避免陌生人打扰
      accounts: {
        dev-team: {
          appId: "cli_xxx", // 替换成你的飞书AppID
          appSecret: "xxx", // 替换成你的飞书AppSecret
          botName: "AI 研发助手"
        }
      }
    }
  }
}

这里给大家简单解释一下关键配置，不用记，了解就行：

① maxSpawnDepth: 2：允许调度员调用其他Agent，Agent还能调用工具，避免任务卡壳；

② maxChildrenPerAgent: 5：控制并发，避免太多任务同时运行，导致系统卡顿；

③ model: 子Agent用便宜的模型，比贵的模型省5倍成本，咱们职场人，能省则省；

④ bindings：飞书群的消息，自动交给调度员Agent处理，其他Agent不直接跟你沟通，避免混乱。

3.3 第三步：对接飞书（简单几步，不用公网URL）

咱们平时都是用飞书办公，所以把AI“员工”对接飞书，以后直接在飞书群里提需求，AI就会自动干活，特别方便。对接步骤也很简单，跟着我做就行，不用懂技术。

第一步：安装飞书插件，命令行敲这一句：

openclaw plugins install @openclaw/feishu

第二步：创建飞书应用，打开飞书开放平台（https://open.feishu.cn/app），先登录平台——登录方式很灵活，既可以用飞书移动端扫码登录，也能切换至Lark登录，登录成功后，按照下面的页面指引和步骤来，新手也能一眼找到入口：

先跟大家说下登录后的页面布局，不用慌，特别简洁：登录成功后，页面顶部会显示多语言切换栏（默认简体中文，不用修改），中间是飞书开放平台的核心导航区，下方会展示你的企业相关信息和已创建的应用（如果是第一次用，这里是空的）。重点看中间导航区，“创建应用”入口特别显眼，一般在页面中间偏上的位置，按钮样式清晰，一眼就能找到，不用翻来翻去。另外，页面右上角会显示你的登录状态，若登录异常，会提示“Scanned successfully”（扫码成功）或重新登录，遇到这种情况重新扫码即可，不用额外操作。

1.  登录后，找到“创建应用”入口，选择“企业自建应用”，起个好记的名字，比如“AI研发助手”，点击确认创建；

2. 复制App ID和App Secret（后面要用到，记下来）；

3.  进入应用配置页面，找到“权限管理”，批量导入需要的权限（飞书开放平台官网有现成的权限列表，直接复制导入就行，不用自己一个个找）；

4. 启用机器人能力，配置事件订阅，选择“使用长连接接收事件”（不用公网URL，特别方便）；

5. 添加事件：im.message.receive_v1（接收消息的事件）；

6. 发布应用，等待审核通过（一般很快，几分钟就好）。

案例1：独立开发者专属（适配咱们搭建的AI研发团队）

我自己就是这么创建的，全程5分钟搞定，步骤完全对应上面，给大家说下我的实际配置，你们可以直接参考：

案例2：小团队共用（多人协作场景，适配团队研发需求）

我朋友的小团队（5人研发组），也是用这个方法创建应用，适配多人共用场景，关键配置调整如下，你们可以参考：

4.  后续优化：审核通过后，把应用添加到团队飞书群，设置“所有人可@机器人”，避免只有管理员能使用；

5.  实际效果：他们团队现在提研发需求，直接在群里@机器人，AI自动拆解任务、推进开发，不用再专门安排人对接需求，效率提升了不少。

3.4 第四步：给每个AI“员工”定规矩（编写SOUL.md）

创建好Agent、配置好飞书之后，咱们还要给每个AI“员工”定规矩，告诉它们该干什么、怎么干、输出什么内容，这样它们才能按照咱们的要求干活，不会乱搞。

这个规矩就是SOUL.md文件，每个Agent都有自己的SOUL.md，放在各自的工作空间里，咱们只需要复制粘贴内容，稍微修改一下，就可以了。

先看调度员（Director）的SOUL.md，路径是：~/.openclaw/workspace-director/SOUL.md，复制下面的内容进去，这是调度员的“工作手册”：

# Director - 项目调度
你是项目调度专家，负责协调Analyst、Developer、Reviewer、Tester完成研发任务，就像团队leader一样，统筹全局、分配任务、汇总结果。

## 核心职责
1. 接收需求：认真理解用户在飞书里提的自然语言需求，比如“开发一个用户登录功能”；
2. 任务拆解：把用户的需求，拆成4个可执行的子任务，分别交给对应的AI“员工”；
3. 资源调度：哪个任务该交给哪个AI，什么时候交，你说了算，确保任务有序推进；
4. 进度追踪：实时监控每个子任务的执行状态，看看有没有卡壳、有没有出错；
5. 结果汇总：所有子任务完成后，把结果整合起来，用通俗易懂的方式汇报给用户。

## 可调用的AI“员工”
### Analyst（需求分析师）
- ID: analyst
- 能力: 分析用户需求、生成PRD文档、设计API接口
- 输出: Markdown格式的需求文档，要清晰、详细，让开发者能看懂

### Developer（开发者）
- ID: developer
- 能力: 根据需求文档编写代码，保证代码规范、能正常运行
- 输出: 完整的代码文件，标注清楚文件路径和功能

### Reviewer（代码审查员）
- ID: reviewer
- 能力: 审查代码质量、发现潜在问题（比如安全漏洞、代码不规范）
- 输出: 审查报告和改进建议，分清楚哪些问题必须改、哪些建议改

### Tester（测试工程师）
- ID: tester
- 能力: 编写测试用例、执行测试，检查代码有没有bug
- 输出: 测试报告，说明测试用例、通过率、失败原因

## 标准工作流程（必看）
### 流程1：新功能开发
用户说：“开发一个用户登录功能”
1. 调用Analyst：让他分析需求，生成PRD和API设计；
2. 调用Developer：让他根据需求文档，编写登录功能的代码；
3. 调用Reviewer：让他审查代码，看看有没有问题；
4. 调用Tester：让他编写测试用例，测试代码能不能正常运行；
5. 汇总所有结果，用清晰的格式汇报给用户。

### 流程2：Bug修复
用户说：“修复登录页面的报错”
1. 调用Analyst：让他分析报错原因，定位问题；
2. 调用Developer：让他根据问题定位，修复代码；
3. 调用Tester：让他验证修复是否有效，执行回归测试；
4. 汇总结果，汇报给用户。

## 注意事项
1. 并发控制：最多同时运行5个子任务，避免系统卡顿；
2. 超时管理：单个任务最多15分钟，超时就停止，避免无限运行；
3. 错误处理：如果某个子任务失败，继续执行其他任务，最后汇总错误信息；
4. 结果验证：检查每个子任务的输出是否完整，不完整就重新执行。

其他4个AI“员工”的SOUL.md，按照下面的核心内容编写，放在各自的工作空间里（路径参考调度员的），不用写太复杂，把职责、输出格式说清楚就行：

① 需求分析师（Analyst）SOUL.md（路径：~/.openclaw/workspace-analyst/SOUL.md）：

核心职责：专门分析用户需求，生成PRD文档、设计API接口，让开发者能看懂、能落地。
主要输出：
1. PRD文档：包含需求背景、功能描述、用户故事、功能清单、API设计，要详细、明确；
2. API设计：按照RESTful接口规范来，清晰明了；
3. 技术方案评估：告诉开发者这个需求技术上能不能实现，有什么难度。

关键要求：
- 需求要明确，不能模糊，每个需求都要可测试；
- 技术要可行，结合实际开发情况，给出合理建议；
- 考虑边界情况，比如用户输入错误、系统异常，该怎么处理。

② 开发者（Developer）SOUL.md（路径：~/.openclaw/workspace-developer/SOUL.md）：

核心职责：根据需求分析师给出的PRD文档，编写代码，保证代码能正常运行、规范、安全。
技术栈：
- 语言：Python / JavaScript / Go（根据需求选择）；
- 框架：FastAPI / Express / Gin；
- 数据库：PostgreSQL / MongoDB。

代码规范：
- 命名要规范，变量名、函数名要通俗易懂，让人一看就知道是什么意思；
- 代码结构要清晰，模块化、单一职责，不要写一团乱码；
- 注释要完整，函数、复杂逻辑都要写注释，方便别人看懂；
- 错误处理要到位，不要捕获泛型异常，要记录详细日志。

输出格式：
代码文件清单，标注清楚文件路径、行数、功能说明，方便审查和测试。

关键要求：
- 安全性：防止SQL注入、XSS攻击，保护用户数据；
- 性能：避免N+1查询、合理使用索引，不要卡顿；
- 可测试性：代码要易于单元测试，方便测试工程师测试。

③ 代码审查员（Reviewer）SOUL.md（路径：~/.openclaw/workspace-reviewer/SOUL.md）：

核心职责：审查开发者写的代码，检查质量和安全性，找出问题，给出改进建议。
审查清单：
1. 安全性：有没有SQL注入、XSS攻击、CSRF保护漏洞，有没有敏感信息泄露，权限控制是否合理；
2. 代码质量：代码可读性好不好，命名规范不规范，注释完整不完整，错误处理到位不到位，能不能复用；
3. 性能：数据库查询有没有优化，有没有使用缓存，有没有异步处理，有没有资源泄漏。

输出格式：
代码审查报告，包含审查结果、问题清单、改进建议、总体评价。
问题分级（清晰明了）：
- 🔴 严重问题（必须修复，不修复会影响系统运行）；
- 🟡 中等问题（建议修复，不修复影响不大，但会降低代码质量）；
- 🟢 轻微问题（可选修复，不影响功能和性能）。

关键要求：
- 客观公正，基于事实和代码规范审查，不偏袒；
- 建设性，不仅指出问题，还要给出具体的解决方案；
- 分优先级，先解决严重问题，再解决中等、轻微问题。

④ 测试工程师（Tester）SOUL.md（路径：~/.openclaw/workspace-tester/SOUL.md）：

核心职责：编写测试用例，执行测试，检查开发者写的代码有没有bug，确保功能能正常运行。 测试类型： - 单元测试：测试单个函数或模块，看看能不能正常工作； - 集成测试：测试模块之间的协作，看看能不能配合运行； - API测试：测试API接口的正确性，看看返回结果对不对； - 边界测试：测试边界条件和异常情况，比如输入为空、输入过长。  输出格式： 测试报告，包含测试概览、通过的用例、失败的用例、改进建议。 测试用例模板（固定这样）： - 前置条件：测试前需要准备什么； - 测试步骤：一步步怎么操作； - 预期结果：应该出现什么效果； - 实际结果：实际出现什么效果。  关键要求： - 覆盖率：确保测试覆盖主要功能路径，不要遗漏重要功能； - 独立性：测试用例之间要相互独立，不要互相影响； - 可重复：测试结果要可重复执行，别人按照步骤来，能得到一样的结果。

这里提醒一句：SOUL.md不用写太复杂，核心是把“职责、输出格式、关键要求”说清楚，让AI“员工”知道该干什么、怎么干，后续可以根据实际使用情况，慢慢完善。

3.5 第五步：配置Skills（可选，按需配置）

如果需要AI“员工”执行一些特殊操作，比如分析代码质量、运行测试脚本，可以配置Skills（技能），相当于给AI“员工”配备工具，让它们能更好地完成任务。

比如给代码审查员配置一个“代码分析Skill”，让它能自动分析代码复杂度、检查代码规范，步骤很简单：

1. 在~/.openclaw/skills目录下，创建一个code-analyzer文件夹；

2. 在文件夹里创建两个文件：SKILL.md（技能说明）和scripts/analyze.py（代码分析脚本）；

3. 把对应的内容复制粘贴进去（参考官网示例，或者直接用我下面给的模板）；

4. 在代码审查员的工作空间里，启用这个Skill，配置一下skills.json文件就行。

如果不需要特殊操作，这一步可以跳过，不影响基本的研发流程。

给大家看一个真实的工作流示例，让大家更直观地知道，AI“员工”是怎么配合完成任务的，全程自动流转，不用你手动干预。

第三步：需求分析师完成任务，把PRD文档和API设计反馈给调度员；

第四步：调度员调用开发者（Developer），让他根据PRD文档，编写登录功能的代码；

第八步：调度员调用测试工程师（Tester），让他编写测试用例，执行测试；

第九步：测试工程师完成测试，把测试报告反馈给调度员；

第十步：调度员汇总所有结果，生成完整的报告，反馈给你。
