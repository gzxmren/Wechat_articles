第一章：理解OpenClaw的记忆机制

1.1 OpenClaw不是一只普通的大龙虾

在深入讨论"失忆"之前，我们必须先理解OpenClaw的记忆是如何工作的。

与ChatGPT、Claude这样的聊天机器人不同，OpenClaw采用了一种非常激进的设计哲学：一切以文件为先（File-first design）。

OpenClaw的所有认知状态都存储为纯文本Markdown文件。不是数据库，不是Redis缓存，就是简单的文本文件。你可以用任何文本编辑器打开它们、编辑它们、甚至用Git做版本控制。

这种设计的好处是透明——你完全掌控自己的数据。坏处也显而易见：如果信息没有写入文件，它就不存在。

1.2 OpenClaw记忆的四层结构

根据OpenClaw官方文档和社区深度分析，OpenClaw的记忆实际上由四个不同层次组成，它们以不同方式工作，也会在不同情况下失效：

第一层：启动文件（Bootstrap Files）

这是你最可靠的盟友。每次会话启动时，OpenClaw都会从磁盘加载这些文件：

·SOUL.md - 代理的人格和道德核心

·AGENTS.md - 会话逻辑和检查清单

·USER.md - 代理对你的模型认知

·IDENTITY.md - 代理的自我形象

·MEMORY.md - 策划的长期记忆

·TOOLS.md - 工具描述和使用方法

这些文件之所以可靠，是因为它们每次会话都会从磁盘重新加载，而不是从对话历史中继承。无论上下文如何压缩，只要文件在磁盘上，它们就会被注入。

启动成本：读取这些文件每次会话需要消耗约4,000-10,000个token。

第二层：每日记忆日志（Daily Notes）

位于`memory/YYYY-MM-DD.md`，这是OpenClaw的短期记忆库。

·今天是`memory/2026-03-14.md`

·昨天是`memory/2026-03-13.md`

每天开始新会话时，OpenClaw会读取今天和昨天的日志（如果是主会话，还会读取MEMORY.md）。这种设计模仿了人类的工作记忆 vs 长期记忆的区别。

关键限制：

·日志是追加写入的，但不会被自动加载到上下文中

·需要通过`memory_search`主动检索

·超过两天的内容需要显式搜索

第三层：会话转录（Session Transcript）

这是最容易让你产生"它记住了"幻觉的一层。

当你与OpenClaw对话时，所有消息（你的输入、它的回复、工具调用结果）都会被保存为JSONL文件。这让会话看起来可以"继续"，但实际上存在巨大陷阱：

上下文压缩（Compaction）会在上下文窗口满时触发。 older的消息会被总结成"紧凑型条目"，原始消息从模型视野中消失——即使磁盘上的JSONL文件仍然存在。

这意味着什么？模型看不到它，就像它不存在一样。

第四层：LLM上下文窗口（Context Window）

这是最核心也最脆弱的一层。

上下文窗口是模型实际"看到"的内容容器，包括：

·系统提示词（所有规则、工具、技能列表）

·对话历史

·工具调用和结果

·附件/转录内容

复杂任务很容易出现上下文窗口溢出。

关键问题：当窗口满时，OpenClaw会进行Compaction——将旧对话总结压缩。这个过程中，任何只存在于对话中的指令都会丢失。

1.3 为什么会失忆？ compaction的真相

现在我们来揭开OpenClaw失忆的根本原因。

压缩 vs 剪枝：很多人搞混了

压缩（Compaction）是对整个对话历史的总结。它会改变模型未来看到的内容，在上下文窗口满时触发，影响一切（用户消息、助手消息、工具调用）。它是有损耗的、永久的。

剪枝（Pruning）只修剪工具结果（toolResult），不影响用户和助手消息。它针对每次请求临时进行，不会触及磁盘上的会话历史。它是无损的、临时的。

大多数人把这两个概念搞混了。压缩才是那个真正危险的杀手。

压缩的生命周期

理想情况（Maintenance Compaction）：

1.上下文接近限制

2.预压缩内存刷新（Memory Flush）触发——代理自动将重要内容保存到磁盘

3.压缩发生，旧对话被总结

4.代理继续使用摘要+最新消息+磁盘文件

糟糕情况（Overflow Recovery）：

5.上下文太大，API拒绝请求

6.OpenClaw进入紧急修复模式

7.所有内容被一次性压缩，只为恢复工作

8.没有内存刷新，没有保存重要内容

9.最大上下文丢失

什么会 survive？什么会 die？

无法 survive 压缩的内容（这是#1杀手）：

·嵌入在对话中的指令

·偏好、纠正、会话中做的决定

·压缩前分享的所有图片

·工具结果和它们的上下文

·原始指令的细微差别（总结是有损耗的）

可以 survive 的内容：

·所有工作区文件：SOUL.md、AGENTS.md、USER.md、MEMORY.md、TOOLS.md

·每日记忆日志（通过搜索按需访问，不是重新注入）

·代理在压缩前写入磁盘的任何内容

最重要的原则：如果它没有写入文件，它就不存在。

第二章：六大解决方案实战指南

现在进入实战部分。经过一个月的摸索，我总结出六种应对OpenClaw失忆的有效策略。

2.1 方案一：建立SKILL，固化操作流程

这是最基础也最重要的一招。不要把操作流程留在对话里，要把它变成SKILL文件。

SKILL是OpenClaw的技能系统，位于`/root/.openclaw/skills/`目录下，每个SKILL都有一个`SKILL.md`文件。

为什么SKILL不会失忆？

因为SKILL文件是按需加载的。虽然系统提示词中包含一个紧凑的SKILL列表（名称+描述+位置），但实际的SKILL指令默认不会被注入。

当代理需要使用某个SKILL时，它会显式调用`read`工具来读取SKILL.md文件。这意味着：

·SKILL内容不会被会话上下文压缩影响

·每次使用都是新鲜加载的

·你可以在SKILL中定义详细的操作流程

SKILL的进阶用法

1. 多SKILL组合

你可以为不同类型的任务创建不同的SKILL：

·`company-research` - 公司深度研究

·`market-analysis` - 市场分析

·`report-generation` - 报告生成

·`email-drafting` - 邮件起草

2. 在AGENTS.md中强制加载SKILL

如果你希望某个SKILL在每次会话时都可用，可以在AGENTS.md中添加：

## 可用技能

- vc-research-workflow - VC投资研究流程

- email-send - 邮件发送

- tavily-search - 深度搜索

3. 定期更新SKILL

当发现流程有问题时，直接编辑SKILL.md文件。下次使用时，OpenClaw会加载最新版本。

2.2 方案二：强化词机制，给关键信息加粗

这个方案听起来有点"笨"，但效果出奇地好。

为什么强化词有效？

大语言模型对加粗内容、大写内容、 emoji 有天然的注意力倾向。虽然这不是什么高深的技术，但在实践中确实很有效。

哪些词需要强化？

1. 关键约束词

⚠️ **绝对禁止**在没有用户确认的情况下执行删除操作。

✅ **必须先问**用户："确定要删除这{X}条记录吗？"

2. 流程关键词

**Step 1**（必须执行）：读取MEMORY.md

**Step 2**（必须执行）：检查历史记录

**Step 3**（必须执行）：搜索最新信息

3. 数据验证词

**关键数据**必须标注来源：

**重要日期**必须确认：2026年3月14日

**核心结论**必须复核：本项目已通过三轮融资

4. 安全提醒词

�� **安全警告**：此操作涉及资金转账，需要双重确认

�� **隐私保护**：MEMORY.md**仅在主会话中加载**，不会在群聊中泄露

在SOUL.md中植入强化词

SOUL.md是代理的"人格核心"，每次会话都会加载。在这里植入强化词效果最好：

# SOUL.md

## 我的核心工作原则

1. **文件优先原则**：所有重要信息必须写入文件，对话只是临时容器

2. **来源必标原则**：**每个关键数据必须有来源链接**，没有来源的数据不可靠

3. **双重确认原则**：涉及删除、修改、发送的操作，**必须先获得用户明确确认**

4. **每日归档原则**：每天结束前必须更新`memory/YYYY-MM-DD.md`

## 记忆强化词

当我看到以下关键词时，必须特别注意：

- **绝对禁止** = 红线，不可触碰

- **必须先** = 强制顺序

- **关键数据** = 需要来源标注

- **重要日期** = 需要二次确认

- **核心结论** = 需要复核验证

结果OpenClaw在执行过程中，每次遇到这些强化词都会特别小心，最终生成的报告质量明显更高。

2.3 方案三：审核机制，打回去重做

这是一种"防呆"设计——如果OpenClaw没有按照流程执行，就打回去让它重做。

建立审核清单

在AGENTS.md中建立强制审核机制：

## 任务完成审核清单

任何任务在标记为"完成"之前，必须通过以下审核：

### 审核点1：文件写入检查

- [ ] 是否创建了新的研究文件？

- [ ] 文件是否包含完整的来源链接？

- [ ] 文件是否保存在正确的目录？

### 审核点2：索引更新检查

- [ ] 是否更新了index.json？

- [ ] 统计信息是否准确？

- [ ] 日期格式是否正确？

### 审核点3：通知发送检查

- [ ] 是否发送了飞书通知？

- [ ] 通知内容是否包含关键发现？

- [ ] 文件路径是否正确？

如果以上任何一项未通过，**必须返回重做**，不得标记任务完成。

自动化审核脚本

对于高频任务，可以写一个简单的审核脚本：

"""审核研究任务是否完整"""

'report_file_exists': False,

'index_updated': False,

}

# 检查报告文件

report_path = f"/root/.openclaw/workspace/session_*_{company_name}.md"

# ... 检查逻辑

# 检查索引更新

# ... 检查逻辑

# 检查通知记录

# ... 检查逻辑

if not all(checks.values()):

return False, f"审核未通过: {checks}"

return True, "审核通过"

2.4 方案四：避免频繁切换话题

这是最容易被忽视，但也最致命的一个点。

OpenClaw是"小丑鱼"，不是"大龙虾"

虽然Logo是一只大龙虾，但在记忆力方面，OpenClaw更像小丑鱼——记忆力很短。

为什么频繁切换话题会导致失忆？

1.上下文污染：每次切换话题，新内容会挤占上下文空间

2.压缩触发：快速的话题切换容易触发Compaction

3.记忆稀释：代理难以判断哪些信息应该被长期保留

如何正确管理话题切换？

策略1：使用独立会话

对于完全不同的任务，开启新会话而不是在现有会话中切换。

❌ 错误做法：

在同一个会话中：

- 先研究A公司

- 突然问"今天天气怎么样"

- 再回来研究B公司

- 再讨论晚饭吃什么

- 再要求生成报告

✅ 正确做法：

会话1：专门研究A公司

会话2：专门研究B公司

会话3：闲聊和日常事务

策略２：显式总结再切换

如果必须在同一会话中切换话题，先显式保存当前状态：

我们现在暂停A公司的研究。请把目前的研究发现保存到memory/2026-03-14.md，然后回答我一个问题...

好，现在回到A公司的研究。请重新读取之前保存的研究进展，继续完成...

策略３：定期压缩管理

主动触发压缩，而不是等它自动发生：

/compact

这会强制OpenClaw总结旧对话，释放上下文空间。

2.6 方案六：建立MEMORY管理机制

这是最高级也最复杂的一招——建立完整的记忆管理系统。

MEMORY压缩的艺术

OpenClaw官方文档提到一个关键数字：700字符。

每次Gateway调用（无论是聊天消息还是自动任务），都会有一个"记忆前奏"（Memory Prelude）作为系统消息注入。这是MEMORY.md的压缩版本，限制在约700字符。

这意味着：

·不是完整的MEMORY.md

·是最信息密集的摘要

·空行被剔除

·超出限制会被硬截断

如何写好MEMORY.md？

原则1：倒金字塔结构

最重要的信息放在最前面：

# MEMORY.md

## 关键事实（必须记住）

## 当前活跃项目

- AI智能硬件研究（进行中，已完成21家公司）

- 具身智能监控（已部署）

## 重要偏好

- 报告格式：Markdown，需要来源链接

- 沟通方式：飞书通知 + 邮件备份

- 研究深度：每公司30分钟内，8000-10000字

## 历史归档（次要信息）

[更详细的历史记录...]

原则2：定期蒸馏

每天或每周，回顾`memory/YYYY-MM-DD.md`，提取重要信息更新到MEMORY.md：

## 本周重要进展

- 2026-03-10: 完成XXX光电深度研究

- 2026-03-11: 完成xxx跟踪研究

- 2026-03-14: 完成21家AI硬件公司研究

原则3：使用结构化格式

便于机器和人都快速理解：

## 用户画像 ��

| 属性 | 值 |

|------|-----|

| 姓名 | Ryan |

| 职业 | VC投资人 |

| 关注赛道 | AI硬件、具身智能 |

| 具身智能监控 | 已部署 | - | 2026-03-14 |

自动内存刷新（Automatic Memory Flush）

OpenClaw有一个隐藏功能：当会话接近自动压缩时，会触发一个静默的代理回合，提醒模型在上下文被压缩之前写入持久记忆。

配置在`openclaw.json`中：

{

"agents": {

"defaults": {

"compaction": {

"memoryFlush": {

"enabled": true,

"softThresholdTokens": 4000,

"systemPrompt": "Session nearing compaction. Store durable memories now."

}

}

}

}

}

这会在上下文快满时自动触发，提醒你保存重要内容。

使用memory_search和memory_get

OpenClaw提供了两个关键工具：

memory_search - 语义搜索

搜索："智谷天厨的融资信息"

这会搜索所有memory文件，找到相关内容。

memory_get - 精确读取

读取：memory/2026-03-14.md，从第1行开始，读取50行

这是精确读取特定文件的工具。

使用外部记忆系统（Mem0）【这个还没有实际尝试】

如果内置记忆系统不够，可以考虑使用Mem0等外部记忆插件：

openclaw plugins install @mem0/openclaw-mem0

Mem0提供：

·自动回忆（Auto-Recall）：每次对话前搜索相关记忆

·自动捕获（Auto-Capture）：对话后自动提取并保存事实

·独立于上下文窗口：不会被Compaction影响

第四章：常见问题FAQ

Q1: OpenClaw的失忆问题能解决吗？

A: 不能完全"解决"，但可以有效"管理"。OpenClaw的失忆是架构层面的权衡（文件优先设计），不是Bug。通过本文的六种方案，可以将失忆带来的影响降到最低。

Q2: 我应该把所有信息都写入文件吗？

A: 不是所有信息，而是重要信息。判断标准：如果明天再问，你还希望它记得，那就写入文件。

Q3: SKILL和MEMORY.md有什么区别？

A: SKILL是"怎么做"（操作流程），MEMORY.md是"是什么"（事实信息）。SKILL每次使用都要读取，MEMORY.md每次会话自动加载。

Q4: 压缩（Compaction）一定会丢失信息吗？

A: 是的，压缩是有损耗的。但可以通过预压缩内存刷新（Memory Flush）和及时写入文件来减少损失。

Q5: 使用更大的上下文窗口模型能解决问题吗？

A: 能缓解，但不能根治。更大的窗口只是延迟了压缩的发生，不能避免压缩。而且大窗口模型的成本和延迟都更高。

Q6: 我应该多久更新一次MEMORY.md？

A: 建议每天或每周。对于活跃项目，每天更新；对于稳定信息，每周回顾一次即可。我自己从来没有手动更新，系统自动更新的。

Q7: 有没有一劳永逸的解决方案？

A: 没有。但可以通过建立良好的工作习惯（文件优先、及时归档、定期蒸馏）来最小化失忆带来的影响。
