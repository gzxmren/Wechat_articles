Openclaw 背后的技术架构，记忆出奇的简洁
C̶l̶a̶w̶d̶b̶o̶t̶ M̶o̶l̶t̶b̶o̶t̶ Openclaw 背后的技术架构，有非常多值得思考的地方：串行优先、简化存储、浏览器语义抽象、安全设计等等。

架构流程 - 六步处理链路：
1. Channel Adapter — 标准化不同渠道的消息格式
2. Gateway Server — 核心协调器，路由到正确会话
3. Agent Runner — 选择模型、组装提示词、管理上下文
4. LLM API Call — 流式调用，支持多提供商
5. Agentic Loop — 工具调用循环，默认最大 ~20 轮
6. Response Path — 返回响应，持久化会话

Lane-based Command Queue
传统 async/await 在多智能体场景下的问题：日志交错、竞态条件、调试困难。
OpenClaw 的解决方案：Lane（通道）抽象
· 每个会话有专属 Lane
· 默认串行执行，仅显式标记的低风险任务并行
· 心智模型从"需要锁什么"转变为"什么可以安全并行"
这与 Cognition 的观点一致：序列化是默认架构，而非事后补救。

内存系统
出乎意料地简单：
· 会话历史：JSONL 文件
· 长期记忆：Markdown 文件（MEMORY.md）
· 检索：SQLite 向量搜索 + FTS5 关键词
写入方式：无专门 API，智能体直接用文件写入工具写 Markdown。
特点：无记忆合并、无压缩、无遗忘曲线，新会话开始时自动生成上次对话摘要。

安全机制 - 白名单 + 黑名单组合：
· 白名单：用户可配置，预批准安全命令（jq、grep、sort 等）
· 黑名单：阻止危险构造（命令替换、重定向、链式执行、子 shell）
设计哲学：给予用户所允许的最大自主权。

其他特性
· 动态系统提示词：根据工具、技能、内存、用户身份等动态构建
· 子智能体：可生成子智能体（但子智能体不能再生成），通过 session_send 通信
· 上下文压缩：接近限制时，历史分块 → LLM 摘要 → 替换旧消息
