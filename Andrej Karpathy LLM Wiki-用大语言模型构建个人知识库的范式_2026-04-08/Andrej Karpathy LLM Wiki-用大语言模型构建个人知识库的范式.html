用大语言模型构建个人知识库的范式

这是一份理念文档，设计用途是复制粘贴到你自己的 LLM 智能体中（如 OpenAI Codex、Claude Code、OpenCode / Pi 等）。它旨在传达核心思路，具体细节由你的智能体与你协作完成。

· · ·

核心理念

大多数人使用 LLM 处理文档的方式本质上是 RAG：上传一批文件，LLM 在查询时检索相关片段，生成回答。这固然可行，但 LLM 每次都在从零开始重新发现知识，毫无积累可言。遇到需要综合五份文档才能回答的微妙问题，LLM 每次都得重新搜寻、拼凑相关碎片。

什么都没有沉淀下来。NotebookLM、ChatGPT 文件上传以及大多数 RAG 系统都是如此。



核心洞察

与其在查询时从原始文档中检索，不如让 LLM 增量式地构建和维护一个持久化的 Wiki——一组结构化、相互链接的 Markdown 文件，横亘在你与原始资料之间。

当你添加新的资料来源时，LLM 不只是将其编入索引等待日后检索，而是通读内容、提取关键信息，并将其整合进现有的 Wiki——更新实体页面、修订主题摘要、标注新数据与旧结论的矛盾之处、强化或挑战正在演进中的综合判断。知识只需编译一次并持续更新，而非每次查询时重新推导。

这就是关键区别：Wiki 是一个持久的、不断复利增长的产物。交叉引用已经建好，矛盾之处已经标出，综合分析已经反映了你读过的全部内容。每添加一个来源、每提出一个问题，Wiki 都会变得更加丰富。

❝
Obsidian 是 IDE，LLM 是程序员，Wiki 是代码库。
❞

你几乎不需要亲手编写 Wiki——一切由 LLM 撰写和维护。你负责的是搜集资料、探索方向、提出好的问题。LLM 承担所有苦力活——概括、交叉引用、归档、记录——正是这些工作让知识库随着时间推移真正发挥价值。

适用场景

个人成长

追踪目标、健康、心理、自我提升——归档日记、文章、播客笔记，逐步构建一幅关于自己的结构化全景图。

学术研究

围绕某个课题持续深入数周乃至数月——阅读论文、文章、报告，增量式地构建一个包含演进中论点的综合 Wiki。

阅读一本书

逐章归档，为角色、主题、情节线索及其关联建立页面。读完时你就拥有了一部丰富的伴读 Wiki。想想 Tolkien Gateway——数千个相互链接的页面，由志愿者社区历经多年建成。你完全可以在阅读过程中个人化地构建类似的东西。

企业 / 团队

由 LLM 维护的内部 Wiki，以 Slack 讨论、会议记录、项目文档、客户通话为素材。Wiki 之所以能保持更新，是因为 LLM 承担了团队中没人愿意做的维护工作。

此外还有：竞品分析、尽职调查、旅行规划、课程笔记、兴趣深耕……凡是需要持续积累知识并希望它有条理的场景，皆可适用。

三层架构

整个体系共分三层，各司其职：

1原始资料 Raw Sources

你精心筛选的来源文档合集——文章、论文、图片、数据文件。它们是不可变的，LLM 只读不改。这是你的事实根基。

2Wiki 知识层

由 LLM 生成的 Markdown 文件目录——摘要、实体页面、概念页面、对比分析、总览、综合判断。这一层完全由 LLM 掌控。你只管读，LLM 负责写。

3纲要 Manifest

一份配置文档（如 CLAUDE.md），告诉 LLM Wiki 的结构规范、格式约定和工作流程。这是核心配置——正是它让 LLM 成为训练有素的 Wiki 维护者，而非泛泛的聊天机器人。

· · ·

三大操作流程

⟐ 消化 Ingest

将新来源放入原始资料集合，然后告诉 LLM 进行处理。典型流程：LLM 阅读来源 → 与你讨论关键要点 → 撰写摘要页面 → 更新索引 → 更新相关实体和概念页面 → 追加日志记录。单个来源可能涉及 10-15 个 Wiki 页面的更新。

⟐ 查询 Query

对 Wiki 提问，LLM 搜索相关页面后综合生成带引用的回答。重要洞察：好的回答可以作为新页面归档回 Wiki。你的对比、分析、发现的关联——这些都有价值，不应湮没在聊天记录里。你的探索也能像消化来源一样在知识库中持续复利。

⟐ 巡检 Lint

定期对 Wiki 做健康检查：排查页面间矛盾、已被取代的陈旧结论、孤立页面、缺失的概念页面、遗漏的交叉引用、可通过搜索填补的数据缺口。LLM 还擅长建议新的探索方向。这使得 Wiki 在不断壮大的同时保持健康。

· · ·

索引与日志

两个特殊文件帮助 LLM（以及你）在 Wiki 增长过程中高效导航：

index.md

面向内容——Wiki 的完整目录，列出每个页面的链接和一句话摘要。LLM 回答查询时先读索引定位，再深入阅读。在约 100 个来源、数百页面的规模下，效果出奇地好。

log.md

面向时间线——只追加的操作记录。记录消化了什么、查询了什么、何时巡检。为你呈现 Wiki 的演进时间线，也帮 LLM 了解近期工作。

工具与技巧

Obsidian Web Clipper

浏览器扩展，能将网页文章转换为 Markdown，非常适合快速将资料收入原始资料集合。

Obsidian 关系图谱

观察 Wiki 全貌的最佳方式——什么与什么相连、哪些页面是枢纽、哪些是孤岛。

qmd 搜索引擎

本地 Markdown 文件搜索引擎，支持 BM25 / 向量混合搜索和 LLM 重排序，提供 CLI 和 MCP 两种接口。

Marp + Dataview

Marp 可直接从 Wiki 内容生成幻灯片；Dataview 可对页面 frontmatter 执行查询，生成动态表格和列表。

Git 版本管理

Wiki 本质上就是一个 Markdown 文件组成的 Git 仓库——版本历史、分支管理、多人协作，开箱即用。

为何有效


维护知识库最令人厌倦的部分不是阅读和思考，而是琐碎的管理工作——更新交叉引用、保持摘要时效性、标注矛盾、在数十个页面间维持一致性。

人类之所以放弃 Wiki，是因为维护负担的增速远超价值的增速。

LLM 不会厌倦，不会忘记更新某个交叉引用，而且一次操作就能触及 15 个文件。Wiki 之所以能持续维护，是因为维护成本趋近于零。

人的职责是筛选资料、引导分析、提出好问题、思考这一切的意义。其余的，交给 LLM。

历史回响

这个理念在精神上与范内瓦·布什 1945 年提出的 Memex 一脉相承——一个私人化的、经过主动策展的知识存储，文档之间以联想性的线索相互串联。布什的愿景其实更接近这里所描述的，而非后来万维网的实际面貌。他未能解决的问题是：谁来做维护。LLM 解决了这个问题。

写在最后

本文档刻意保持抽象。它描述的是理念，而非某个具体实现。确切的目录结构、纲要规范、页面格式、配套工具——这些都取决于你的领域、你的偏好和你选用的 LLM。

正确的使用方式是将本文档分享给你的 LLM 智能体，然后与它协作，落地一个契合你需求的版本。

本文档的唯一使命是传达这个范式。
剩下的，你的 LLM 自会搞定。
