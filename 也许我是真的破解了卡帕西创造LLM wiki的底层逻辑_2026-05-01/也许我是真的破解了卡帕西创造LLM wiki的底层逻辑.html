也许我是真的破解了卡帕西创造LLM wiki的底层逻辑

最近我在研究 Karpathy 提出的 LLM Wiki。

一开始，我以为它只是一个新的知识管理方法：把资料丢给 LLM，让它帮我总结、归类、整理成 Markdown 文档。

但聊着聊着，我突然意识到，LLM Wiki 真正有意思的地方，可能不只是“用 AI 做笔记”。

高手的创新，往往不是凭空发明一个全新的东西，而是把已有技术中的可迁移结构，抽象、泛化，然后重新组合到一个新场景里。

这篇文章，就是我对 LLM Wiki 底层逻辑的一次拆解。

一、RAG 是内容解释器，LLM Wiki 是内容编译器

我最开始理解 LLM Wiki，是从它和 RAG 的区别开始的。传统 RAG 更像一个“内容解释器”。你问一个问题，它临时去资料库里搜索相关片段，然后让 LLM 现场阅读、理解、拼接、回答。

原始资料 -> 切块/向量库 -> 用户提问 -> 临时检索 -> 临时综合 -> 回答

这很像解释器：每次运行，每次解释。

而 LLM Wiki 更像一个“内容编译器”。它不是等你提问时才理解资料，而是在资料进入系统时，就先让 LLM 把 raw 原始资料读一遍、理解一遍、提炼一遍，然后编译成结构化 Markdown 页面。

raw 原始资料 -> LLM 编译 -> Markdown Wiki 知识网络 -> 后续查询

所以 LLM Wiki 的核心不是“有没有检索”，而是：

RAG：查询时临时解释 raw

LLM Wiki：摄入时先把 raw 编译成 wiki

RAG 是内容解释器，LLM Wiki 是内容编译器。

二、Wiki 不是 raw 本身，而是 raw 的精华节点网络

LLM Wiki 里的 wiki，不是把原始资料原封不动存进去。它更像是 raw 资料的提炼版。

raw 是证据，wiki 是理解。

比如我放进去一篇文章，LLM 不只是生成一篇总结，而是可能生成一组互相关联的 Markdown 页面：

summaries/某篇文章摘要.md

concepts/RAG.md

concepts/LLM Wiki.md

concepts/知识编译.md

entities/Karpathy.md

synthesis/RAG 与 LLM Wiki 对比.md

这些页面不是孤立的，它们会通过 [[Wiki 链接]] 连接起来；在 Obsidian 这类工具里，反向链接会被自动计算和展示。

LLM Wiki = raw 知识的精炼节点 + 节点之间的显式链接网络

每个页面可以看成一个节点。节点可能是一篇文章的摘要、一个概念、一个人物、一个项目、一次高质量问答，或者一个跨资料综合分析。节点之间的链接，就是知识之间的关系。

所以 LLM Wiki 并不是“一篇超级总结”，而是一个会逐渐生长的知识网络。

三、LLM Wiki 像神经网络，但又不是神经网络

GPT 的训练过程，大概是：

海量 raw 语料 -> 训练 -> 亿万/万亿级参数

知识被压缩进模型权重里，成为一种隐式表示。而 LLM Wiki 的过程是：

raw 资料 -> LLM 编译 -> Markdown 知识节点网络

两者都在做一件事：把大量原始信息，转化成后续可高效调用的知识表示。但区别在于：

对比

GPT 参数

LLM Wiki

知识形态

隐式权重

显式节点网络

可读性

人看不懂

人能读

可修改

很难

直接改 Markdown

可追溯

弱

可链接 raw 来源

更新方式

再训练/微调

ingest / update / lint

适合内容

通用知识

个人 / 团队 / 项目知识

所以我后来形成了一个理解：GPT 是把知识压缩进参数空间；LLM Wiki 是把知识组织成可读的节点网络。GPT 像内隐记忆，LLM Wiki 像外显记忆。

它没有试图改变模型的大脑，而是给模型配了一个可以持续维护的外部大脑。

四、LLM Wiki 和 Skill 机制很像

Skill 通常由 SKILL.md 这类 Markdown 指令文件定义；项目层面还可以配合 AGENTS.md、CLAUDE.md 等说明，告诉 LLM 遇到什么任务、应该怎么处理、按什么步骤执行、输出什么格式、注意哪些约束。

Skill = 教 LLM 怎么做事

LLM Wiki = 教 LLM 怎么维护一套长期知识网络

LLM Wiki 里的几个关键文件，也很像一套“知识操作手册”：

AGENTS.md    告诉 LLM 怎么工作

CLAUDE.md    告诉 Claude 这类工具怎么工作

SCHEMA.md    告诉 LLM Wiki 页面应该长什么样

wiki/*.md    存储已经编译好的知识状态

raw/         保存原始事实来源

所以 LLM Wiki 其实是把 Skill 这种“用 Markdown 约束 LLM 行为”的思想，用到了长期知识管理上。

从“让 LLM 完成一次任务”，迁移到“让 LLM 长期维护一个知识系统”。

五、LLM Wiki 也很像 Git

Git 在本地建立一个仓库，记录项目文件的变化：

文件

版本

diff

commit

log

rollback

而 LLM Wiki 也是本地文件库。只不过 Git 管的是代码变化，LLM Wiki 管的是知识变化。

raw/       原始资料

wiki/      编译后的知识页面

index.md   索引

log.md     更新记录

lint       知识健康检查

git diff   审查 LLM 改了什么

rollback   回滚错误修改

代码会变，所以需要 Git。知识也会变，所以 LLM Wiki 也需要版本记录、变更追踪、冲突处理、回滚机制。

把 Git 式版本管理，迁移到了知识管理。

六、LLM Wiki 如何更新？

LLM Wiki 不是“自动神秘生长”。它的更新机制其实很清楚：

发现新变化

-> 判断影响哪些知识节点

-> 新建/修改/补强节点

-> 更新链接

-> 标记冲突

-> 写入日志

-> 必要时人类审核

触发更新的来源可以是新增 raw 资料、旧 raw 修改、用户提出一个好问题，或者 Lint 发现断链、矛盾、孤立页面。

比如我新增了一篇关于 RAG 和 LLM Wiki 混合架构的文章，系统可能会：

补强 [[LLM Wiki]] 节点

补强 [[RAG]] 节点

新增 [[混合架构]] 节点

更新 [[RAG 与 LLM Wiki 对比]] 综合页

更新 index.md

追加 log.md

LLM Wiki 是一个可被 LLM 持续维修的外部知识网络。

七、一个 LLM Wiki 项目，其实就是一个 Markdown 文件夹

Karpathy 公开分享的并不是一个完整软件系统，而更像是一篇 llm-wiki.md 方法论文档。真正的 LLM Wiki 项目，可以非常简单，就是一个本地 Markdown 文件夹：

my-llm-wiki/

  AGENTS.md

  SCHEMA.md

 

  raw/

    inbox/

    assets/

 

  wiki/

    index.md

    log.md

    summaries/

    concepts/

    entities/

    synthesis/

    queries/

基础设施并不复杂。复杂的是背后的工作流：如何 ingest、如何分类、如何生成页面、如何建立链接、如何更新 index、如何记录 log、如何 lint、如何让人类审核关键变更。

这说明它的创新不在于某个新工具，而在于重新组织已有工具的方式。

八、我真正理解的底层逻辑：可迁移、可泛化的排列组合

聊到最后，我突然想到一句话：太阳底下没有新鲜事，排列组合就是创新。但后来我觉得还不够准确。更准确的说法应该是：

创新不是随机排列组合，而是基于可迁移、可泛化结构的排列组合。

低级组合，是把东西堆在一起。高级组合，是发现不同领域之间的结构同构。

代码需要 compile

知识也可以 compile

 

代码需要 lint

知识也可以 lint

 

代码需要 version control

知识也可以 version control

 

Skill 可以用 Markdown 指导 LLM 完成任务

Wiki 也可以用 Markdown 指导 LLM 维护知识

 

神经网络把 raw 语料压缩成参数

LLM Wiki 把 raw 资料压缩成 Markdown 节点网络

这才是 LLM Wiki 真正让我兴奋的地方。它不是简单地说 Markdown + LLM = 新笔记工具，而是把软件工程里的编译、lint、schema、版本管理、依赖网络、agent instruction，迁移到了知识管理领域。

九、所以我也许真的破解了它的底层逻辑

我现在对 LLM Wiki 的理解是：它不是一个工具，而是一种知识工程范式。

它把原始资料变成可读、可改、可追溯、可持续更新的 Markdown 知识网络。它的核心不是“AI 帮我总结文章”，而是：

让 LLM 维护一个本地的、白盒的、可审计的、持续生长的外部知识系统。

它综合了很多已有思想：

GPT 训练：raw 信息 -> 可复用表示

Skill 机制：Markdown 指导 LLM 行为

Git：本地文件库 + 版本演化

Wiki：页面节点 + Wiki 链接 / 反向链接

编译器：源材料 -> 中间表示

Lint：持续检查系统健康

Obsidian：人机共读的知识图谱

需要说明的是，这些更像是我对其思想来源的结构性类比，并不等于 Karpathy 明确说自己参考了 GPT 训练、Skill 或 Git。

所以，Karpathy 的厉害之处可能不在于发明了某个全新的零件，而在于他看见了这些旧零件之间的同构关系，然后把它们重新组合到了一个新问题上：如何让 LLM 帮人类长期维护知识？

这就是我理解的 LLM Wiki 底层逻辑：

raw 是事实来源

schema 是编译规则

agent 是维护者

wiki 是知识状态

git 是演化记录

lint 是健康检查

人类是最终治理者

LLM Wiki 不是把知识塞进模型大脑，而是在模型外部建立一个可读、可修、可追溯的知识大脑。
