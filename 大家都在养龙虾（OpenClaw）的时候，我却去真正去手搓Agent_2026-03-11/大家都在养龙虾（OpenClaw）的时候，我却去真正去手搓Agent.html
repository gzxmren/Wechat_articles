02 大胆的猜测
假设已经有了，小树就要进行论证。首先要理解这个整个流程大概是什么样子，LLM在里面又起到了什么作用？

回答这个问题之前，先回答一个问题，也是小树最近老在想的一个问题“自动化”和“智能化”哪个是最终目标？

小树觉得“自动化”才是，“智能化”只是“自动化”的一个过程。而这里“自动化”如果说的更片面一点，就像自动化测试一样，就是写的一段if else 固定在某个分支逻辑里面做某个事情。那在“智能化”里面，它就像是if else不是固定的，它就是一个“随机”或者说“类人”的逻辑决策应该命中哪个分支逻辑，然后执行某一个动作。

这里的“智能化”的一种实现就是LLM大模型了。用伪代码来看就是
const result = llm response #调用大模型
switch result:  case x:  #大模型决定用哪个skill    do x job;   case y: #大模型决定用哪个tool    do y job;  default :    break;  
然后呢，有一个点很实际，prompt越完整，LLM的效果越好，返回的结果越容易产生好的“化学反应”。但是呢，如果人肉补充，这个着实是一个又费时又费事的买卖。有没有什么好的办法呢，现在外面都是这样的文章，“上下文”，“记忆”，“rag”，比比皆是。

在小树的理解里面，什么都可以，只要能让prompt更“丰满”，效果就更好，有点像rag语义越接近越容易找到最像的那个答案。上面提到技术，都可以让这个prompt优化得更好，而如果自动填充或者智能填充，那就真的“美”了。

04 核心实现逻辑
大模型的时代，也是梦想走进现实的时代，人人都可以“张嘴就来”，人人都可以成为手执画笔的“马良”，谁都可以是提出“五彩斑斓的黑”需求的产品。

当然，这里小树不需要再说vibe coding 有多厉害了。主要说下小树的流程方案，这次为了让这个Agent更方便，更适配LLM的模式，先给一个流程图。
   User Chat Query  #用户输入问题       │       ▼  Skill Selection Mode  #减少token开销，指定技能  ├─ Manual Skill (specified by system)  │       │  │       ▼  │   Skills to call injected into prompt #把skill核心逻辑插入prompt  │  └─ Auto Skill (LLM decides)#技能过多的时候，可以用描述性语言，让llm自己决定用哪个skill          │          ▼     LLM determines required skills based on query + context          │          ▼      ┌─────────────────────────┐      │ Reasoning Loop (multi-turn) │#拿到skill以后开始循环执行，递归执行      └─────────────────────────┘                 │         ┌───────┼────────┐         ▼       ▼        ▼   Skill: analysis   Skill: rag_retrieve   Skill: reasoning         │             │                  │         ▼             ▼                  ▼   Analyze query    Tool: rag_search    Generate reasoning   & context        Knowledge Base      steps                     / Vector DB  #分析问题        #知识库知识自动补全prompt  #重新细化步骤         │             │                 │         └───────Context updates─────────┘                 │         Tool Execution (optional)                 │                 ▼           Skill: critique #检查分析                 │                 ▼       Final Output (Markdown / Chat)                 │                 ▼
          Skill： summary #总结结果                 │                 ▼          Return to User
说的更白话一点：
1. 用户写问题
USER : 帮我做一次MR/5的代码Review
1.1 问题里明确指明要调用的技能名称
USER : @code_review 帮我做一次MR/5 的代码Review
1.2 如果没有写明就拼接上一些prompt template 比如“以下是我有的技能 [{'name':'code_review','description':'***'},***], 帮我找到一个匹配的技能并输出 {'call_skill':'skillname'}”
USER : 我有2个技能，[{name:code_review,desc:code review against a mr},{name:requirement_analysis,desc: analysis***}], 根据{ user query： 帮我做一次MR/5 的代码Review}  帮我决定用哪个技能来执行， 用{'call_skill':'skillname'}的格式输出。  #这里是一个细节，必须指定格式输出，不然agent解析不了skill
2. agent 代码解析到这个技能信息以后，就开始进行如下操作来进行技能的执行。
LLM :  {'call_skill':'code_review'}
2.1 把skill的prompt template 注入到原先的user query 上（或者用脚本的方式实现技能，然后通过执行脚本来执行skill）
AGENT ：method:skill_invoke_by_name,args:'code_review'
2.2 如果skill还有绑定相应的tool，就把tool的tool definitions也放到llm的请求里，让llm确定具体需要调用的方法。（或者其他更高明的方法，进行描述和方法名的映射）
    AGENT：llm request : {messages:[{user:query + skill content}],    tools:[{method:git_mr_diff,args:[{name:mr_url,type:string}]}]}
2.2.1 大模型返回结果后，在2.1 的基础上再拼接上tool_calls方法调用返回的信息组成更完整的prompt
LLM ： { tool_calls: [{name: git_mr_diff , args: git@abc.com/***/1}]}
AGENT ：method: llm_prompt_build , args : {messages:[{user:query + skill content}],    tool_calls:[{method:git_mr_diff,args:[{name:mr_url,type:string}],    result:{****}}]}
2.2.2 如果技能有递归找下一个技能，就要继续展开执行延展的技能。（可以用子流程来执行）
AGENT : llm request :{messages:[{user: ****},{system:检查是否还有可以用的skill}]}
3. 继续和llm交互，进行问题的执行。
AGENT ： llm request：{ messages :[{....}]，tool_calls:[{name:git_mr_diff,args,response:[....] , }] }
4. 问题返回后，从头开始重复，直到无法通过技能带来更多的提示词的优化或补充。
AGENT : #call chat
5. 通过 summary的技能，进行最后的总结并回答给用户。
AGENT: LLM request : {messages:[{...},{type:assistant, content:***},{user:'帮我总结下'}]}
