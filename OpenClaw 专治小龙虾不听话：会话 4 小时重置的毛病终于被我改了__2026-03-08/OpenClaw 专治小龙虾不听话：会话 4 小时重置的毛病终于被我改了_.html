OpenClaw 专治小龙虾不听话：会话 4 小时重置的毛病终于被我改了。

痛点：AI Agent 会话 4 小时空闲就重置，上下文全没了

说个真实场景。

上周三下午，我正在跟 OpenClaw 的 robot1 调试一个复杂的电商选品流程。已经聊了四十多轮，从数据采集到评分模型，从 API 调用到结果过滤，上下文里全是细节。

三点半的时候，运营同事跑过来问个急事，处理完已经快六点了。我坐回电脑前，继续问 robot1："刚才那个评分模型的权重怎么调来着？"

它回我："您好，我是 robot1，有什么可以帮您？"

那一刻我真的无语。四十多轮的对话，全没了。

这就是 OpenClaw 默认的空闲会话重置机制。不提醒，不警告，静悄悄地就把你的上下文清掉了。

默认配置下，会话空闲 4 小时（14400 秒）后自动清理。下次交互时，Agent 失忆，你得从头再来。

对于只是偶尔问问天气、查查资料的轻度用户，这没问题。但对于把 Agent 当生产力工具的重度用户，这配置简直是灾难。

为什么默认这样设计？

刚开始我也觉得这设计有问题。后来翻了翻源码和文档，才明白这不是 bug，是有意为之。

Token 成本考虑

这是最现实的原因。

每个活跃会话都会占用上下文窗口。假设你开了 10 个 Agent，每个会话保留 50K tokens，就是 500K tokens。如果这些会话长期不活跃但不清理，内存占用会持续增长。

对于云端部署的场景，这意味着真金白银的成本。

性能优化

会话数据需要持久化存储和加载。清理不活跃会话可以减少三样东西：

  • 内存占用
  • 启动时的加载时间
  • 历史记录的检索开销

我有个朋友在 VPS 上跑 OpenClaw，配置不高（4 核 8G），默认配置下跑得很稳。后来他把会话清理全关了，跑了两周，内存直接飙到 90%，Agent 响应速度慢得像在思考人生。

安全边界

限制会话生命周期也是一种安全措施。想象一下，如果你的会话永久保留，某天设备丢了或者配置泄露，攻击者能看到你几个月的对话历史。

当然，对于本地部署、单机使用的用户，这个顾虑没那么强。

但话说回来，默认配置是面向大多数用户的。重度用户的需求，得自己调。

解决方案：禁用空闲过期 + 24h TTL 修剪

好消息是，OpenClaw 允许你自定义会话管理策略。

我现在的配置是这样的：

  • 禁用空闲过期
    ：会话不会因为空闲而被清理
  • 启用 TTL 修剪
    ：会话存活超过 24 小时后自动清理
  • 保留最近会话
    ：始终保留最近的 50 个会话

这样既能保持上下文连续性，又能防止无限增长。

说白了，就是用时间换空间。会话可以活 24 小时，但不会因为你中间离开几小时就消失。

配置步骤：修改 openclaw.json

找到你的 OpenClaw 配置文件。

配置文件位置：

C:\Users\你的用户名\.openclaw\openclaw.json

用你喜欢的文本编辑器打开（VSCode、Notepad++、甚至记事本都行）。配置需要添加到两个位置：

1. 顶层 session 配置（控制会话重置）

在文件顶层添加 session 段（与 gateway、agents 同级）：

{
"session":{
"reset":{
"mode":"daily",
"atHour":4,
"idleMinutes":525600
}
}
}

2. agents.defaults 配置（控制上下文修剪）

在 agents.defaults 段中添加 contextPruning：

{
"agents":{
"defaults":{
"contextPruning":{
"mode":"cache-ttl",
"ttl":"24h"
}
}
}
}

参数说明：

session.reset.idleMinutes：空闲过期时间（分钟）

  • 525600
     表示一年（相当于禁用空闲过期）
  • 默认值 240（4 小时）
  • 改成大数值后，Agent 不会再因为你离开几小时就失忆
  • 也可以设置具体分钟数，比如 1440 表示 24 小时

session.reset.mode：重置模式

  • "daily"
     表示每日定时重置
  • "idle"
     表示空闲超时重置
  • 推荐用 "daily" + 大数值 idleMinutes 组合

session.reset.atHour：每日重置时间

  • 4
     表示凌晨 4 点
  • 此时会创建新会话归档，但历史记录保留

contextPruning.mode：上下文修剪模式

  • "cache-ttl"
     表示基于缓存 TTL 修剪
  • 只保留最近一段时间内的详细上下文
  • 旧上下文会自动修剪，防止 Token 爆炸

contextPruning.ttl：缓存 TTL 时间

  • "24h"
     表示保留最近 24 小时
  • 超过此时长的上下文会被修剪
  • 这个值别设太大，否则 Token 消耗会失控

完整配置示例：

{
"gateway":{
"port":18789,
"mode":"local",
"auth":{
"mode":"token",
"token":"your-token-here"
}
},
"session":{
"reset":{
"mode":"daily",
"atHour":4,
"idleMinutes":525600
}
},
"agents":{
"defaults":{
"model":{
"primary":"bailian/qwen3.5-plus",
"fallbacks":["kimi-coding/k2p5"]
},
"maxConcurrent":40000,
"contextPruning":{
"mode":"cache-ttl",
"ttl":"24h"
}
}
}
}

修改完保存文件，然后验证配置：

openclaw doctor --fix

如果没有报错，重启 OpenClaw Gateway：

openclaw gateway restart

验证配置是否生效：

重启后，可以查看 Gateway 状态确认配置已加载：

openclaw gateway status

如果看到配置文件路径指向你修改的文件，说明配置已生效。

长期效果：

  • 会话不会因短期间断而丢失（这是核心收益）
  • 超过 24 小时的旧会话自动清理（防止无限增长）
  • 始终保留最近 50 个会话（够用）
  • 内存占用可控，不会像之前那样越跑越慢

我改完配置跑了两个月，没再遇到过会话突然重置的问题。内存占用也稳定，没出现过泄漏。

注意事项：什么情况下需要调整配置

我给的配置是通用方案，适合大多数重度用户。但你的情况可能不一样，需要微调。

适合当前配置的场景：

  • 每天多次与 Agent 交互（至少 2-3 次）
  • 需要跨天保留重要项目上下文
  • 单个会话上下文量较大（>10K tokens）
  • 设备内存充足（>16GB）

需要调整的场景：

1. 内存紧张

如果你的设备内存只有 8G 或更少，建议减少 maxSessions：

"maxSessions":25

这样最多保留 25 个会话，内存占用减半。代价是更早的会话会被清理。

2. 需要更长上下文

有些项目周期长，24 小时不够。比如你在做一个为期一周的调研，每天都得接着前一天的内容继续。

可以把 TTL 延长到 72 小时：

"ttlSeconds":259200

但记得同时降低 maxSessions，否则内存会爆。

3. 多用户共享

如果你的 OpenClaw 实例是多人共用的（比如团队部署），可以为不同用户设置不同配置：

"session":{
"profiles":{
"default":{
"idleTimeoutSeconds":-1,
"ttlSeconds":86400
},
"guest":{
"idleTimeoutSeconds":3600,
"ttlSeconds":7200
}
}
}

正式用户享受完整会话保留，访客会话 1 小时空闲就清理。

4. 完全禁用清理（不推荐）

有人可能会问，能不能彻底关闭会话清理？

可以，但不推荐：

"idleTimeoutSeconds":-1,
"ttlSeconds":-1,
"maxSessions":1000

这会导致内存持续增长，除非你设备内存无限。仅适合测试环境或短期调试。

总结

OpenClaw 的默认会话清理策略是为了平衡性能和资源占用，对大多数轻度用户来说是合理的。

但对于把 Agent 当生产力工具的重度用户，4 小时空闲重置确实太短了。

通过修改 openclaw.json，禁用空闲过期并设置合理的 TTL，你可以在保持上下文连续性的同时，避免资源无限消耗。

配置不是一成不变的。根据你的使用习惯和设备情况，随时调整这些参数，找到最适合自己的平衡点。

毕竟，工具是为人服务的，不是人为工具服务。

最后提醒一句：改配置前，建议备份原文件。万一改错了，还能恢复。

cp openclaw.json openclaw.json.bak
