OpenClaw + GitHub 实战：让 AI 自动处理 Issues、修 Bug、发 PR

作为开发者，我最烦的事情之一是处理 GitHub Issues。

不是说这件事不重要——恰恰相反，Issues 是项目健康的关键。烦的是那些"明明看起来很简单、但就是得腾出时间去看"的 bug。

有时候一个 issue 躺在那里好几周，不是因为难修，而是因为优先级一直排不上来。

我现在的做法是：让 OpenClaw 替我盯着这些 Issues，能修的直接帮我修掉，发 PR，等我审。

这是个什么功能？

OpenClaw 有一个内置技能叫 gh-issues。

它的工作逻辑很直接：

  1. 从你的 GitHub 仓库拉取 open issues
  2. 对每个 issue 起一个独立的 Sub-agent
  3. 每个 Sub-agent 自己分析代码、定位问题、写修复、跑测试
  4. 测试通过后自动 push 分支、开 PR
  5. 有 review comments 的话，它也会帮你回复、推送更新

你只需要在最后看 PR、做最终审核、合并。

前置准备

需要的东西不多：

  • 一个 GitHub 账号，仓库在里面
  • GitHub Personal Access Token（需要 repo 权限）
  • OpenClaw 已经装好

在 OpenClaw 配置里加上 token：

或者在 OpenClaw 界面的 Skills 配置里直接填。

基本用法

最简单的用法，直接发消息给 AI：

ounter(line/gh-issues owner/repo-name

AI 会拉取这个仓库最近的 open issues，列一张表让你确认，然后开始处理。

如果你想只处理带 bug 标签的：

ounter(line/gh-issues owner/repo-name --label bug

只看前 5 个：

ounter(line/gh-issues owner/repo-name --limit 5

处理前会展示一张确认表：

ounter(lineounter(lineounter(lineounter(line| # | 标题 | 标签 ||---|------|------|| 42 | Fix null pointer in parser | bug, critical || 37 | Add retry logic for API calls | enhancement |

确认后 AI 开始并行处理，每个 issue 一个独立 Sub-agent，同时跑。

AI 是怎么修 bug 的？

每个 Sub-agent 接到任务之后，它的工作流是这样的：

第一步：信心评估

AI 会先给自己打个分（1-10），评估它能不能修这个 issue：

  • 问题描述清楚吗？
  • 能在代码里找到对应的地方吗？
  • 修复范围合理吗？

如果评分低于 7，它不会盲目尝试，而是直接告诉你："这个 issue 我没把握，原因是 [...]"

这个机制很重要——它防止 AI 乱改代码。

第二步：分析和定位

用 grep/find 搜索相关代码，读取关键文件，理解当前行为和预期行为的差距。

第三步：实现修复

做最小改动。只改必要的地方，不重构不扩展，遵循现有代码风格。

第四步：跑测试

自动发现项目的测试配置（package.json、Makefile、pytest 等），跑测试，验证修复没破坏其他东西。

如果测试失败了，它会尝试修正一次，再次失败就报告给你，不会继续盲目折腾。

第五步：提交和发 PR

创建分支名为 fix/issue-42 的分支，写符合 Conventional Commits 格式的提交信息，push 到仓库，开 PR，PR 描述里自动包含 "Fixes #42"。

定时自动处理

如果你想让它每天自动跑，不需要每次手动触发，可以用 --cron 模式。

在 OpenClaw 的 heartbeat 或者 cron 配置里加上：

ounter(line/gh-issues owner/repo --label bug --cron

每次触发，它会从上次处理的 issue 继续往后，一次处理一个，火跑即走，不等结果。

多次触发之间有防重机制——它会记录哪些 issue 已经在处理中，不会重复派 Sub-agent。

这样你的 Issues 就有了一个"自动处理队列"：新 bug 进来，慢慢被消化，你只需要审 PR。

处理 Fork 项目

如果你在给别人的项目贡献代码，用 --fork 参数：

ounter(line/gh-issues upstream-owner/repo --fork your-username/repo --label "good first issue"

AI 会：

  • 从上游仓库拉 issues
  • 把修复代码 push 到你的 fork
  • PR 从你的 fork 开向上游

适合参与开源项目——找 good first issue，批量尝试贡献。

自动处理 PR review 意见

这是我觉得最实用的功能之一。

PR 开了之后，review 意见来了，通常要：

  1. 看 comment
  2. 找代码
  3. 改
  4. 推
  5. 回复 comment

每一步都不难，但加起来很费时间。

OpenClaw 的 --reviews-only 模式专门处理这种情况：

ounter(line/gh-issues owner/repo --reviews-only

它会扫描所有 fix/issue-* 分支的 PR，找到有未处理 review comment 的，分析哪些是真正需要操作的（不是自动 bot 的消息、不是已经回复过的），然后起 Sub-agent 去修改代码、推送、回复 comment。

能自动识别多种类型的 review 意见：

  • 直接的代码 review 评论
  • PR 正文里嵌入的 review（比如 Greptile 之类的工具）
  • 标记了 CHANGES_REQUESTED 的 review

结果通知

如果你想让 AI 处理完了直接发消息告诉你 PR 链接，配置一个通知频道：

ounter(line/gh-issues owner/repo --cron --notify-channel -1002xxxxxxxxx

一个真实场景

我前段时间用这个处理过一个小项目的积压 issues。

情况： 一个 Node.js 项目，9 个标了 bug 的 open issues，躺了 2-3 周没动。

操作：

ounter(line/gh-issues my-org/my-project --label bug --limit 9

AI 拉出 9 个 issues，我确认全部处理。

结果（大概 40 分钟后）：

  • 6 个：成功开了 PR，测试通过
  • 2 个：AI 说置信度不够（问题描述太模糊），附上了分析报告
  • 1 个：涉及整个子系统重构，AI 判断超出范围，没动

6 个 PR 我花了大概 1 小时审核，合并了 5 个（有 1 个 AI 的方向不对，我自己改了）。

2 个"置信度不够"的 issues 对我来说也有参考价值——AI 的分析帮我快速定位了问题所在，我自己修的时候快了不少。

使用注意

适合的场景：

  • 有明确复现步骤的 bug
  • 文档类 issues
  • 小功能增强、配置类改动

不太适合：

  • 需要大量上下文的架构讨论
  • 模糊描述的 feature request
  • 涉及多个系统交互的复杂 bug

AI 自己的置信度评分会过滤掉大多数"不适合"的情况。但你也应该对结果做基本审核，不要无脑合 PR。

关于测试：如果你的项目没有测试，AI 修的代码风险会高一些。有测试套件的项目用这个工具效果最好。

小结

gh-issues 功能改变了我处理技术债的方式。

以前 Issues 是一个压力来源——知道有问题，但没时间处理，越堆越多。

现在有了一个自动处理管道：新 issue 进来 → AI 尝试修 → 我审 PR。不能修的 AI 会说明原因，能修的我只需要审核最终代码。

这不是"AI 替代开发者"，而是"AI 处理掉重复性劳动，开发者专注判断和决策"。

代码最终该不该合，还是得你来决定。但从 issue 到 PR 的这段路，可以交给 AI 跑了。
