你的 .claude/ 文件夹正在变成垃圾堆吗？Claude Code 配置治理实战手册


大多数人用 Claude Code 都是这么开始的：项目根目录冒出个 .claude/ 文件夹，随手扔个 CLAUDE.md 进去，写几行项目介绍和常用命令，觉得够用了。

小项目确实能跑。但一旦代码库膨胀，这种随意配置就开始露馅，instructions 越写越长却找不到重点，scripts 散落各处命名混乱，团队成员想加新规则时完全不知道往哪塞。.claude/ 慢慢从"有用的配置"变成"看不懂的杂物间"，Claude 反而更难用了。

这篇文章聊聊怎么把 .claude/ 从垃圾堆变成真正的项目运营层：结构清晰、易于维护、能随项目一起长大。

核心原则：两层分离

先记住这个总框架：

层级
作用
典型文件
项目层
（repo 内）
团队共享的标准，保证一致性
CLAUDE.md
、.claude/rules/、.claude/settings.json
个人层
（全局或本地）
个人工作偏好，不污染团队配置
~/.claude/
、CLAUDE.local.md、.claude/settings.local.json

混用这两层是大多数结构问题的根源。在团队配置里塞进个人习惯，或者把该共享的标准放在个人目录，都会让设置逐渐失控。

顶层放什么：入口必须一眼找到

项目根目录和 .claude/ 顶层只放定义全局的文件：

your-project/
├── CLAUDE.md              # Claude 理解项目的入口
├── CLAUDE.local.md        # 个人覆盖（不提交）
└── .claude/
    ├── settings.json      # 控制层：权限、hooks、项目级行为
    ├── settings.local.json # 个人本地配置
    └── ...
  • • CLAUDE.md 讲"项目是什么、怎么组织的"，技术栈、架构决策、关键约定、常用命令
  • • settings.json 管"Claude 能做什么、不能做什么"，权限、自动化 hooks、触发行为

两个文件分工明确：一个给 guidance，一个给 control。别在 CLAUDE.md 里写"运行测试前要格式化"，那是 settings.json 里 hook 该干的活。

Instructions 瘦身：CLAUDE.md vs rules/

CLAUDE.md 最容易犯的错是什么都往里塞。项目初期还好，后期变成几十页的万能文档，Claude 抓重点反而更难。

更好的做法：CLAUDE.md 保持宽泛，细节拆进 rules/。

放 CLAUDE.md
放 rules/
技术栈和架构总览
前端/后端各自的编码规范
最常用的 3-5 个命令
测试策略、安全规则
跨仓库适用的约定
特定模块的详细说明

举个例子，一个清晰的项目这样组织：

项目根路径/
|-- CLAUDE.md
|--.claude/
    └── rules/
        ├── frontend.md      # Next.js 组件规范、状态管理约定
        ├── backend-api.md   # FastAPI 验证规则、分页要求
        ├── testing.md       # 测试写法、覆盖率标准
        └── data-pipelines.md # 批处理任务约定

更新时只改相关文件，不用翻几十页的 CLAUDE.md。团队里负责测试的人改 testing.md，不会碰乱主文档。

还可以更进一步：按路径限定规则。后端规则只作用于 app/api/，迁移规则只在改数据库时触发，减少 Claude 的上下文噪音。

行动层分离：hooks/ 和 commands/

Instructions 讲"该做什么"，action 文件管"实际怎么跑"。一定不要把这两类混在一起了。

hooks/：自动化的脚本

hooks/ 放 Claude workflow 特定节点自动执行的脚本：

典型场景
示例
拦截危险操作
block-dangerous-commands.sh
自动格式化
format-edits.sh
（用 ruff 处理 Python 文件）
质量门禁
run-tests-before-stop.sh

这些不是文档，是可执行的 operational scripts。即使只有三行，也该待在 hooks/ 里，让团队一眼知道这是什么类型的文件。

commands/：可复用的 prompt 工作流

commands/ 存你反复跑的任务模板，省得每次重写 prompt：

.claude/
└── commands/
    ├── review-pr.md          # PR 审查：正确性、边界 case、API 变更、测试覆盖
    ├── write-tests.md        # 补测试：给定函数生成测试用例
    ├── investigate-bug.md    # 排查 bug：从报告到根因分析
    └── prepare-release.md    # 发版准备：变更汇总、版本说明草稿

每个文件单一目的，命名直接体现用途。review-pr.md 清楚，misc.md 或 temp-review.md 就是坑队友。

一个实用的 command 例子

假设你的团队经常让 Claude 在合并前 review backend changes。不是每次写新的 prompt，而是创建：

# review-pr

Review the current changes with a focus on:
- correctness
- missing edge cases
- API contract changes
- test coverage gaps
Summarize:
1. critical issues
2. medium-risk issues
3. suggested improvements

然后把它保存到：

.claude/
└── commands/
    └── review-pr.md

这会让 workflow 更容易复用，也更容易随时间而改进。

进阶能力：skills/ 和 agents/（非必须）

这两个文件夹不是起点。简单项目用不上。如果是为了用而用，会画蛇添足，反而添乱。

什么时候加 skills/

Command 文件不够用时，workflow 有多个步骤、需要配套文档、值得长期标准化。一个 skill 是自包含的目录：

.claude/
└── skills/
    ├── api-review/
    │   ├── SKILL.md      # 技能定义：目的、触发条件、执行步骤
    │   └── checklist.md  # 配套模板：版本说明格式、检查清单
    ├── pre-release/
    │   ├── SKILL.md
    │   └── release-template.md
    └── docs-audit/
        ├── SKILL.md
        └── style-guide.md

对比 commands/ 的轻量任务，skills/ 打包的是有深度的完整工作流。

什么时候加 agents/

需要专门的角色 时，不是每次用通用 Claude，而是定义特定的审查员：

.claude/
└── agents/
    ├── code-reviewer.md      # 专注：正确性、风险假设、边界 case、测试覆盖
    ├── security-auditor.md   # 专注：注入风险、敏感数据暴露、权限问题
    └── docs-editor.md        # 专注：风格一致性、术语准确性、可读性

每个 agent 文件边界清晰，避免"helper.md"这种模糊命名。

关键纪律：这两个文件夹保持小而精。实验性文件、重叠的 persona、未完成的工作流，该删就删，别让 .claude/ 变成草稿箱。

个人配置的边界

全局个人目录 ~/.claude/ 存跨项目复用的偏好：

~/.claude/
├── CLAUDE.md                    # 个人写作风格、默认行为偏好
├── skills/
│   ├── article-outline/         # 技术文章大纲生成（自用）
│   └── sql-review/              # 个人 SQL 审查习惯
└── agents/
    └── documentation-editor.md  # 个人文档润色风格

项目内的 CLAUDE.local.md 和 settings.local.json 则是本地覆盖：想在特定仓库试个新 hook，又不想污染团队配置，就在这改。

实用蓝图：从小开始，按需生长

别一上来就创建所有的文件夹。应该是随着项目或需求的发展而发展，一个设计结构便于扩展，又足够简单让团队任何人都能快速理解的结构。

阶段 1：核心
├── CLAUDE.md
└── .claude/
    └── settings.json

阶段 2：拆分 instructions
└── rules/
    ├── backend.md
    └── testing.md

阶段 3：加自动化
└── hooks/
    └── format-edits.sh

阶段 4：可复用工作流
└── commands/
    └── review-pr.md

阶段 5：深度能力（确认需要时）
└── skills/
    └── release-prep/
└── agents/
    └── code-reviewer.md

每个阶段解决真实的问题，而非提前设计。下面是一个 SaaS 团队的成熟结构示例：

项目根目录/
├── CLAUDE.md
├── CLAUDE.local.md
└── .claude/
    ├── settings.json
    ├── settings.local.json
    ├── rules/
    │   ├── backend.md
    │   ├── frontend.md
    │   ├── testing.md
    │   └── security.md
    ├── hooks/
    │   ├── block-dangerous-commands.sh
    │   ├── format-edits.sh
    │   └── run-checks-before-stop.sh
    ├── commands/
    │   ├── review-pr.md
    │   ├── write-tests.md
    │   └── summarize-changes.md
    ├── skills/
    │   └── release-prep/
    │       ├── SKILL.md
    │       └── release-template.md
    └── agents/
        ├── code-reviewer.md
        └── security-auditor.md

写在最后

好的 .claude/ 结构不是关于复杂性，而是关于清晰性。当文件夹组织良好时，Claude 变得更易于引导、更值得信赖，也更容易在长期的项目中使用。当结构混乱时，即使是好的 instructions 也会失去价值，因为周围的系统变得更难维护。

最好的设置通常从小开始，保持明确的意图，只在明确需要时才增长。

最好的 .claude/ 文件夹不是拥有最多文件、最多文件夹或最先进设置的那个。而是每个部分都有清晰目标的那个。

这通常意味着保持核心简单，只在需要扩展时才拆分 instructions，将 automation 与 guidance 分开组织，并在团队配置和个人偏好之间做出清晰区分。它还意味着抵制在 workflow 实际需要之前添加额外结构的冲动。

当 .claude/ 组织良好时，团队之间的Claude 变得更容易合作。也更容易让团队共享。随着项目增长，文件夹也随之增长但不会变得混乱。

最终，最高效率来自清晰。结构越干净，越容易用好Claude。
