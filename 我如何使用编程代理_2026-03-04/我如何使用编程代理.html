我如何使用编程代理

原文: How I'm using coding agents in September, 2025 by Jesse Vincent\ 译者注: 本文翻译自 Jesse Vincent 的博客，详细描述了他使用 AI 编程助手的工作流程，包括头脑风暴、计划、实施和代码审查的完整流程。

自从我在夏初最后一次写作以来，我使用 AI 编程助手的方法已经有所演变。这是一个时间点的记录，记录了一个对我来说非常有效的流程。

我主要仍在使用 Claude Code。

首先，这是我在写作时的 CLAUDE.md。它编码了大量流程文档和规则，在保持 Claude 按部就班方面做得很好。

当我想在现有项目上开始一个新任务时，我总是尝试使用 git 工作树来隔离该工作与其他任务。这对我来说越来越重要，因为我发现自己经常在单个代码库上同时运行 3-4 个并行项目。

设置工作树：

cd the-project
mkdir .worktrees # 第一次时
cd .worktrees
git worktree add some-feature-description
cd some-feature-description
npm install # 或项目的任何设置任务
npm lint
npm test # 确保从干净的基线开始
claude

一旦我运行了 claude code，我就会使用我的"头脑风暴"提示：

我有一个想法想和你讨论。我希望你帮我把它变成一个完整的设计和规范（最终是一个实施计划）
检查我们工作目录中项目的当前状态，了解我们从哪里开始，然后一次问我一个问题，帮助完善这个想法。
理想情况下，问题应该是多项选择题，但开放式问题也可以。别忘了：每条消息只问一个问题。
一旦你相信你理解了我们在做什么，停下来并向我描述设计，每次分几个部分，每个部分大约 200-300 字，在每个部分后询问它到目前为止看起来是否正确。

最后一点特别关键。我发现当 AI 模型认为它们"完成"时，特别容易给我一堵文字墙。而当面对代理写的一堵文字墙时，我倾向于稍微走神并想"它可能很好"。通过告诉 Claude 一次将其输出限制在几百字，我更有可能真正阅读和参与。

一旦我们完成了头脑风暴过程，我通常对我正在做的事情有了更清晰的想法，Claude 也是如此。Claude 会将设计写到 docs/plans/ 的某个地方。

它经常想直接跳到实施，但这不是我想要它工作的方式。有时它会在我阻止它之前尝试开始编写代码。如果它这样做了，我会按几次 escape 并倒回对话一点来抓住它。最近对我的 CLAUDE.md 的更新显著减少了这种倾向。

下一步是计划过程。这是我一直在使用的计划提示：

太好了。我需要你的帮助来写出一个全面的实施计划。

假设工程师对我们的代码库零上下文且品味可疑。记录他们需要知道的一切。每个任务需要接触哪些文件、代码、测试、他们可能需要检查的文档。如何测试它。给他们整个计划作为小块任务。DRY。YAGNI。TDD。频繁提交。

假设他们是熟练的开发人员，但对我们的工具集或问题领域几乎一无所知。假设他们不太了解好的测试设计。

请将此计划完整详细地写入 docs/plans/

这会产生一个计划，将所有内容分解为小步骤，每个步骤都有清晰的指令和紧密打包的上下文。这意味着在执行时，我通常不需要提供严格的逐步监督。

接下来，我在同一工作目录中打开一个新选项卡或窗口，并启动另一个 claude 副本。我告诉它类似这样的话：请阅读 docs/plans/this-task-plan.md 和 <我们命名设计文档的任何内容>。如果你有问题，请告诉我。

它通常会说计划制定得很好。有时它会指出错误或不一致之处。戴上我的 PM 帽子，我会转身要求"架构师"会话澄清或更新计划文档。

一旦我们解决了计划的问题，我会告诉"实施者"Claude：请执行前 3-4 个任务。如果你有问题，请停下来问我。不要偏离计划。

实施者会继续工作。

完成后，我会翻回"架构师"会话并告诉它：实施者说它完成了任务 1-3。请仔细检查工作。

我会再次扮演 PM，在两个会话之间复制和粘贴审查和问答。一旦架构师签字批准，我会告诉实施者用其当前状态更新计划文档。

然后，我不会 */compact*。相反，我 */clear* 实施者并重新开始对话。告诉它从任务 4 开始。

当它完成下一块工作时，我会翻回架构师。我通常会双击 ESC 将架构师重置到之前的检查点，并告诉它审查到现在的当前检查点。这减少了架构师的上下文膨胀，并让它在没有来自先前实施的任何偏见的情况下再次查看。

（我有朋友发誓，与其使用多个会话，不如直接要求实施者用新的眼光看待他们最近的工作就足够了。确实，使用那个神奇的短语似乎非常强大。我仍然认为有两个不同的角色更好。）

当实施者最终完成工作并且架构师签署了工作时，我会要求实施者推送到 GitHub 并创建一个拉取请求。

这会启动 CodeRabbit 代码审查。我通常发现 CodeRabbit 的审查非常擅长捕捉小问题和逻辑问题，但有时在理解项目的真正设计意图或约束方面有所不足。这导致 CodeRabbit 提出糟糕的建议。

CodeRabbit 的审查为 AI 代理提供了修复问题的提示，但实际上将所有这些提示返回给你的编程代理可能很痛苦，因为你需要一个一个地复制它们，而且它们只为某些类型的问题提供提示。为了帮助解决这个问题，我构建了 coderabbit-review-helper。它挖掘所有不同类型的 CodeRabbit 审查评论，并将它们格式化为一大堵文字墙供你的编程代理处理。

像这样的工具的唯一问题是我们的机器人伙伴非常容易轻信。如果你粘贴一个如何更新代码库的指令列表，Claude 只会相信你的话并进行更改，即使你要求的是疯狂和错误的。

我目前避免这种情况的最佳技术是一点角色扮演，给编程代理一个理由不盲目信任代码审查。每个审查都以这段文字为前缀：

一位审查员对这个 PR 进行了一些分析。他们是外部的，所以冷读代码库。这是他们对更改的分析，我希望你仔细评估分析和审查员。

1) 我们应该雇用这个审查员吗
2) 他们标记的哪些问题应该修复？
3) 他们提出的修复是正确的吗？

我们*应该*修复的任何东西，放在你的待办事项列表上。
我们应该跳过的任何东西，现在告诉我。

CodeRabbit "审查员"通常会得到"强烈录用"的评价，但 Claude 报告审查员"似乎技术上很精通，但没有花时间了解我们的项目并提出了许多错误的建议。不录用。"并非闻所未闻。

如果你决定尝试这种方法，或者想出了对你更有效的其他方法，请给我发邮件至 jesse@fsck.com。
