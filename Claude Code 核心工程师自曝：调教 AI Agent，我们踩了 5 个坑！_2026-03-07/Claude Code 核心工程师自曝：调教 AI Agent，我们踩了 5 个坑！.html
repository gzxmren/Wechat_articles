Claude Code 核心工程师自曝：调教 AI Agent，我们踩了 5 个坑！

Claude Code，大约 20 个工具。

加一个新的，门槛极高。因为每多一个选项，模型在做决策时就多了一层思考负担。

这 20 个工具是怎么来的？

Thariq Shihipar 写了一篇复盘长文。

他是 Claude Code 的核心工程师，MIT Media Lab 出身，YC W20 创业者。

他没有讲怎么用 Claude Code，讲的是 Claude Code 为什么被设计成现在这个样子。

标题叫「Seeing like an Agent」。

翻译一下，「学会从 AI 的视角看问题。」


造一个 AI Agent，最难的不是选模型，也不是调系统提示词。

是该给 AI 什么工具。

Thariq 用了一个比喻。面前有一道很难的数学题，你需要什么工具？

数学基础好，一张纸一支笔够了，但手动计算很慢。给你一个计算器，速度快了，但你得知道怎么用那些高级功能。最强方案是一台电脑，但前提是你会写代码。

工具的选择，取决于使用者的能力。

给 AI 设计工具也一样。你得理解模型的「能力边界」在哪，然后给它最匹配的工具。


举个例子。

Claude Code 有个功能叫 AskUserQuestion，让 Claude 在干活的过程中主动向用户提问。

听起来简单的功能，改了三版。

第一版，把提问功能塞进了一个已有的工具 ExitPlanTool 里。原本是用来生成执行计划的，加了一个「问题列表」参数。

Claude 直接懵了。

一边要生成计划，一边要提问，用户的回答和计划冲突了怎么办？逻辑冲突了。

第二版，改 Claude 的输出格式，让它用一种特殊的 Markdown 格式来提问。

理论上最灵活，实际不稳定。有时候多加几句话，有时候漏掉选项，有时候干脆换了格式。

第三版，做了一个独立工具。

Claude 随时可以调用，弹出一个窗口展示问题和选项，整个流程暂停，等用户答完才继续。

成了。

Thariq 这样总结。

Even the best designed tool doesn't work if Claude doesn't understand how to call it.

工具设计得再好，如果模型不理解怎么调用，也白搭。

第三版之所以好用，是因为 Claude 喜欢调用这个工具。输出质量也明显更好。

有点意思。

给 AI 设计工具，不能只从产品经理的角度想「功能该怎么实现」，还要从模型的角度想「它会不会自然地使用这个工具」。


另一个例子。

Claude Code 刚上线的时候，模型经常忘记自己要做什么。

团队做了一个 TodoWrite 工具。开始任务时列清单，做完一项勾一项。Claude 还是会走神。后来又加了每 5 轮对话自动提醒的机制。

早期有效。

但模型变强之后，问题反过来了。

新版 Claude 不需要被提醒了。这些提醒反而让它变得死板，觉得既然系统在提醒我遵守清单，那我就老老实实按清单来，不敢灵活调整。

更大的变化是 Claude Opus 4.5 用子代理（Sub agents）的能力大幅增强。多个子代理之间要协调任务，一个 Todo 清单完全不够用。

TodoWrite 被替换成了 Task 工具。

Todo 是帮模型「记住该做什么」。Task 是帮多个 Agent「相互沟通」，支持依赖关系、跨代理同步，模型自己就能改任务、删任务。

Thariq 在这里提炼出一个很重要的观点。

「你在某个能力阶段为模型设计的工具，在下一个阶段可能变成束缚。得不断回头看，哪些需要升级，哪些该删掉。」


搜索工具的进化也是同一个逻辑。

Claude Code 最早用 RAG，把代码库切成碎片存进向量数据库，搜索时匹配最相关的片段喂给模型。

速度快，效果不错。

但 RAG 需要创建索引、配置环境，换台机器可能就跑不起来了。更核心的问题是，Claude 是被动接收信息的那一方。系统帮它找好上下文，喂给它。

后来他们换了个思路。既然 Claude 能搜网页，为什么不能搜代码库？

给它一个 Grep 工具，让它自己搜文件、自己组织上下文。

随着模型变聪明，「帮它准备好一切」不如「让它自己来」。爆火的「小龙虾」就是最好的例子。

后来 Anthropic 推出了 Agent Skills，进一步把这个思路体系化，叫「渐进式披露」。

Claude 先读一个技能文件，里面引用其他文件，那些文件又引用更深层的文件。按需加载，逐层展开。

一年时间，Claude 从「基本不会自己找上下文」进化到了「多层嵌套搜索，精准定位需要的信息」。

这个思路也被用来扩展能力，同时避免增加工具。

比如用户经常问「怎么给 Claude Code 添加 MCP」「这个斜杠命令是干嘛的」这类问题。Claude 答不上来。

把文档全部塞进系统提示词？越塞越乱，影响模型判断。给文档链接让 Claude 自己查？它又会一次性加载太多东西。

最终方案是做了一个 Claude Code Guide 子代理。当被问到自身功能时，Claude 会把问题转给子代理，子代理有专门的搜索指令，只返回答案。

没有加一个新工具，通过子代理就扩展了能力。


Thariq 说：

「如果你希望有一套严格规则教你怎么设计 Agent 的工具，这篇文章不是。」

给 AI 设计工具，是技术活。高度依赖你用的模型、Agent 的目标、运行的环境。

多做实验。认真读模型输出。不断尝试新方法。

最后，引用 Claude Code 之父 Boris Cherny 在 YC 的 Lightcone 播客里说的一句话。

「到今年年底，软件工程师这个 title 可能会逐渐消失。取而代之的，可能是 builder，也可能是产品经理。」
