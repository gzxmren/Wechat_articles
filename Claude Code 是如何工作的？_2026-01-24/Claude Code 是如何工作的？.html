Claude Code 是如何工作的？

 

Claude 官方文档整理非常完善，可惜这篇关键的导读篇没有中文版，今天整理分享出来。本文帮助你理解 Claude Code 智能体循环、内部构建以及与项目交互的核心概念，可以作为我后文章 《Claude Code 从入门到精通》的导读篇。

Claude Code 是一个运行在终端上的智能助手，它精通编程，可以在命令行做任何事情：写文档、运行构建产物、搜索文件、研究课题等等。

本指南涵盖了核心架构、内置功能以及有效工作的技巧。有关逐步演练，请参阅 常用工作流程 。有关可扩展性功能（如技能、MCP 和 Hook），请参阅 Claude Code 扩展 。

1. 智能体循环

当你给 Claude 下发一个任务后，它会经历三个阶段：收集上下文、采取行动和验证结果。这些阶段相互融合。Claude 整个过程都在使用工具，无论是搜索文件理解代码，编辑文件，还是运行测试验证结果。

这个循环会根据你的要求进行调整。关于你代码库的问题可能只需要收集上下文，但修复 bug 可能就要反复循环所有三个阶段，而重构则可能会涉及更多的验证。Claude 会根据前一步骤中学到的内容来决定后续步骤需要什么，将数多个操作串联在一起，并在此过程中随时纠正方向。

你也是这个循环的一部分。你可以在任何时候中断，引导 Claude 朝不同的方向发展，提供额外的上下文，或要求它尝试不同的方法。Claude 自主工作，但对你的输入一直保持响应。

智能体循环由两个部分驱动：推理模型和执行工具。Claude Code 充当 Claude 的代理框架：它提供工具、上下文管理和执行环境，将语言模型转变为有能力的编码代理。

模型

Claude Code 使用 Claude 模型来理解你的代码并推理任务。Claude 可以阅读任何语言的代码，了解组件如何连接，并确定需要进行哪些更改才能实现你的目标。对于复杂的任务，它会将工作分解为多个步骤，执行这些步骤过程中，会根据所学到的知识进行调整。

有多种模型可供选择，各有优缺点。Sonnet 可以很好地处理大多数编码任务，Opus 为复杂的架构决策提供更强的推理能力。在会话期间使用 /model 切换，或以 claude --model 启动。 本文档提到的“Claude 选择”或“Claude 决定”时，是指模型在进行推理。

工具

工具是使 Claude Code 具有代理能力的原因。如果没有工具，Claude 只能用文本进行回复。有了工具，Claude 就可以采取行动：读取你的代码、编辑文件、运行命令、搜索网络以及与外部服务交互。每次使用工具都会返回信息，这些信息会反馈到循环中，从而为 Claude 的下一个决策提供信息。

内置工具通常分为四类，每类代表一种不同的代理能力：

分类
Claude 可以做什么
File operations
读取文件、编辑代码、创建新文件、重命名和重新组织
Search
按模式查找文件、使用正则表达式搜索内容、浏览代码库
Execution
运行 shell 命令、启动服务器、运行测试、使用 git
Web
搜索网络、获取文档、查找错误消息

这些只是主要的功能，Claude 还有用于生成子代理、向你提问和其他编排任务的工具。有关完整列表，请参阅 Claude 提供的工具。

Claude 会根据你的提示以及在此过程中学到的东西来选择使用哪些工具。当你说“修复失败的测试”时，Claude 可能会：

  1. 1. 运行测试套件以查看哪些测试失败
  2. 2. 读取错误输出
  3. 3. 搜索相关的源文件
  4. 4. 读取这些文件以理解代码
  5. 5. 编辑文件以修复问题
  6. 6. 再次运行测试以验证

每次使用工具都会为 Claude 提供新的信息，从而为下一步提供依据，这就是智能体循环的运作方式。

扩展基础能力：除了基础的内置工具以外，你还可以使用技能扩展 Claude 的知识。使用 MCP 连接到外部服务，使用 hooks 自动化工作流程，以及将任务转移给 Sub-Agent 。这些扩展在核心智能体循环之上形成一个层。根据需求选择正确的扩展指导，具体请参阅 Claude Code 高级技能。

2. Claude 可以访问哪些内容

本文侧重于终端操作，其实 Claude Code 也可以在 VS Code、JetBrains IDE 和其他环境中运行。

当你在目录中运行 Claude 时，Claude Code 可以访问：

  • • 项目：目录和子目录中的文件，以及你允许的其他位置的文件。
  • • 终端：你可以运行的任何命令：构建工具、git、包管理器、系统实用程序、脚本。可以通过命令行执行的命令，Claude 都可以。
  • • git 状态：当前分支、未提交的更改和最近的提交历史。
  • • CLAUDE.md ：一个 markdown 文件，你可以在其中存储项目特定的说明、约定和 Claude 每次会话都应了解的上下文。
  • • 扩展程序：用于外部服务的 MCP 服务器、用于工作流程的 skill 、用于委派工作的 subagent ，以及用于浏览器交互的 Chrome 中的 Claude。

因为 Claude 可以看到你的整个项目，所以它可以跨项目工作。当你要求 Claude “修复身份验证错误”时，它会搜索相关文件，读取多个文件以了解上下文，对它们进行协调编辑，运行测试以验证修复，并在你要求时提交更改。这与仅查看当前文件的内联代码助手不同。

3. 使用会话

Claude Code 会在你工作时将你的对话本地保存。每个消息、工具使用情况和结果都会被存储，因此你可以会退、恢复和派生会话。在 Claude 进行代码更改之前，它还会对受影响的文件进行快照，以便你在需要时可以恢复。

会话是临时的。 与 claude.ai 不同，Claude Code 在会话之间没有持久的记忆。每个新会话都是一个全新的开始。Claude 不会随着时间的推移“学习”你的偏好，也不会记住你上周所做的事情。如果你希望 Claude 了解跨会话的内容，请将其放入你的 CLAUDE.md 中。

3.1. 跨分支工作

每个 Claude Code 对话都是一个会话，与你当前的目录相关联。当你恢复时，你只会看到来自该目录的会话。

Claude 可以看到你当前分支的文件。当你切换分支时，Claude 会看到新分支的文件，但你的对话历史记录保持不变。即使在切换后，Claude 仍然记得你讨论过的内容。

由于会话与目录相关联，你可以通过使用 git worktree 运行并行 Claude 会话，git worktree 为各个分支创建单独的目录。

3.2. 恢复或派生会话

当你使用 claude --continue 或 claude --resume 恢复会话时，你将使用相同的会话 ID 从上次中断的地方继续。新消息会附加到现有对话中。你的完整对话历史记录会被恢复，但会话范围的权限不会，你需要重新批准这些权限。

要分支并尝试不同的方法而不影响原始会话，请使用 --fork-session 标志：

claude --continue --fork-session

这将创建一个新的会话 ID，同时保留到那时为止的对话历史记录，原始会话保持不变。与恢复会话一样，派生的会话不会继承会话范围的权限。

在多个终端中使用同一个会话：如果你在多个终端中恢复同一个会话，两个终端都会写入同一个会话文件。来自两者的消息会被交错，就像两个人在同一个笔记本上书写一样。没有任何东西会被损坏，但对话会变得混乱。每个终端在会话期间只能看到自己的消息，但如果你稍后恢复该会话，你将看到所有内容都交错在一起。对于从同一个起点开始的并行工作，请使用 --fork-session 为每个终端提供其自己的干净会话。

上下文窗口

Claude 的上下文窗口包含你的对话历史记录、文件内容、命令输出、CLAUDE.md、加载的技能和系统指令。随着你的工作，上下文会逐渐填满。Claude 会自动压缩，但对话早期的一些指令可能会丢失。将持久性规则放入 CLAUDE.md 中，并运行 /context 以查看哪些内容占用了空间。

当上下文填满时

当你接近限制时，Claude Code 会自动管理上下文。它会首先清除较旧的工具输出，然后在需要时总结对话。你的请求和关键代码段会被保留，对话早期的一些详细指令可能会丢失。将持久性规则放入 CLAUDE.md 中，而不是依赖于对话历史记录。

要控制在压缩期间保留哪些内容，请将 “Compact Instructions” 部分添加到 CLAUDE.md 中，或使用焦点运行 /compact（例如 /compact focus on the API changes）。

运行 /context 查看哪些内容占用了空间。MCP 服务器将工具定义添加到每个请求，因此几个服务器可能会在你开始工作之前消耗大量上下文。运行 /mcp 检查每个服务器的成本。

使用技能和子代理管理上下文

除了压缩之外，你还可以使用其他功能来控制加载到上下文中的内容。

技能按需加载。Claude 会在会话开始时看到技能描述，但只有在使用技能时才会加载完整的内容。 对于你手动调用的技能，请设置 disable-model-invocation: true 以使描述在你需要之前不会出现在上下文中。

子代理获得自己的全新上下文，与你的主要对话完全分离。它们的工作不会使你的上下文膨胀。完成后，它们会返回 摘要 ，这种隔离是子代理有助于处理长时间会话的原因 。

请参阅 上下文成本 文档以了解每个功能的成本，并 减少令牌使用量 以获取有关管理上下文的提示。

4. 使用检查点和权限保障安全

Claude 具有两种安全机制：检查点 允许你撤消文件更改，而 权限 控制 Claude 在未经请求的情况下可以执行的操作。

4.1. 使用检查点撤消更改

每次文件编辑都是可逆的。在 Claude 编辑任何文件之前，它会快照当前内容。如果出现问题，请按 Esc 键两次以回滚到之前的状态，或让 Claude 撤消。

检查点是你会话本地的，与 git 分离，它们仅涵盖文件更改。影响远程系统（数据库、API、部署）的操作无法设置检查点，这就是为什么 Claude 在运行具有外部副作用的命令之前会询问的原因。

4.2. 控制 Claude 可以执行的操作

按 Shift+Tab 键可在权限模式之间切换：

  • • Default（默认）：Claude 在文件编辑和 shell 命令之前会询问
  • • Auto-accept edits（自动接受编辑）：Claude 编辑文件无需询问，但仍会询问命令
  • • Plan mode（计划模式）：Claude 仅使用只读工具，创建你可以在执行前批准的计划

你还可以在 .claude/settings.json 中允许特定命令，这样 Claude 就不会每次都询问。这对于受信任的命令（如 npm test 或 git status）非常有用。设置可以从组织范围的策略到个人偏好进行配置。有关详细信息，请参阅 权限 。

5. 高效使用 Claude Code

下面这些技巧可帮助你从 Claude Code 获得更好的结果。

5.1. 向 Claude Code 寻求帮助

Claude Code 可以教你如何使用它。提出诸如“如何设置 hook？”或“构建我的 CLAUDE.md 的最佳方法是什么？”之类的问题，Claude 会解释。

内置命令还可以指导你完成设置：

  • • /init 引导你为你的项目创建 CLAUDE.md
  • • /agents 帮助你配置自定义子代理
  • • /doctor 诊断你的安装中的常见问题

5.2. 这是一个对话

Claude Code 是会话式的，你不需要完美的提示。从你想要的内容开始，然后进行改进：

> Fix the login bug

[Claude investigates, tries something]

> That's not quite right. The issue is in the session handling.

[Claude adjusts approach]

当第一次尝试不正确时，你不必重新开始，你需要迭代。

5.3. 中断和引导

你可以随时中断 Claude。如果它走错了方向，只需输入你的更正并按 Enter 键。Claude 将停止它正在做的事情，并根据你的输入调整其方法，不必等待它完成或重新开始。

5.4. 提前具体说明

你的初始提示越精确，你需要的更正就越少。引用特定文件，提及约束，并指向示例模式：

> The checkout flow is broken for users with expired cards.
> Check src/payments/ for the issue, especially token refresh.
> Write a failing test first, then fix it.

像“修复登录错误”这样模糊的提示虽然有效，但你将花费更多时间进行引导。像上面这样的特定提示通常在第一次尝试后就会成功。

5.5. 给 Claude 提供一些可以验证的内容

当 Claude 可以检查自己的工作时，它的表现会更好。包括测试用例，粘贴预期 UI 的屏幕截图，或定义你想要的输出。

> Implement validateEmail. Test cases: 'user@example.com' → true,
> 'invalid' → false, 'user@.com' → false. Run the tests after.

之后运行测试。 对于视觉工作，粘贴设计的屏幕截图，并要求 Claude 将其实现与屏幕截图进行比较。

5.6. 在实施之前进行探索

对于复杂的问题，将研究与编码分开。使用计划模式（按 Shift+Tab 键两次）首先分析代码库：

> Read src/auth/ and understand how we handle sessions.
> Then create a plan for adding OAuth support.

审查计划，通过对话改进它，然后让 Claude 实现。这种两阶段的方法比直接跳到代码产生更好的结果。

5.7. 委托不要命令

考虑委托给一位有能力的同事。提供上下文和方向，然后信任 Claude 来确定细节：

> The checkout flow is broken for users with expired cards.
> The relevant code is in src/payments/. Can you investigate and fix it?

你无需指定要读取的文件或要运行的命令， Claude 会自己解决这个问题。
