三个文件夹 + 一个 AI = 你的第二大脑

前 OpenAI 创始成员、前 Tesla AI 总监 Andrej Karpathy，前两天在 X 上发了一条推文。

他说自己最近发现了一件非常有用的事情："用 LLM 为各种研究主题建立个人知识库。我最近的大部分 token 使用量都花在了处理内容上，而不是写代码或改代码。"

两天时间，这条帖子被 44K+ 人收藏，全网千万人围观。

💡 核心思路极其简单：不要把笔记散落在各种 App 里——全扔进一个文件夹，然后让 AI 把它们整理成个人维基。

更关键的是：Karpathy 公开了完整实现方案，而且不需要数据库、不需要复杂工具、不需要任何编程基础。只需要文件夹、文本文件和一个能读文件的 AI 工具（Claude Code / Cursor 都行）。

下面是完整的保姆级搭建教程，跟着做就行。

第一步：建好三个文件夹，两分钟搞定

在你的电脑上任意位置创建一个项目文件夹，在里面建三个子目录：

my-knowledge-base/
├── raw/ # 原始素材收纳箱
├── wiki/ # AI 帮你整理好的维基
└── outputs/ # 存放问答输出

📌 三层架构说明

raw/ —— 你的原始素材箱。文章、论文、笔记、截图、会议记录……什么都往里扔，不用整理。

wiki/ —— AI 的主战场。它会把 raw/ 里的内容自动编译成结构化的 Markdown 维基，按主题分类，带摘要和交叉引用。

outputs/ —— 你向知识库提问后，AI 生成的答案存这里。优质答案可以再喂回 wiki/ 形成正循环。

就这三层。没有数据库，没有向量索引，没有插件。Karpathy 自己说的："超级简单，完全扁平，只是一个嵌套的 .md 文件目录。"

📁 文件夹结构 🚀 零门槛 📄 纯文本驱动

第二步：别整理，什么都往 raw 里扔

大多数人在这一步就卡住了——创建了空文件夹，然后盯着它发呆，不知道该放什么。

答案是：什么都往里扔。

  • 看到好文章？复制粘贴成 .md 或 .txt 扔进去

  • 有图表和截图？直接保存图片文件

  • 从 Notion / Obsidian / 飞书导出笔记？扔进去

  • 会议记录、研究论文、项目文档、囤了几个月的书签？统统扔

⚡ 关键原则

别整理、别重命名、别分类。

这些活儿全是 AI 干的。你唯一的工作就是当个"搬运工"，把素材丢进 raw/ 就完事了。

📝 博主实测：做 X 内容时攒了 17 个原始文件——剪藏文章、竞品分析、数据报告，没有一个手工整理的。

第三步：让 AI 自动抓网页存进来

手动复制粘贴太慢了？Karpathy 没提到但真正加速这个过程的部分来了——自动化收集。

Vercel Labs 发布了一个叫 agent-browser 的开源命令行工具（GitHub 上 26K+ Star），让你的 AI 直接操控 Chrome 浏览器抓取任意网页：

# 安装
npx agent-browser init
# 让 AI 打开并抓取指定页面
agent-browser open https://some-article-you-want.com

AI 会自动打开页面、提取文章文本，你直接导入到 raw/ 文件夹即可。不需要浏览器扩展，不需要手动复制粘贴。

🤖 agent-browser 能处理哪些场景？

✅ JavaScript 动态加载的内容
✅ 需要登录才能看的页面
✅ 带交互式图表的研究论文
✅ 需要不停滚动点"加载更多"的长页面
✅ 在菜单里翻来翻去才能找到完整内容的站点

💡 效率对比

agent-browser 比 Playwright MCP 省 82% 的 token。同样一轮对话里，你的 AI 能抓取 5-6 倍的页面数量。用在知识库上就是：看到一个好文章，跟 AI 说一句"把这个 URL 抓下来存到 raw/"，搞定。

🌐 自动抓取 📉 省 Token ⭐ 26K+ Star

第四步：给 AI 写一份说明书（CLAUDE.md）

这是大多数人会跳过的一步——千万别跳。

在项目根目录创建一个叫 CLAUDE.md 的文件（名字不重要，叫 AGENTS.md 也行）。这个文件告诉 AI：你的知识库是什么、怎么组织、规则是什么。

这是一个可以拿来即用的起始模板：

# 知识库 Schema
## 组织规则
- 所有内容以 Markdown 格式保存在 wiki/ 目录
- 每个主题一个独立文件，文件名清晰描述主题
- 每个文件开头必须有一段摘要（3-5 句话）
- 内部链接到所有相关主题
- 引用 raw/ 中的原始材料作为来源
## 输出规范
- 用简洁的语言写摘要
- 对矛盾观点要标注出来
- 未解释的概念单独列出
- 保持语气中立客观
💡 为什么这个文件很重要？

它相当于给 AI 写了一份"知识库专属操作手册"。没有它，AI 不知道你要什么风格、什么粒度、什么格式。有了它，每次生成的维基质量都会稳定在你期望的水平上。

📋 规则配置 🎯 质量控制

第五步：一条指令，AI 把笔记编成维基

现在打开 Claude Code 或 Cursor，指向你的项目文件夹，然后输入：

读取 raw/ 中的所有内容。然后按照 CLAUDE.md
中的规则，在 wiki/ 中编译一个维基。
要求：
1. 先创建 INDEX.md 索引文件
2. 为每个主要主题创建一个 .md 文件
3. 链接所有相关主题
4. 总结每个来源的核心观点
5. 标注出有矛盾的地方

然后走开，让它跑。

跑完后你会得到一个 wiki/ 文件夹——文章按主题分好了类、相关内容互相链接了、每个主题都有摘要、还有一份总索引。想查什么几秒就能定位。

🔑 核心原则：不要手动编辑 Wiki，那是 AI 的工作。你只负责阅读、提问、策展。维护和更新全部交给 AI。

第六步：开始提问，打造「活」的知识库

等维基有了 10 篇以上的内容之后，就可以开始真正的互动了：

基于 wiki/ 中的所有内容，
我对 [某个主题] 理解中最大的三个空白是什么？

AI 会读取整个维基，根据<strong你自己攒的材料< strong="" style="margin: 0px; padding: 0px; box-sizing: border-box;">给你分析答案。这不是泛泛而谈的 AI 回答，而是基于你的阅读积累给出的个性化洞察。</strong你自己攒的材料<>

🔄 正循环飞轮

① 你向知识库提问 → AI 基于你的资料给出答案
→ ② 把优质答案保存回 outputs/ 或直接更新 wiki/
→ ③ 下次提问时 AI 有更多上下文，回答更好
→ ④ 循环往复，知识库越用越厚、越用越准

这就是 Karpathy 说的"知识复利"。每一次使用都在为下一次打基础。

🔄 正循环 📈 知识复利 🧠 主动进化

第七步：定期体检，不让错误「滚雪球」

有人问了一个关键问题："如果 AI 写错了怎么办？"

Karpathy 帖子下的高赞回复一针见血："当输出被归档回去时，错误也会复利。"

⚠️ 错误复利风险

如果 AI 写了一些稍有偏差的东西而你保存了它，下一个回答就会基于这个错误继续推导，越偏越远。所以必须定期做"健康检查"。

解决方法很简单——定期跑一次审查指令：

审查整个 wiki/ 目录：
1. 标记所有文章之间的矛盾之处
2. 找出提到但从未详细解释的主题
3. 列出没有任何 raw/ 来源支持的声明
4. 建议 3 个需要补充的新方向

把检查结果作为下一步行动指南，该修正的修正，该补充的补充。就像代码需要 code review 一样，知识库也需要定期 audit。

🏥 健康巡检 🛡️ 错误防控

为什么这比传统 RAG 更好用？

你可能听过 RAG（检索增强生成）——每次提问时从文档库里检索相关片段，然后拼凑回答。LLM Wiki 和它的核心区别在于：

对比维度LLM Wiki（Karpathy 方案）传统 RAG
知识模式
主动编译、持续累积
被动检索、每次从零拼凑
谁来维护
LLM 全职维护（自动更新+标注矛盾）
无维护，每次临时检索
回答去哪了
优质回答可归档回 Wiki，成为新页面
消失在对话历史中
需要什么
文件夹 + 文本 + 一个 AI
向量数据库 + Embedding 模型 + 检索管线
可控性
通过 CLAUDE.md 白盒自定义一切
固定流程，难以干预

一句话总结：RAG 是"每次从头想"，LLM Wiki 是"越想越聪明"。

关于工具选择：简单胜过一切

Karpathy 帖子下有一半人在安利各种 Obsidian 插件、Notion 模板、双链笔记工具……

但当别人问他到底用什么工具时，他的回答很干脆：

🗣️ "我试图保持超级简单和扁平。它只是一个嵌套的 .md 文件目录。"

AI 不在乎你用什么软件打开文件。 VS Code 行、Obsidian 行、记事本也行、甚至命令行的 cat 都行。重要的是文件夹结构和那个 Schema 文件。

⚠️ 别掉进"工具陷阱"

装了 47 个 Obsidian 插件的笔记系统，又是一个 Notion 陷阱——你花在配置工具上的时间，可能比真正用知识库的时间还多。扁平文件 + 好的 Schema，90% 场景下比花哨的工具栈管用得多。

🔧 极简主义 🚫 反工具崇拜
