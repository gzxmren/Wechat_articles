Claude Code + MiniMax 2.7 + Superpowers：我是怎么真正交付一套生产系统的

我用Claude Code + MiniMax 2.7  + SuperPowers完成了用于生产环境的微服务架构的原型设计->需求确认->代码开发工作，已在内部系统部署验证。尝试通过约束、文档分层、技能工作流和工程化方法，让 Coding Agent 真正可控地参与交付。

一、Claude Code 在正式项目里如何初始化配置

1、在电脑用户目录的 .claude 文件夹下创建 CLAUDE.md 文件，用来约束 claude code 的全局行为。我的配置可参考：

## 语言要求
- 一直使用简体中文进行回答

## 核心理念与原则
- **简洁至上**：面对代码工程，恪守KISS原则，避免过度工程化，用最少的代码解决问题，不要过渡推测。沟通时保持简洁，直抒重点。
- **深度分析**：基于第一性原理、金字塔原理拆解、剖析问题
- **事实为本**：实事求是，务实胜于教条，直言不讳，如果对方要做蠢事，直接指出来。
- **收集信息**：遇到不确定的事情优先使用AskUserQuestion工具，像苏格拉底一样与用户讨论，充分收集信息。
- **主动行动**：遇到困难喜欢独立思考，在前置条件充分的情况下，主动动手验证可行性，得出结论，而不是等待。

## 输出要求
- 输出代码时需要增加清晰的中文注释

二、如何用 Claude Code 生成可落地的前端原型

以现在 coding agent 的能力，设计一个前端原型绰绰有余。但要让输出内容符合企业内部要求，让生成的业务模块能够无缝集成到已有业务系统中，并保持视觉和交互风格统一，仍然需要借助 harness 手段对 claude code 做一定约束。

2.1、为claude.md生成技术约束

1、反向输出技术架构和开发规范

如果有可参考的前端源码，可以直接让 claude code 扫描现有代码工程，反向整理出技术栈、核心依赖、项目目录、布局方式、主题样式、开发规范、构建流程等信息，生成到新项目目录的 claude.md 文档内，再结合公司内部要求手动调整一版。

如果没有可参考的项目代码，最好先找开发同事了解他们使用的技术栈和版本信息。比如，需要先确认企业内部或目标系统使用的是 vue 还是 react？如果是 vue，还要确认是版本是2 还是 3，还需要了解使用的 UI 样式库是哪个，因为这一步会直接决定生成原型的整体风格更偏向管理系统还是个人系统，也会影响后续移植和融合的难易程度。

最好还是要到源码或者让技术人员帮忙准备一份markdown格式的技术说明书，然后让模型反向整理后，输出到新项目目录下的claude.md文件里。

2.2、为claude.md添加项目信息

有了技术约束后，就可以在 claude.md 的文件里补充项目背景、目标用户、用户故事、业务结构（一级业务）、菜单目录等概述性信息，用来指导 claude code 生成项目框架代码。

注意，这一步不需要把每个子模块或者子菜单的详细功能都写进去，只需要描述有哪些入口及这些模块解决什么实际问题即可。防止上下文过长导致技术约束被稀释。

2.3、生成前端项目框架代码

为了尽快看到前端演示效果，需要在项目目录下的 claude.md 中补一条约束：“目前项目处于原型阶段，为了便于快速演示，暂不考虑后端动态接口服务，所有交互数据均采用 mock 方案模拟。”

接下来，手动切换到 plan 模式，然后输入提示词：“按照 claude.md 的内容生成项目架构代码。动手之前如果还有不确定的信息，需使用 AskUserQuestion 工具像苏格拉底一样与我讨论，只有在充分收集信息后再执行。”

或者使用 superpower 的 /brainstorming 命令激活 skill。这个 skill 也会先主动收集必要信息，然后再规划、执行。

框架代码生成后，根据模型的提示安装依赖、启动服务。有时候模型也会主动把服务运行起来，供用户通过浏览器查看。

服务起来后，用户可以通过浏览器重点检查页面布局、风格样式、菜单目录是否符合要求。如果差异较小，可以通过新的对话做局部调整；如果差异较大，就需要有针对性地补充、修正 claude.md 的内容，必要时还可以补充参考系统截图，然后在新的目录里重新生成工程代码，或者先手动删除已生成内容后重来，直到抽卡满意为止。

不要吹毛求疵做到100%的一致，因为这样需要花费很多的时间。一般跑起来80%是满足需求的，或者说大框架没有错就可以，之后的补充都可以利用新对话对局部调整解决。

2.4、完成各菜单的功能设计

目前 claude.md 里已经融合了技术约束和业务说明概要。但是如果生成具体子模块的功能是还把需求都塞进 claude.md，势必会造成上下文爆炸，使生成的效果很差。所以此时需要用渐进式加载的方式来优化上下文。操作如下：

1、在项目根目录创建doc文件夹，专门用来放需求文档，目录参考：

doc/
    需求文档
        需求规范说明.md
        项目总需求.md
        模块一名称
        模块二名称

需求文档一开始不用拆得特别细，而是要根据模块复杂度，尝试按模块级>菜单级>页面级>按钮级逐步细化。你需要在实践过程中通过与模型交互，体会当前模型的能力边界，再根据它的上限决定怎么拆分，以及把内容写到多么细致。

2、简化 claude.md 的内容，让它成为需求入口，而不是把所有需求都塞进去。这时 claude.md 里关于业务部分只需保留项目概述，之前的项目背景、目标用户、用户故事、业务结构（一级业务）、菜单目录等业务内容，都可以迁移到 ./doc/需求文档/项目总需求.md 内。

注意：要在claude.md里增加需求文档的说明，可参考：

## 需求说明
- 详细需求文档在当前工程“doc/需求文档”目录下，当用户对功能进行讨论时，应优先加载对应文档，查看功能描述和约束条件。
- 但是，文档难免会有滞后性，当出现文档与代码冲突时，优先以最新的代码作为事实依据。 
- 用户和agent需尽量保持文档与代码的一致性。

3、具体功能模块开发顺序

  • 先编写markdown格式的prd文件，内容包括：用户故事、页面布局、交互逻辑、角色定义、节点设计（审批流）、功能描述等内容。

注意：不要想一口吃成一个胖子，把子模块的所有功能全写到一个文件里期望一把全生成。这样只会失控，应该像咱们做框架代码那样，每次只解决一层的问题，保证效果。保持模型可控，比做的快更重要。

  • 创建新的会话，@需求文档，声明式使用 /brainstorming skill 激活对应流程，先与模型进行讨论，然后在沟通中补全、纠正模型的信息。

  • 如遇复杂的布局问题，可以让模型在对话中以 ASCII 编码的方式为你画一个模拟页面，快速示意，快速调整。

  • 如需引入新的技术组件，可以开启一个新会话与模型讨论兼容性和影响，尽量不要在当前主会话里分叉讨论。

  • 如果当前会话里临时需要引用其他网站的内容，可以声明式让 claude code 启动 subagent 去调研，并直接返回结果。而不是在当前主会话里展开，这也是为了避免主会话分叉，防止上下文被污染。

  • /brainstorming 讨论结束后，待用户确认后，下一步会自动激活 /writing-plans skill 编写执行计划文件，然后询问你是否启动 /subagent-driven-development skill 进行子 agent 驱动开发。只要前面的讨论没有偏差，实现层面通常不会有太大问题。

  • /subagent-driven-development 开发过程中因为会按步骤执行计划，所以会主动提交 git commit，来保持可追溯性。

  • 计划完成后模型一般会启动一个前端服务，如果你在框架阶段已经启动了一个前端服务，此时就应该约束模型对于服务的管理行为。目前前端框架都是支持热加载的，也就是说全局只需要启动一个 Web 服务就可以满足新老功能的开发展示。所以，可以在项目目录下的claude.md内约束一下本地调试的行为：

### 本地开发调试原则
- 用户会使用独立的终端手动启动**前端服务**，所以每次完成任务后，不用主动启动新的服务。
- 为了方便agent查看系统日志进行debug，用户将日志文件存放到了 front-debug.log文件
- 用户启动命令如下，占用8080端口

```bash
npm run serve > front-debug.log
```

这样模型就不会在每次任务完成后新开一个 Web 服务了，但相应地你也得记得手动启动服务。你也可以让模型写一个启动脚本，在每次打开项目时双击运行。

同时，上述启动命令通过重定向符号把控制台日志写入项目目录下的文件。这样在调试中一旦出现错误，模型就可以通过日志文件自行debug了。

注意：记得将日志文件添加到git忽略文件里

  • 用户在浏览器里点击测试时，如果发现问题，可以直接在当前会话继续提交修改需求，因为之前的讨论里有充分的上下文信息。但有一个前提需要注意：一定要先看 claude-hub 显示的上下文占用情况，如果已经超过 70%，而要修改的问题又比较复杂，此时建议先使用 /compact 主动压缩后再继续；或者直接开启新的对话，使用 /systematic-debugging 激活skill辅助debug 。

因为不同的模型的上下文大小不同，70%不是一个绝对的值。实战中，需要根据模型的交互反馈，找到一个合适的值。同时主动压缩也会损失很多细节，如果要修改的bug不太依赖于之前的回答，优先采用新会话的方式效果更佳。

  • 沉淀错误：不要执着于一定要在一次对话里解决问题，因为会话可能会因为沟通分叉而被污染，被污染后，如果还一直尝试，可能会越走越远。实际使用中，大概尝试 3-5 步后如果没有任何进展，就可以考虑丢弃当前会话，在新的会话里换个方向重试。而对于那些需要多轮沟通才能解决的问题，个人需要反思：“是不是现有文档或代码里缺少必要的参考信息，模型才需要你通过多轮对话去补足，最后才找到正确解决办法？“，然后做出相应的调整，尤其需要把缺失的信息、解决思路，以及模型犯过的错误回写到需求文档里，避免它之后再犯。

如果是全局错误，可考虑更新到项目根目录下的claude.md或专门创建一个“错题本”文件，然后在claude.md内引用。

  • 虽然 claude code 支持检查点还原，但是个人更推荐使用 git 来管理版本，这样控制粒度更高。

  • 不需要硬上 worktree，因为在上述工作流里，人始终是 in the loop 的，需要随时纠正方向，不会有太多空闲时间。通常只有在模型写代码、跑测试时，人才能被短暂释放出来，而这段时间我一般会用来优化需求文档。虽然也可以要求模型在完成任务后同步更新文档，但文档内容最终还是要由人来兜底。如果这时还有空闲时间，最好选择与当前任务不相关的业务模块同步推进，这样能够尽量避免冲突。就算用了 worktree，如果同时操作同一个文件，后续合并时也很容易出问题。并且，新 worktree 里的代码也没法直接在原来的 Web 服务上通过浏览器测试，因为两边的文件是隔离的。

  • 通过上述方式，基本上就可以完成前端原型的设计与开发工作。在这套方法里，人的能力依然决定产品的好坏，因为在与模型沟通的过程中会有很多分支选择。虽然模型会给出第一顺位答案，但要找出最优解依然依赖使用者的经验和品味。

三、如何使用Claude Code 生成后端接口服务

原型完成后，就可以跟用户确认了，直到确定所有页面的功能和交互。接下来，就可以进行后端服务的开发了。

3.1、修改工程目录结构

  • 在根目录下创建frontend文件夹，把现有的前端代码移动进去
  • 在根目录创建backend文件夹，用来放之后生成的后端代码
  • 在frontend文件夹内创建一个新的claude.md文件，并将原来根目录下claude.md内关于前端代码目录、技术栈、开发规范、编码规范等约束，剪切到frontend下的claude.md文件里，实现记忆分层。
  • 重新在根目录的claude.md内维护最新的项目目录结构
## 项目目录结构
整个项目采用前后端分离设计
backend: 项目后端代码
frontend: 项目前端代码
docs: 项目所需文档（需求、接口、操作手册等说明类文档）

3.2、生成后端技术约束

进入 backend 文件夹，手动创建CLAUDE.md

参照前端架构约束的思路，如果你已经有可参考的项目代码，可以让 claude code 逆向整理出对应的技术栈、项目结构、编码规范、命名习惯等内容，并写回到 ./backend/CLAUDE.md 文件内。如果没有可参考的代码，最好先去找后端同事聊一聊。如果两者都没有，而且之后也没有迁移合并的需求，就可以直接进入下一步。

3.3、生成后端框架代码

新建会话，激活 /brainstorming skill，然后 @./backend/claude.md、@./doc/需求文档/项目总需求.md，让模型根据当前技术约束和总需求，参考前端代码工程，生成后端架构代码。

根据 brainstorming 的流程约束，模型会先和你讨论技术栈选型、数据库选型、是否引入中间件、接口返回规范、统一异常捕获原则、事务控制、锁机制（如果有对应需求场景）等内容。

如果之前有参考代码工程，这些信息在上一步通常都能被逆向出来。如果没有，模型也会给出第一顺位建议。此时的原则是：优先选择最常用、最普适、自己也最熟悉的技术栈，对于不懂的概念，最好单独开个新会话了解清楚后再决策。与模型对话时依然要避免分散主任务的注意力。

后端相比前端一般依赖的内容会多，生成之后一定要能够成功启动接口服务，完成一个测试接口（可以开个新会话让 claude code 编写并测试）。

后端服务启动后，记得让模型根据最新的项目代码、主对话中做的方向性的决策，更新回写到./backend/claude.md，让文档保持同步，更好的指导之后的业务代码生成。

3.4、完善子模块接口服务

新建会话，激活 /brainstorming skill，然后 @./backend/claude.md、@对应模块的需求文档，让模型参考前端工程（主要参考 mock 时创建的 api 文件，必要时可以手动 @ 上），生成后端对应 api 服务。

skill 内会按照 spec 流程引导用户完成讨论、计划、行动、测试、提交这几个阶段。这一步会生成数据库脚本，通常需要用户手动执行；如果不想手动执行，就把数据库用户名和密码配置到环境变量里，让模型自动处理。这一步最好不要把自己数据库的用户名和密码放进项目文件中，以免被误上传到代码仓库。。

后端在按模块干活时，最好按照领域模型进行划分，这样能够更好的保证生成的表结构、数据模型完整，以减少返工。

如果遇到功能较多的模块，可以尝试分阶段生成，比如先生成 API 文档、表结构、实体对象，然后再生成逻辑代码，此时也可借助 /Planning-with-files skill 进行长任务规划，这样哪怕到了下班的点plan还没有执行结束，明天重启电脑依然可以继续。

修改根目录下的 claude.md内容，优化本地调试原则：把前后端日志路径告诉模型，这样一旦出了 bug，它们就可以自己去 debug 了。尤其在后端调试阶段，经常需要在复杂业务流里打印顺序日志，如果模型不会看日志，就只能靠人肉翻出来再贴给模型，非常耗时。可参考的配置：

### 本地调试原则
- 用户会使用独立的终端手动启动前后端服务
- 前端服务启动命令如下，占用8080端口
```bash
cd frontend
npm run serve > ../logs/front-debug.log
```

- 后端服务使用maven启动,启动后会占用8888端口
```bash
cd backend
mvn spring-boot:run > ../logs/backend-debug.log 2>&1
```

- 在调试过程中不需要agent来管理服务的启停。但是在调试中如果需要重启服务需告知用户。
- 当agent需要debug信息时，可主动从./logs的backend-debug.log（后端）、front-debug.log（前端）内查找。

每完成一个模块，我会手动做 UAT 测试。这样不仅能发现功能缺陷，尤其能在测试过程中引发更多思考，比如业务流程能否优化、状态展示是否完整、业务是否连贯、交互是否简洁、概念是否冗余、校验规则是否充分、事务是否得到控制等。对于简单页面，我会让模型跑 e2e 测试，模型会通过 playwright 边截图边测试。

尽管原型阶段做了大量的设计与确认工作，但到了代码阶段，势必会出现新的想法、新的改动以及新的妥协，这个无论是古法编程还是vibe coding都是在所难免的。只要我们做好版本管理，遵循“先思考、再规划、后执行”的思路，产品就会被一点点打磨的更好的。

四、交付系统后，我看到的下一阶段

通过上面的过程你会发现，为了减少幻觉，人需要一直在 loop 里充当指挥和监工，几乎每个动作都要加约束。这样虽然能最大限度减少幻觉，但人也会成为速度瓶颈。目前以我的精力，同时跟进 2-3 个会话已经到上限了。

但我们也能看到，网上有些高手已经在用 Teams 或多个编程工具协作，实现从需求、功能到部署的全链路 AI 自动化。我觉得他们肯定是通过了某些验证拿到了正反馈，或者看到了某种能力，达成了信心，才会做出这种行为方式。要走到这一步，我觉得要先升级到最强的模型，比如：claude opus 4.7、gpt-5.3-codex。因为按照我现在使用 minimax 2.7 的表现来看，还不能完全放手。

记得刚开始用 cursor 做 vibe coding 的时候，我几乎不做约束，想到哪聊到哪，只会一味优化提示词，结果就是经常出错，连开一个稳定会话都很吃力。后来换成 kiro，用了 spec 之后成功率确实提升了不少，但冗长的流程也确实很慢。一个长任务跑完后，往往还要做很多局部优化，有时候跑了半天还没跑对，很打击人。会让人对结果没有信心，自然就不敢多开会话。后来换成 claude code + opus4.5-thinking，逐渐掌握了 claude code 的记忆机制，理解了上下文工程，安装了 mcp、skill，配置了 agent 和一些 rules 之后，基本能做到指哪打哪，信心一下就上来了，多会话也就自然而然的安排上了，所以信心决定行为。

后来账号被封了，然后国产coding模型也全面铺开了，就尝试使用国内模型了。目前表现是：任务能够完成，只是需要投入很多注意力。所以，如果你想探索更多可能性、释放自己的注意力，最好的模型依然是必须的；但如果你跟我一样，都是“花自己的钱买token”的普通牛马，那国产模型的性价比确实很高。
