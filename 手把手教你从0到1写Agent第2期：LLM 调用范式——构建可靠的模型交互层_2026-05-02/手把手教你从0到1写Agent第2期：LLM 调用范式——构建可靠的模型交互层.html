1. 为什么 LLM 调用层需要「封装」

1.1 从一段「裸调」代码开始

第1期我们写了一个最小化的 AgentLoop 骨架。在那个骨架里，调用 LLM 的代码只占了几行：

const result = await generateText({
  model: this.config.model,
  system: this.config.systemPrompt,
  messages: this.state.history,
});

这五行代码在原型阶段完全够用。但一旦 Agent 开始处理真实任务，问题就会一个接一个地暴露出来。让我用五个具体的场景来展示。

1.2 翻车现场：五个真实场景

场景1：网络抖动

Agent 正在执行一个六步的数据分析任务——查询数据库、汇总统计、生成趋势图、输出报告。第1步、第2步顺利完成。第3步调用 LLM 时，OpenAI 的服务器恰好经历了一次短暂的网络波动——请求超时了。

裸调代码的行为：直接抛出 NetworkError。Agent 崩溃。前两步的执行结果全部丢失。用户看着终端里的错误堆栈，不知道发生了什么。

这不是假设。任何调用外部 API 的系统都会遇到网络问题。区别在于：专业的系统会优雅地处理它，原型会直接崩溃。

场景2：频率限制

Agent 的表现不错，用户开了三个终端窗口同时运行不同的任务。第4步调用时，三个 Agent 同时向 OpenAI 发请求，触发了 API 的频率限制——API 返回 429 Too Many Requests。

裸调代码的行为：Agent 崩溃，用户只看到一个 429 状态码。它不知道「等几秒再试」就能恢复，也不知道「应该降低请求频率」。

场景3：上下文爆炸

Agent 执行到第8步，对话历史已经积累了超过 40 条消息。其中有几轮执行了 cat /var/log/nginx/access.log，返回了数千行的日志内容——每一次工具调用结果都被原样塞进了对话历史。下一次 LLM 调用时，输入 Token 已经悄悄超过了模型的上下文窗口上限。

裸调代码的行为：API 返回 context_length_exceeded。但此时 Agent 无法通过调用 LLM 来压缩上下文——因为连「帮我压缩一下这段对话」这条请求本身都超出限制了。

场景4：用户焦虑

Agent 正在处理一个复杂任务，第5步的 LLM 调用花了整整 12 秒——模型需要处理大量的上下文。用户盯着一个空白的终端，12 秒里没有任何反馈。

裸调代码的行为：用户不知道 Agent 是在「思考」还是「卡死」。很多用户会忍不住按 Ctrl+C，然后重试。但重试意味着之前的进度全部作废，一切从头开始。

场景5：成本失控

Agent 跑了一周的自动化任务。到了月底，账单显示 LLM API 费用远超预期。但没有人知道这些 Token 具体是怎么花的——哪个任务最费钱？哪一步的上下文最长？有没有重复调用浪费 Token 的情况？

裸调代码的行为：只有总账单，没有明细。成本完全是不透明的黑盒。

1.3 封装层的职责边界

上面五个场景，分别指向五个核心问题。它们不是 Agent 编排逻辑要解决的——编排层关心的是「什么时候调用 LLM」、「调用 LLM 之后走哪个分支」。这五个问题是 LLM 调用层自身的职责：

除了解决这些问题，封装层还有两个重要的工程价值：

统一接口：无论底层用的是 OpenAI、Anthropic、Google，还是未来可能出现的任何新模型，Agent 核心代码不需要改动。对于后续的模型选型实验、成本优化、灰度切换，这个抽象层是第一道防线。

关注点分离：Agent 的编排逻辑——ReAct 循环、任务规划、反思纠错——不应该和「怎么调 API」、「怎么处理网络错误」、「怎么统计 Token」这些底层细节搅在一起。把 LLM 调用层独立封装，每个模块各司其职，代码的可读性和可维护性会大幅提升。后续第4期引入 LangGraph 时，这一点会变得更加清晰——LangGraph 的节点内部，调用的是这个封装好的 LLMClient，不需要知道底层是 OpenAI 还是 Anthropic。

2. 对话历史：Agent 的「短期记忆」载体

在动手封装之前，需要先搞清楚一个更基础的问题：对话历史到底应该怎么存？

不管编排层用 LangGraph 还是自研循环，与 LLM 交互的基本单位都是「消息数组」。每一条消息有明确的角色和位置规则。这些规则不是 API 的「建议」，而是硬约束——违反规则 API 会直接返回 400 错误。

2.1 消息的四种角色

LLM API 的消息模型定义了四种角色。理解每种角色的语义，是正确管理对话历史的前提：

┌─────────────────────────────────────────────────────────┐
│                    消息角色体系                          │
├──────────┬──────────────────────────────────────────────┤
│ system   │ 系统级指令。                                  │
│          │ · 定义 Agent 的身份、能力和行为边界            │
│          │ · 必须放在消息数组的最前面                     │
│          │ · 在一次对话中保持不变                        │
│          │ · 用户看不见 system 消息                      │
├──────────┼──────────────────────────────────────────────┤
│ user     │ 用户输入。                                    │
│          │ · 用户的提问、指令、确认                      │
│          │ · Agent 不能修改 user 消息                    │
├──────────┼──────────────────────────────────────────────┤
│ assistant│ Agent 的回复。                                │
│          │ · Agent 的思考结论                            │
│          │ · 可以包含工具调用请求（第5期展开）            │
│          │ · 每一条都是 Agent「说过的话」                │
├──────────┼──────────────────────────────────────────────┤
│ tool     │ 工具执行结果。                                │
│          │ · 命令的输出、API 的返回值                    │
│          │ · 必须关联到具体的 tool_call_id               │
│          │ · 可以是纯文本或结构化数据                    │
│          │ · 第5期开始使用，本期只需知道存在             │
└──────────┴──────────────────────────────────────────────┘

2.2 消息序列的结构规则

这些角色不是随意排列的。消息序列必须遵从严格的顺序规则：

✅ 正确的消息序列：
[system] → [user] → [assistant] → [user] → [assistant] → ...

❌ 违反规则的序列：
[user] → [user]           
  → 两条 user 紧邻，大部分 API 会拒绝
[assistant] → [assistant] 
  → 两条 assistant 紧邻，API 直接返回 400
[tool] → [user]           
  → tool 消息必须紧跟在发起工具调用的 assistant 之后

这些规则背后有一个简单的心智模型：对话是交替进行的。用户说一句，助手回一句；用户再说一句，助手再回一句。system 是对话开始前的「规则设定」，tool 是助手「做事」后拿到的结果——拿到结果后，助手继续推理，然后回应用户。

2.3 对话历史管理的三个常见陷阱

陷阱1：无限增长

第1步：history = [system, user]
第2步：history = [system, user, assistant, user]
第3步：history = [system, user, assistant, user, assistant, user]
...
第20步：history = [system, user, assistant, user, assistant, ... 40条消息]

每一步调用 LLM 时，都把从对话开始到现在的完整历史传过去。这意味着：

  • • Token 消耗线性增长。第20步的调用成本可能是第1步的 5-10 倍。
  • • 长上下文导致 LLM 推理质量下降——LLM 倾向于关注开头和结尾的信息，中间部分容易被「忽略」（这就是著名的「迷失在中间」问题）。
  • • 接近上下文窗口上限时 API 直接报错，且此时无法通过调用 LLM 来压缩——因为压缩请求本身就超限了。

这个问题的完整解决方案将在第7-8期（上下文管理）展开。本期要做的，是打好 Token 统计的基础——在问题发生之前，至少能监测到它正在靠近。

陷阱2：工具返回结果的全量拼接

Agent 执行：cat /var/log/nginx/error.log

工具返回：一个 5000 行的日志文件内容。

如果直接把 5000 行日志作为 tool 消息加入 history：
→ 这次调用的 Token 消耗暴增
→ 后续每一步都要带着这 5000 行
→ 但这 5000 行里真正有价值的信息可能只有 3 行

处理方式是：对超长工具返回做截断或微摘要。这将在第8期（上下文压缩）中实现。

陷阱3：System Prompt 的位置和重复

❌ 错误：把 system prompt 塞在用户消息后面
[user: "帮我查天气"] + [system: "你是一个助手"] 
→ system 必须放在数组最前面，这个位置才有意义

✅ 正确：system 独立在最前面，只出现一次
[system: "你是一个助手"]
[user: "帮我查天气"]

⚠️ 注意：不要在每次 LLM 调用时重新发送 system 消息。
虽然功能正常，但浪费了 Prompt Caching 的机会（第3期会讲到）。
一次对话中，system 消息只发一次。

3. 流式输出：让 Agent「实时思考」

3.1 用户为什么不耐心

先做一个思维实验。

你在使用一个命令行 Agent，你输入了：「帮我分析一下这个 500MB 的日志文件，找出过去七天里所有的 5xx 错误，按小时统计频率，然后生成一个汇总表格。」

你的终端里，接下来的 8 秒钟什么也没有出现。白屏。光标闪烁。

你的心理活动大概是这样的：

  • • 第 2 秒：它在处理吗？
  • • 第 4 秒：网络断了？
  • • 第 6 秒：是不是卡死了？我要不要 Ctrl+C？
  • • 第 8 秒：刷——一大段完整的分析报告出现在屏幕上。

这就是非流式调用的问题。LLM 在服务端逐个 token 生成回复，但客户端要等到所有 token 都生成完、拼成完整文本，才一次性显示。这段等待时间里，用户处于完全的信息真空状态。

流式调用改变了这个体验。LLM 每生成几个 token，就立即通过 SSE（Server-Sent Events）推送到客户端：

非流式：
[LLM 内部生成 token1...token800] → [5秒后，一次性返回完整文本]

流式：
[生成 token1] → 推送 "我"          ← 50ms
[生成 token2] → 推送 "正在"        ← 55ms
[生成 token3] → 推送 "分析"        ← 60ms
[生成 token4] → 推送 "日志"        ← 65ms
...
[生成 token800] → 推送 "。"        ← 5000ms
[done] → 推送结束信号

用户体验完全不同：文字像人在打字一样逐字出现。用户能感受到 Agent 在工作，可以跟着推理的节奏阅读，也能在推理出错的早期就发现并介入。

3.2 对 Agent 系统的额外价值

对于 Agent 而言，流式输出不止是用户体验优化。它直接关系到可观测性。

在第1期我们强调过，Agent 的每一个循环包含四个步骤：思考、决策、行动、观察。其中「思考」这一步是 LLM 内部的推理过程。如果用户在 Agent 思考时看到的是空白屏幕，那 Agent 就是黑盒。

但如果你把流式输出接入 Agent 的推理阶段，用户看到的就是：

Agent 思考中：用户需要分析日志文件。这个文件 500MB，直接读取会超出上下文。
            我应该先用 split 命令分块，然后逐个分析...
Agent 决策：执行 split -l 10000 access.log chunk_
Agent 行动中...
Agent 观察：生成了 47 个 chunk 文件。
Agent 思考中：逐个分析 chunk 中的 5xx 错误...

Kimi Code 的「思考过程」展示就是这个思路的典型实践——它在执行操作前，流式展示推理，让用户实时理解「它在想什么」。这种透明性在 Agent 做关键决策时尤为重要。

3.3 流式调用的代码形态

Vercel AI SDK 对流式输出的封装很简洁。下面是流式和非流式的对比：

// 非流式调用（generateText）
import { generateText } from "ai";

const result = await generateText({
  model: openai("gpt-4o"),
  messages: [{ role: "user", content: "解释 Docker 分层存储的原理" }],
});
// result.text 包含完整回答，但要在所有 token 生成完才能拿到

// 流式调用（streamText）
import { streamText } from "ai";

const result = await streamText({
  model: openai("gpt-4o"),
  messages: [{ role: "user", content: "解释 Docker 分层存储的原理" }],
});

// 逐 chunk 消费——每个 chunk 到达时立即处理
for await (const chunk of result.textStream) {
  process.stdout.write(chunk);  // 立刻显示，不等待剩余 token
}

两者共享相同的 messages 格式和模型参数。切换只改函数名和消费方式。这为我们的 LLMClient 封装提供了统一的接口基础——同一个客户端，既可以 chat()（非流式，适合规划和反思阶段），也可以 streamChat()（流式，适合实时展示推理过程）。

4. Token 统计：Agent 的成本记账系统

4.1 为什么需要精确 Token 计数

Token 在 Agent 系统中扮演两个角色：计费单位和容量单位。追踪 Token 用量服务于三个目的：

  1. 1. 成本可见：知道每个任务、每一步花了多少钱，月底对账有据可查。
  2. 2. 容量预警：在 Token 接近上下文窗口上限之前发出预警。等到 API 返回 context_length_exceeded 就晚了——那时 Agent 已经无法调用 LLM 来完成压缩。预警必须提前，在还剩 20%-30% 空间时触发。
  3. 3. 性能诊断：识别哪些任务类型 Token 消耗异常高。是 System Prompt 写得太啰嗦？还是某个工具经常返回超长结果？还是对话循环的步数设置过高？

4.2 Token ≠ 字数

Token 是 LLM 内部的词汇单位，和字数没有简单的换算关系。不同语言、不同类型的文本，token 化行为差异很大：

内容类型
大约 Token 数
1 个英文字符
~0.25 token
1 个常见英文单词
~1.3 token
1 个中文字
~1.5-2 token
一段 1000 字的英文文章
~750 token
一段 1000 字的中文文章
~1500-2000 token
一段 JSON 结构化数据
token 密度高（花括号、引号、逗号各占 1 token）

「大约」这个词是关键——tokenizer 会把常见词汇编码为更短的 token（比如 "the" 是 1 个 token，但 "antidisestablishmentarianism" 是好几个），所以精确计数必须通过模型官方的 tokenizer 工具，不能靠简单的字符估算。

英文场景下用「字符数 ÷ 4」估算可能误差 10-20%。中文场景下这个误差可以到 50% 以上。对于成本分析和上下文上限判断，50% 的误差意味着不可靠——你以为是 60K token 的对话，实际可能已经接近 100K token，随时可能触发 API 错误。

4.3 基于 tiktoken 的精确计数器

// ============================================================
// 模块：Token 计数器（token-counter.ts）
// 职责：精确统计文本和消息数组的 Token 数量，为成本控制和
//       上下文预检提供数据基础
// 
// 技术选型：使用 tiktoken（OpenAI 开源的 tokenizer），
// 而非简单的字符估算。原因：
//   英文场景：字符估算误差 10-20%
//   中文场景：字符估算误差可达 50%+
//   代码场景：特殊符号和缩进的 token 化行为完全不可预测
//   tiktoken 与 GPT-4 计费 token 高度一致，误差 < 1%
// ============================================================

import { encoding_for_model, type TiktokenModel } from "tiktoken";

/**
 * 单次调用的 Token 统计结果
 */
interface TokenCount {
  /** 输入 Token 数（System Prompt + 对话历史） */
  inputTokens: number;
  /** 输出 Token 数（LLM 生成的回复） */
  outputTokens: number;
  /** 总 Token 数 */
  totalTokens: number;
}

/**
 * TokenCounter - 基于 tiktoken 的精确 Token 计数器
 * 
 * 使用方式：
 *   const counter = new TokenCounter("gpt-4o");
 *   const count = counter.count("这是一段中文文本");
 *   console.log(count); // 精确的 token 数
 * 
 * 关于"模型不同，tokenizer 不同"的问题：
 * 每个 LLM 模型的 tokenizer 实现有差异（GPT-4 和 Claude 计费方式不同）。
 * 对于精确计费，应该使用对应模型的官方 tokenizer。
 * tiktoken 覆盖 OpenAI 全系模型。如果需要兼容 Anthropic 或本地模型，
 * 可以在初始化时传入对应的 encoder 实例。
 */
export class TokenCounter {
  private encoder: ReturnType<typeof encoding_for_model>;

  /**
   * @param model - 模型标识符，用于选择对应的 tokenizer
   *                例如 "gpt-4o"、"gpt-4-turbo"、"gpt-3.5-turbo"
   */
  constructor(model: TiktokenModel = "gpt-4o") {
    this.encoder = encoding_for_model(model);
  }

  /**
   * 统计单条文本的 Token 数
   * 
   * @param text - 要统计的文本
   * @returns Token 数量。空字符串返回 0
   * 
   * 注意：返回值是「纯文本 Token 数」。
   * 实际 API 调用中，每条消息的结构（role 标记、分隔符等）
   * 还会额外消耗少量 Token（约 4-6 个/条消息）。
   * countMessages 方法已经包含了这个额外开销。
   */
  count(text: string): number {
    if (!text || text.length === 0) return 0;
    const tokens = this.encoder.encode(text);
    return tokens.length;
  }

  /**
   * 统计消息数组的总 Token 数（含结构开销）
   * 
   * @param messages - 标准格式的消息数组
   * @returns 所有消息（含结构开销）的 Token 总数
   * 
   * 设计考量：每条消息除了 content 本身，role 标记、分隔符等
   * 也消耗 Token。OpenAI 文档建议每条消息额外加约 4 Token。
   * 这里每条消息加 4 Token 的 padding，并在最后加 2 Token 的
   * 整体结构开销。这样可以确保估算值略高于实际值——
   * 在上下文窗口判断中使用「偏高」的估算更安全，
   * 不会出现"以为还够但实际已经超限"的情况。
   */
  countMessages(messages: Array<{ role: string; content: string }>): number {
    let total = 0;
    for (const msg of messages) {
      // 消息文本 + 每条消息的结构开销（约 4 token）
      total += this.count(msg.content) + 4;
    }
    // 整个数组的结构开销（约 2 token）
    return total + 2;
  }

  /**
   * 释放编码器资源
   * 
   * tiktoken 的编码器会占用少量内存。在长期运行的 Agent 服务中，
   * 如果不再需要计数器，调用此方法释放资源。
   */
  dispose(): void {
    this.encoder.free();
  }
}

5. 动手：构建生产级的 LLMClient

前面四节分别讨论了对话历史管理、流式输出、Token 统计和错误处理策略。现在把这些能力整合到一个完整的 LLMClient 类中。

5.1 完整代码

// ============================================================
// 模块：生产级 LLM 客户端（llm-client.ts）
// 职责：封装 LLM API 调用，提供流式输出、自动重试、Token 统计、
//       调用日志记录等生产级能力
// 
// 设计原则：
// 1. 统一接口——Agent 编排层不需要知道底层用的是 OpenAI 还是 Anthropic
// 2. 容错性——网络波动和临时限流不应导致任务崩溃
// 3. 可观测性——每次调用的 Token 消耗、耗时、错误都有记录
// 
// 架构位置：
// 本模块位于 Agent 技术栈的「LLM 调用层」。
// 上层：第4期引入的 LangGraph 编排层，通过 LLMClient 调用 LLM
// 下层：本期构建的 TokenCounter，提供精确的 Token 统计
// 
// 这个分层在第4期会变得更清晰——LangGraph 的 agentNode 内部
// 调用的是 LLMClient.streamChat()，不直接接触 API 细节。
// ============================================================

import { generateText, streamText, type CoreMessage, type LanguageModel } from "ai";
import { TokenCounter } from "./token-counter";

// ─── 类型定义 ────────────────────────────────────────────────

/**
 * LLM 客户端配置
 * 
 * 所有参数都有合理的默认值，调用方可以只传 model。
 * 后续各期会通过修改配置来适配不同场景——
 * 比如第7期会调整 timeout 以应对长上下文调用。
 */
export interface LLMClientConfig {
  /** LLM 模型实例。使用 Vercel AI SDK 的统一接口。
   *  切换模型只需传入不同的 provider：
   *    openai("gpt-4o") → anthropic("claude-sonnet-4-20250514")
   *  第20期会展示如何根据不同任务选择不同模型来优化成本 */
  model: LanguageModel;
  
  /** 最大重试次数。默认 3 次。
   *  设为 0 表示不重试（仅适用于测试环境）。
   *  超过 3 次重试通常意味着问题不是临时的，应该向上层抛出 */
  maxRetries?: number;
  
  /** 重试基础延迟（毫秒）。默认 1000ms。
   *  第 1 次重试等 ~1 秒，第 2 次 ~2 秒，第 3 次 ~4 秒。
   *  加上随机抖动后，实际等待时间有所浮动。
   *  指数增长给服务端更多恢复时间，避免「惊群效应」 */
  retryBaseDelay?: number;
  
  /** 单次调用超时（毫秒）。默认 60 秒。
   *  大部分 LLM 回复在 5-30 秒内完成。
   *  超过 60 秒通常意味着网络或服务端异常 */
  timeout?: number;
}

/**
 * 单次 LLM 调用的完整结果
 * 
 * 为什么要包一层，而不直接返回 AI SDK 的原始结果？
 * 1. 统一不同 provider 的返回格式（OpenAI 和 Anthropic 字段名不同）
 * 2. 附加调用元数据（耗时），方便可观测性分析
 * 3. 隔离 AI SDK 的 API 变更——如果 SDK 接口升级，只改这一层
 */
export interface LLMResponse {
  /** LLM 生成的完整文本 */
  text: string;
  /** 终止原因：
   *   "stop"——正常完成
   *   "length"——达到 maxTokens 限制（回复被截断）
   *   "tool_calls"——LLM 请求调用工具（第5期开始使用）
   *   "error"——调用失败（所有重试已用尽） */
  finishReason: string;
  /** Token 用量明细 */
  usage: {
    promptTokens: number;
    completionTokens: number;
    totalTokens: number;
  };
  /** 调用耗时（毫秒），从发起请求到收到完整回复 */
  duration: number;
}

/**
 * 流式输出事件
 * 
 * 不只是暴露文本流，也暴露开始和结束事件。
 * 这让消费方可以：
 * - 在收到第一个 token 时显示「思考中…」动画
 * - 在流结束时显示 Token 统计
 * - 在出错时立即通知用户而非等待超时
 */
export type StreamEvent =
  | { type: "token"; content: string }              // 新生成的文本 token
  | { type: "done"; response: LLMResponse }         // 流正常结束，附带完整统计
  | { type: "error"; error: Error };                 // 流出错

/**
 * 调用日志条目
 * 
 * 每次 LLM 调用（包括重试）都记录一条日志。
 * 这是第19期可观测性系统的基础数据源。
 * 
 * 注意：messages 只记录最后 5 条的内容预览，
 * 避免日志文件膨胀（完整对话历史可能数万字）
 */
interface CallLogEntry {
  timestamp: Date;
  model: string;
  messages: Array<{ role: string; contentPreview: string }>;
  response?: LLMResponse;
  error?: string;
  retryCount: number;
  duration: number;
}

// ─── 核心类 ──────────────────────────────────────────────────

/**
 * LLMClient - 生产级 LLM 调用封装
 * 
 * 这个类解决第1节提出的五个核心问题：
 * 
 * 问题 → 解决方案：
 * 1. 网络异常 → 自动重试 + 指数退避
 * 2. 频率限制 → 识别 HTTP 429，自动延长退避时间
 * 3. 上下文超限 → Token 预检 + 提前预警（完整压缩方案在第7-8期）
 * 4. 无实时反馈 → streamChat() 流式输出
 * 5. 成本不可见 → getUsageStats() 汇总统计
 * 
 * 使用示例：
 * ```typescript
 * const client = new LLMClient({ 
 *   model: openai("gpt-4o"),
 *   maxRetries: 3 
 * });
 * 
 * // 非流式（适合 Agent 的规划和反思阶段）
 * const response = await client.chat(messages);
 * 
 * // 流式（适合实时展示 Agent 的推理过程）
 * for await (const event of client.streamChat(messages)) {
 *   if (event.type === "token") process.stdout.write(event.content);
 * }
 * ```
 */
export class LLMClient {
  // 将可选配置转为必选（填充默认值），简化内部逻辑
  private config: Required<LLMClientConfig>;
  private tokenCounter: TokenCounter;
  private callLog: CallLogEntry[] = [];

  constructor(config: LLMClientConfig) {
    this.config = {
      model: config.model,
      maxRetries: config.maxRetries ?? 3,
      retryBaseDelay: config.retryBaseDelay ?? 1000,
      timeout: config.timeout ?? 60000,
    };
    this.tokenCounter = new TokenCounter();
  }

  // ─── 公开方法 ──────────────────────────────────────────

  /**
   * 标准（非流式）调用
   * 
   * 在以下 Agent 场景中使用：
   * - 规划阶段：Agent 需要拿到完整的任务拆解计划再做下一步
   * - 反思阶段：Agent 需要完整的失误分析才能存入长期记忆
   * - 上下文压缩：需要完整的历史摘要而非逐字输出
   * 
   * 不适合：
   * - 执行阶段展示给用户的推理过程 → 用 streamChat()
   * 
   * @param messages - 对话历史消息数组
   * @param options.system - 系统提示词（可选）。传入后自动放在数组最前面
   * @param options.maxTokens - 限制输出 Token 数（可选）。防止回复过长
   * @returns LLM 的完整回复，包含文本和元数据
   * @throws LLMClientError - 所有重试用尽后仍然失败时抛出
   */
  async chat(
    messages: CoreMessage[],
    options?: {
      system?: string;
      maxTokens?: number;
    }
  ): Promise<LLMResponse> {
    const startTime = Date.now();
    let lastError: Error | null = null;

    // 准备消息数组：确保 system 在最前面，不修改原始数组
    const fullMessages = this.prepareMessages(messages, options?.system);

    // ─── Token 预检 ───────────────────────────────────
    // 第一道防线：在浪费 API 调用之前检查上下文大小。
    // 
    // GPT-4o 的上下文窗口是 128K token。我们在 100K 时发出警告。
    // 这 28K 的余量是为了：
    //   - 给 LLM 的回复留出空间（maxTokens 通常 4K-16K）
    //   - 给 Token 计数的估算误差留出空间
    //   - 给 Agent 几轮对话的缓冲，而不是到极限才报警
    // 
    // 注意：这里只是预警。真正的压缩逻辑在第7-8期实现。
    // 如果确实超限，API 会返回 context_length_exceeded 错误，
    // 这个错误在 shouldRetry 中被识别为「不可重试」。
    const tokenEstimate = this.tokenCounter.countMessages(
      fullMessages.map(m => ({
        role: typeof m.role === "string" ? m.role : "unknown",
        content: typeof m.content === "string" ? m.content : JSON.stringify(m.content),
      }))
    );
    if (tokenEstimate > 100000) {
      console.warn(
        `[LLMClient] 上下文 Token 数较高（约 ${tokenEstimate.toLocaleString()}），` +
        `接近模型窗口上限。建议启用上下文压缩（第7期会详细实现）。`
      );
    }

    // ─── 重试循环 ───────────────────────────────────
    // 
    // 重试是「乐观策略」：假设大多数失败是临时的，
    // 等一等就能恢复。代价是增加延迟。
    // 
    // 不重试的情况由 shouldRetry() 判断：
    //   401/403（认证/权限错误）→ 重试没用，直接抛
    //   400（请求格式错误）→ 重试没用
    //   context_length_exceeded → 需要压缩，不是重试能解决的
    for (let attempt = 0; attempt <= this.config.maxRetries; attempt++) {
      try {
        // 带超时的 LLM 调用
        const result = await this.executeWithTimeout(
          generateText({
            model: this.config.model,
            messages: fullMessages,
            maxTokens: options?.maxTokens,
          }),
          this.config.timeout
        );

        // 组装统一的返回格式
        const response: LLMResponse = {
          text: result.text,
          finishReason: result.finishReason,
          usage: {
            promptTokens: result.usage?.promptTokens ?? 0,
            completionTokens: result.usage?.completionTokens ?? 0,
            totalTokens: (result.usage?.promptTokens ?? 0) + (result.usage?.completionTokens ?? 0),
          },
          duration: Date.now() - startTime,
        };

        // 记录成功日志
        this.logCall(fullMessages, response, attempt);
        return response;

      } catch (error) {
        lastError = error instanceof Error ? error : new Error(String(error));

        // 判断是否应该重试。注意这个判断在「是否还有重试次数」之前，
        // 因为永久性错误（如 401）即使还有重试次数也不应该继续
        if (!this.shouldRetry(lastError)) {
          // 不可重试的错误：立即终止，不消耗剩余重试次数
          break;
        }

        if (attempt >= this.config.maxRetries) {
          // 重试次数用完
          break;
        }

        // 计算退避延迟后重试
        const delay = this.calculateBackoff(attempt);
        console.warn(
          `[LLMClient] 调用失败（第 ${attempt + 1}/${this.config.maxRetries} 次重试），` +
          `${delay}ms 后重试... 错误：${lastError.message.slice(0, 100)}`
        );
        await this.sleep(delay);
      }
    }

    // ─── 所有尝试都失败了 ──────────────────────────
    // 
    // 这里选择直接抛出错误，而不是返回一个「看起来正常」的 Response。
    // 原因：让上层（Agent 编排层）明确知道调用失败了。
    // 编排层可以根据错误类型做不同处理——
    // 比如 context_length_exceeded 触发上下文压缩后重试，
    // 网络错误则暂停任务并通知用户。
    const errorResponse: LLMResponse = {
      text: "",
      finishReason: "error",
      usage: { promptTokens: 0, completionTokens: 0, totalTokens: 0 },
      duration: Date.now() - startTime,
    };
    this.logCall(fullMessages, undefined, this.config.maxRetries, lastError?.message);
    
    throw new LLMClientError(
      `LLM 调用失败，已重试 ${this.config.maxRetries} 次：${lastError?.message ?? "未知错误"}`,
      lastError
    );
  }

  /**
   * 流式调用
   * 
   * 在以下 Agent 场景中使用：
   * - Agent 执行阶段：实时展示思考过程（Thought），让用户知道它在想什么
   * - 工具调用前的推理：用户可以看到 Agent 为什么选择调用某个工具
   * - 生成最终回答：用户体验更像「对话」而非「等待加载」
   * 
   * 参考 Kimi Code 和 Claude Code 的思考过程展示——
   * 它们都使用流式输出来实现「透明推理」。
   * 
   * @param messages - 对话历史
   * @param options - 同 chat() 方法
   * @returns AsyncGenerator，逐 token 产出事件
   */
  async *streamChat(
    messages: CoreMessage[],
    options?: {
      system?: string;
      maxTokens?: number;
    }
  ): AsyncGenerator<StreamEvent> {
    const startTime = Date.now();
    const fullMessages = this.prepareMessages(messages, options?.system);
    let fullText = "";

    try {
      const streamResult = await streamText({
        model: this.config.model,
        messages: fullMessages,
        maxTokens: options?.maxTokens,
      });

      // 逐 token 产出——每个 chunk 到达后立即 yield
      for await (const chunk of streamResult.textStream) {
        fullText += chunk;
        yield { type: "token", content: chunk };
      }

      // 流结束后，获取最终统计
      const finishResult = await streamResult.finishReason;
      const usage = await streamResult.usage;

      const response: LLMResponse = {
        text: fullText,
        finishReason: finishResult ?? "stop",
        usage: {
          promptTokens: usage?.promptTokens ?? 0,
          completionTokens: usage?.completionTokens ?? 0,
          totalTokens: (usage?.promptTokens ?? 0) + (usage?.completionTokens ?? 0),
        },
        duration: Date.now() - startTime,
      };

      this.logCall(fullMessages, response, 0);
      yield { type: "done", response };

    } catch (error) {
      const err = error instanceof Error ? error : new Error(String(error));
      this.logCall(fullMessages, undefined, 0, err.message);
      yield { type: "error", error: err };
    }
  }

  /**
   * 获取累计 Token 用量统计
   * 
   * 用途：
   * - Agent 任务完成后，展示本次任务的 Token 消耗概览
   * - 成本监控：按天/按周聚合 Token 用量
   * - 性能诊断：对比不同任务的 Token 效率
   * 
   * 第19期（可观测性）会基于此方法构建更完整的监控面板
   */
  getUsageStats(): {
    totalTokens: number;
    totalCalls: number;
    averageTokensPerCall: number;
  } {
    const completedCalls = this.callLog.filter(entry => entry.response);
    const totalTokens = completedCalls.reduce(
      (sum, entry) => sum + (entry.response?.usage.totalTokens ?? 0),
      0
    );
    return {
      totalTokens,
      totalCalls: this.callLog.length,
      averageTokensPerCall: completedCalls.length > 0
        ? Math.round(totalTokens / completedCalls.length)
        : 0,
    };
  }

  /**
   * 获取调用日志（用于调试和可观测性分析）
   */
  getCallLog(): ReadonlyArray<CallLogEntry> {
    return this.callLog;
  }

  // ─── 私有方法 ──────────────────────────────────────────

  /**
   * 准备消息数组
   * 
   * 确保 system 消息在最前面，且不修改原始输入数组。
   * 不产生副作用很重要——Agent 编排层保持对对话历史的完全控制，
   * LLMClient 不私自修改它。
   */
  private prepareMessages(
    messages: CoreMessage[],
    systemPrompt?: string
  ): CoreMessage[] {
    const prepared = [...messages];

    // 如果传入了 system prompt，且消息数组第一个不是 system，
    // 则在最前面插入
    if (systemPrompt && prepared[0]?.role !== "system") {
      prepared.unshift({ role: "system", content: systemPrompt } as CoreMessage);
    }

    return prepared;
  }

  /**
   * 判断错误是否值得重试
   * 
   * 核心问题：这个错误是临时的还是永久的？
   * - 临时错误 → 重试（网络抖动、限流、服务暂时不可用）
   * - 永久错误 → 不重试，直接抛出（认证失败、权限不足、参数错误）
   * 
   * 判断方法：解析错误信息中的关键标识
   * - HTTP 429 / "too many requests" → 临时，且应延长退避
   * - HTTP 5xx → 临时，服务端故障
   * - HTTP 401/403 → 永久，认证/权限问题
   * - HTTP 400 → 永久，请求格式有问题
   * - "context_length_exceeded" → 永久（需要压缩，不是重试能解决的）
   * - "fetch failed" / ECONNREFUSED / ETIMEDOUT → 临时，网络问题
   * 
   * 未知错误：保守策略——重试一次。如果还失败，不再重试。
   */
  private shouldRetry(error: Error): boolean {
    const message = error.message.toLowerCase();

    // 网络层错误
    const networkErrors = [
      "fetch failed",
      "econnrefused",
      "etimedout",
      "enotfound",
      "network error",
      "abort",
    ];
    if (networkErrors.some(e => message.includes(e))) {
      return true;
    }

    // HTTP 429：限流是临时状态，需要重试
    if (message.includes("429") || message.includes("too many requests")) {
      return true;
    }

    // HTTP 5xx：服务端临时故障
    if (message.includes("500") || message.includes("502") || 
        message.includes("503") || message.includes("504")) {
      return true;
    }

    // HTTP 401/403：认证/权限错误，重试没有意义
    if (message.includes("401") || message.includes("403")) {
      return false;
    }

    // HTTP 400：请求格式错误，重试不会改变格式
    if (message.includes("400") && !message.includes("429")) {
      return false;
    }

    // 上下文超限：需要压缩上下文，重试不会解决
    if (message.includes("context_length_exceeded") || 
        message.includes("maximum context length")) {
      return false;
    }

    // 未知错误：保守重试（不重试可能错过可恢复的故障）
    return true;
  }

  /**
   * 指数退避 + 随机抖动
   * 
   * 公式：delay = baseDelay × 2^attempt + random(0, 1000)
   * 
   * 为什么是指数而非固定间隔？
   * 当大量请求同时失败（比如服务故障），固定间隔重试会让所有请求
   * 在相同时间点再次同时涌向服务端——这就是「惊群效应」。
   * 
   * 指数退避让各请求的重试时间逐渐分散。
   * 随机抖动让这种分散更加自然。
   * 
   * @param attempt - 当前是第几次重试（从 0 开始）
   */
  private calculateBackoff(attempt: number): number {
    const exponentialDelay = this.config.retryBaseDelay * Math.pow(2, attempt);
    const jitter = Math.random() * 1000;
    return exponentialDelay + jitter;
  }

  /**
   * 带超时的 Promise 执行
   * 
   * 原理：Promise.race——谁先完成就用谁的结果。
   * 
   * 注意：JavaScript 的 Promise 不能被真正「取消」。
   * 如果 timeout 先触发，原 Promise 仍在后台执行，
   * 只是调用方不再等待它的结果。这对于 LLM API 调用没有副作用——
   * 服务端会正常完成请求，只是客户端不再消费。
   */
  private async executeWithTimeout<T>(
    promise: Promise<T>,
    timeoutMs: number
  ): Promise<T> {
    if (timeoutMs <= 0) return promise;

    return Promise.race([
      promise,
      new Promise<never>((_, reject) =>
        setTimeout(
          () => reject(new Error(`LLM 调用超时（${timeoutMs}ms）`)),
          timeoutMs
        )
      ),
    ]);
  }

  /**
   * 记录调用日志
   * 
   * 设计考量：
   * - messages 只保留最后 5 条消息的角色和内容预览（前 50 字）
   *   完整对话历史可能上万字，全量存储会导致日志膨胀
   * - 日志保存在内存中（this.callLog）。对于长时间运行的服务，
   *   第19期会扩展为持久化存储（文件/数据库）
   */
  private logCall(
    messages: CoreMessage[],
    response?: LLMResponse,
    retryCount: number = 0,
    error?: string
  ): void {
    this.callLog.push({
      timestamp: new Date(),
      model: this.config.model.modelId ?? "unknown",
      messages: messages.slice(-5).map(m => ({
        role: typeof m.role === "string" ? m.role : "unknown",
        contentPreview: typeof m.content === "string"
          ? m.content.slice(0, 50) 
          : "[非文本内容]",
      })),
      response,
      error,
      retryCount,
      duration: response?.duration ?? 0,
    });
  }

  /**
   * 异步延迟工具
   */
  private sleep(ms: number): Promise<void> {
    return new Promise(resolve => setTimeout(resolve, ms));
  }
}

// ─── 专用错误类 ────────────────────────────────────────────

/**
 * LLMClientError - LLM 调用层的专用错误类型
 * 
 * 为什么要自定义错误类，而不是用通用的 Error？
 * 1. Agent 编排层可以用 instanceof 精确捕获 LLM 调用错误
 * 2. 携带原始错误（cause），方便诊断
 * 3. 与工具调用错误、沙箱执行错误等区分开，
 *    各层做各层的错误处理，不混淆
 */
export class LLMClientError extends Error {
  public readonly cause: Error | null;

  constructor(message: string, cause?: Error | null) {
    super(message);
    this.name = "LLMClientError";
    this.cause = cause ?? null;
  }
}

5.2 使用示例

// ============================================================
// 使用示例：LLMClient 的基本用法
// 
// 后续各期会在 Agent 编排、工具调用、上下文管理等场景中
// 频繁使用 LLMClient。这里先展示最基础的两种调用模式。
// ============================================================

import { LLMClient } from "./llm-client";
import { openai } from "@ai-sdk/openai";

async function demoLLMClient() {
  // 1. 初始化客户端——只需要传模型，其余用默认值
  const client = new LLMClient({
    model: openai("gpt-4o"),
    maxRetries: 3,           // 临时故障自动重试 3 次
    retryBaseDelay: 1000,    // 基础退避 1 秒
    timeout: 30000,          // 超过 30 秒视为失败
  });

  // 2. 非流式调用——适合需要完整结果再继续的场景
  console.log("=== 非流式调用 ===");
  const response = await client.chat(
    [{ role: "user", content: "用三句话解释什么是 Docker" }],
    { system: "你是一个技术专家，回答简洁准确，每句话不超过 40 字" }
  );
  console.log(`回复：${response.text}`);
  console.log(`Token 消耗：${response.usage.totalTokens}`);
  console.log(`耗时：${response.duration}ms\n`);

  // 3. 流式调用——适合实时展示 Agent 推理过程
  console.log("=== 流式调用 ===");
  process.stdout.write("Agent: ");
  for await (const event of client.streamChat(
    [{ role: "user", content: "列出 Node.js 的五个核心模块并简要说明其作用" }],
    { system: "你是一个 Node.js 专家，回答简洁专业" }
  )) {
    if (event.type === "token") {
      // 每个 token 到达时立即显示
      process.stdout.write(event.content);
    } else if (event.type === "done") {
      // 流结束，显示统计
      console.log(`\n\n[Token: ${event.response.usage.totalTokens.toLocaleString()}, 耗时: ${event.response.duration}ms]`);
    } else if (event.type === "error") {
      console.error(`\n流式调用出错：${event.error.message}`);
    }
  }

  // 4. 查看累计 Token 用量
  console.log("\n=== 累计 Token 用量 ===");
  const stats = client.getUsageStats();
  console.log(`总调用次数：${stats.totalCalls}`);
  console.log(`总 Token 消耗：${stats.totalTokens.toLocaleString()}`);
  console.log(`平均每次调用：${stats.averageTokensPerCall.toLocaleString()} tokens`);
}

6. 本期的定位——以及它不做什么

到这里，你已经拥有了一个生产级的 LLM 调用层：自动重试、流式输出、Token 统计、调用日志、错误分类。这五行代码：

const result = await generateText({
  model: this.config.model,
  messages: this.state.history,
});

已经成长为 300 行代码、涵盖五种异常场景的 LLMClient 类。

但你可能在想：「有了 LLMClient，是不是就可以开始写 Agent 了？」

确实是——但还不够完整。

LLMClient 解决了「如何可靠地调用 LLM」的问题。它是一个优秀的调用层。但从下一期开始，我们将面对一个不同性质的问题：如何编排 LLM 调用和工具执行的循环？如何管理 Agent 的对话状态？任务执行到一半崩溃了，怎么从断点恢复？

这些问题属于编排层的职责。在第4期，我们会引入 LangGraph——一个专注于 Agent 状态管理和流程编排的框架。不是说用它来替代 LLMClient，而是让它坐在 LLMClient 之上：

┌─────────────────────────────────┐
│        LangGraph 编排层          │  ← 第4期引入
│  "什么时候调用LLM？调完走哪个分支？" │
│  "状态怎么存？断了怎么恢复？"       │
└────────────┬────────────────────┘
             │ 调用
┌────────────▼────────────────────┐
│        LLMClient 调用层          │  ← 本期构建
│  "怎么可靠地调LLM？流式怎么推？"    │
│  "Token用了多少？出错怎么重试？"    │
└────────────┬────────────────────┘
             │ 调用
┌────────────▼────────────────────┐
│     OpenAI / Anthropic API      │
└─────────────────────────────────┘

LLMClient 是 Agent 的「语言中枢」——它负责与 LLM 进行稳健的通信。
LangGraph 是 Agent 的「大脑皮层」——它决定什么时候调用语言中枢，什么时候执行工具，什么时候应该停下来反思，状态应该怎么持久化。

在第3期，我们先完成 Prompt 工程的设计——学会如何定义 Agent 的身份、能力和边界。然后第4期，用 LangGraph 把 LLMClient 和 System Prompt 组合起来，让 Agent 真正「活起来」。
