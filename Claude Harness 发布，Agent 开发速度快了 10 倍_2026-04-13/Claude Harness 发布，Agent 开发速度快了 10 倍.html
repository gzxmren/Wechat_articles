Claude Harness 发布，Agent 开发速度快了 10 倍

从0到AI，每天为你拆解硅谷最前沿的 AI 创业实战与商业洞察。全文约 2500 字，建议收藏慢读。

从「概念」到「产品」，Anthropic 只用了一个季度

2026 年初，OpenAI 和 Anthropic 几乎同时发表了关于 Harness Engineering 的深度文章。

LangChain 工程师 Viv 给出了一个流传至今的公式：Agent = Model + Harness。模型提供智能，Harness 让这个智能能真正投入生产。

道理所有人都懂。但问题是，搭建一套生产级的 Harness，你需要处理沙箱执行、状态管理、权限控制、端到端追踪——这些基础设施工作可能耗费数个月。大多数团队在这里就卡住了：有想法、有模型，但就是到不了"上线"这一步。

昨天，Anthropic 把这个问题一刀切掉了。

他们正式发布了 Claude Managed Agents——一套完整的云托管 Agent API 套件。你定义 Agent 要做什么、用什么工具、边界在哪里；Anthropic 的基础设施负责让它真正跑起来。

为什么把 Agent 搬上生产这么难？

这个问题值得先认真回答一遍。

很多人以为调用 Claude 的 API、搭一个对话循环，就算是 Agent 了。在 Demo 阶段确实够用。但一旦进入生产，这个认知就会碰墙：

沙箱。Agent 要执行代码，不能在你的生产环境里直接跑，必须隔离。自己搭一套安全的代码执行沙箱，光这一项就是几周的工程工作。

长会话。一个真正有用的 Agent 任务——比如审查合同、生成财务报告、完成代码重构——可能需要跑数小时。中途断连怎么办？进度丢失怎么办？这些都需要检查点机制和状态持久化。

凭证管理。Agent 要访问数据库、调用第三方 API，这些密钥必须加密存储、按需注入，不能硬编码在 prompt 里。

权限控制。Agent 不能有无限权限。它能访问什么文件？能调用哪些 API？能执行什么操作？这些作用域必须明确、可审计。

错误恢复。工具调用失败了怎么办？网络超时了怎么办？下游服务返回了意外格式怎么办？这些都要有健壮的重试和回退逻辑。

每一项单独拿出来都不是天大的事，但全部加在一起，就是数个月的基础设施工程，而且全是用户看不到的部分——上线之前没有任何可见的功能产出。

Anthropic 的判断是：这是重复劳动，我们来做一次，所有人受益。

Claude Managed Agents 给了开发者什么？

四个核心能力，每一个都直接对应上面列出的某个痛点。

生产级运行时。安全沙箱、身份验证、工具执行——全部由平台处理，开发者不需要自己搭。Agent 可以安全地执行代码、访问文件系统、调用外部服务，整个过程在隔离环境中运行。

长运行会话。Agent 可以自主工作数小时，进度和输出全部持久化。即使网络连接断开，任务不会丢失，可以从最近的检查点继续。这让 Agent 能够处理真正复杂的任务，而不只是几秒钟的简单请求。

多 Agent 协调。主 Agent 可以生成子 Agent 并行处理不同的子任务，最后汇总结果。复杂任务可以被分解成并行流水线，大幅缩短完成时间。

可信治理。作用域权限、身份管理、执行追踪，全部内置。每一次工具调用都有记录，每一步操作都可审计。这是企业最担心的问题——AI 越权访问敏感数据——在这里有了系统性的解决方案。

在 Anthropic 的内部基准测试中，针对结构化文件生成任务，Managed Agents 相比标准的提示循环，任务完成率提升了最高 10 个百分点，而且提升幅度在最复杂的问题上最显著。

三个设计决策，揭示 Anthropic 的真实想法

发布公告之外，Anthropic 同步发布了一篇关于 Harness Engineering 的技术文章，讲他们在设计这套系统时做出的关键选择。读完之后，有几个判断值得单独拿出来说。

判断一：给 Claude 用熟悉的工具，而不是专用工具。

Anthropic 没有为 Agent 设计一套新的、专用的工具集。他们给 Claude 的就两样东西：bash 工具和文本编辑器。

这个选择背后有清晰的逻辑。Claude 在 SWE-bench Verified（软件工程任务基准测试）上达到了 49% 的成绩，用的就是这两样工具。Claude 对这两个工具的理解最深、使用最熟练，而且会随着模型版本升级而持续改善。给 Claude 通用工具，让它自己组合出解决特定问题的模式——而不是为每种任务定制一套专用接口。

判断二：编排逻辑从框架转移到模型本身。

传统的 Agent 框架有一个隐含假设：每一次工具调用的结果，都必须经过模型的上下文窗口，才能决定下一步行动。

这样做有显著的成本和效率问题。如果工具调用的输出只是被传递给下一个工具，或者模型只需要关注其中很小的一部分，那么让整个输出占用上下文窗口就是浪费。

给 Claude 代码执行能力之后，它可以用代码表达工具调用之间的逻辑：哪些结果需要进入上下文、哪些可以直接 pipe 给下一步、哪些只需要关注某个字段。只有最终输出才进入上下文。编排的决策权从 harness 转移到了模型本身。

这个设计的副产品是：一个强大的编码模型，就是一个强大的通用 Agent。

判断三：只有需要边界的操作，才被提升为专用工具。

不是所有操作都应该走 bash 通道。某些操作——难以逆转的、需要用户确认的、需要呈现给用户选择的——有必要提升为独立的专用工具，这样 harness 才能在执行前拦截、检查、记录。

编辑工具里内置了过期检查：如果 Claude 上次读取文件之后，文件已经被修改，它不会直接覆盖，会先提示。外部 API 调用可以被包装成需要用户确认的模态框。这些都是通过专用工具来实现的安全边界。

真实案例：这些公司已经在用了

Notion，把 Claude 直接集成进了工作空间，让团队可以把任务委托给 Agent 处理。工程师用它交付代码，知识工作者用它生成网站和演示文稿。数十个任务可以并行运行，整个团队协作处理产出。

Sentry，让调试 Agent Seer 和补丁生成 Agent 配合工作。整个流程是：标记 bug → AI 生成补丁 → 自动打开 PR，一气呵成。工程总监 Indragie Karunaratne 说，Managed Agents 消除了维护定制 Agent 基础设施的持续运营开销。这套集成从构想到上线只用了数周。

Rakuten，在产品、销售、营销和财务四个部门同时部署了企业 Agent，每个专业 Agent 接入 Slack 和 Teams，员工可以直接分配任务、获取交付物。每个 Agent 的部署周期：一周。

Vibecode，做的是"从 prompt 到部署应用"。联合创始人 Ansh Nanda 说，在使用 Managed Agents 之前，搭建同等水平的 Agent 基础设施需要数周乃至数月；现在只需几行代码，速度快了至少 10 倍。

Anthropic 不再只是卖模型了

这个产品发布背后有一个更大的商业转变值得关注。

过去，Anthropic 的商业模式很清晰：卖 Claude API，按 token 计费。开发者调用 API，自己搭框架，自己处理基础设施。

现在，他们提供的是一个完整的 Agent 运行环境。沙箱、会话管理、权限控制，全部托管。这是云服务商做的事——提供计算资源和运行环境，只不过跑在上面的是 AI Agent，不是普通应用。

定价方式也印证了这个转变。除了标准的 token 费用，Managed Agents 另外按会话活跃时间收费：每小时 0.08 美元。这是典型的基础设施计费方式，类似云主机按运行时长收费。

从技术设计上，Anthropic 也为这个方向留了足够的扩展空间。Managed Agents 不强制某一套固定的编排逻辑，而是一个能容纳不同控制器的系统。Claude Code 可以跑在上面，针对特定任务优化的自定义控制器也可以。随着模型能力提升，整个系统可以持续演进，不需要推倒重来。

对自部署用户的意义

当然，Anthropic 提供的是托管方案。如果你对数据隐私有要求，或者不想把 Agent 的运行环境完全交给第三方，还有另一条路。

开源平台 OpenClaw 走的是相反的方向——把同等的基础设施能力打包成可以本地部署、完全自主控制的方案。Long-running 会话、多 Agent 协调、Skill 系统、内存管理，这些能力 OpenClaw 都有，区别在于你的数据不离开你的环境。

两条路代表不同的取舍：Managed Agents 给你极低的运维成本和极快的起步速度；自部署方案给你完整的数据主权和定制灵活性。具体选哪条，取决于你的场景和对"控制感"的优先级。

写在最后

Claude Managed Agents 发布，说明了一个大趋势：AI 基础设施正在产品化。

从 2022 年的"自己搭框架"，到 2024 年的"开源框架满天飞"，再到 2026 年的"官方托管接管基础设施"，这个演进轨迹很清晰。门槛在系统性地降低。

对开发者来说，最大的价值是省掉了从原型到生产之间那段看不见的基础设施工程。Notion、Sentry、Rakuten、Vibecode 的案例都在说同一件事：当基础设施不再是瓶颈，团队就能把精力放到真正重要的地方——设计 Agent 能做什么、怎么做、边界在哪里。

这不是"AI 改变一切"的泛泛之谈。而是具体的、可操作的能力，已经降落到了任何一个中小团队的手边。
