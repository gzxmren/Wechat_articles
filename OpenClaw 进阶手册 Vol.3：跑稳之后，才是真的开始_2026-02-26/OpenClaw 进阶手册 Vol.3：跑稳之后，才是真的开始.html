Vol.2 讲了让 OpenClaw 真正跑起来——Cron、Webhook、子 Agent、沙箱、Tailscale。如果你已经配好了，恭喜你，大多数人止步于此。

但真正的问题在后面。

三个月后你打开账单，发现多了几百块，想不明白哪里出了问题。或者某天早上发现早报已经三天没到了，去查 sessions_list，全是 completed——Agent 每天按时跑，只是什么输出都没到你。

第一章：成本控制（Tips 01–05）

账单超预期，大多数人的第一反应是「哪个模型太贵」。

往往不是模型的问题。

Tip 01｜OpenClaw 没有 runBudget，但有这三个替代手段

很多人在找 runBudget 或者类似「设个上限」的字段。这个字段不存在。

OpenClaw 控制成本靠另外三个：

第一个： runTimeoutSeconds

字段路径是 agents.defaults.subagents.runTimeoutSeconds。设了超时，子 Agent 跑超了自动停，不会无限消耗 token。

  1. {
  2.   agents:{
  3.     defaults:{
  4.       subagents:{
  5.         runTimeoutSeconds:120,// 2 分钟还没跑完就停
  6. }
  7. }
  8. }
  9. }

默认是 0，也就是永不超时。生产建议至少设一个：日常抓取任务 120 秒，复杂分析 600 秒。

第二个： /usage full

在 Telegram / WhatsApp 等消息 app 里发这条命令，OpenClaw 先回一条确认（ ⚙️Usagefooter:full.），之后每一条回复底部都会带 token 用量和估算费用。

不是一次性查询，是一个开关。用完了发 /usage off 关掉，或者 /usage tokens 只显示 token 数不显示费用。

两个注意点：

  1. 费用估算只在 API key 鉴权下才显示。如果你用的是 Anthropic OAuth 登录（Claude.ai 账号），只会显示 token 数，不显示费用——这是 OAuth 模式的限制，不是 bug。
  2. Control UI 看法不同。在 Gateway Dashboard 的网页界面用，footer 不会拼在消息底部，token 数据在左侧的 Usage 面板里查。

第三个： /status

在聊天里发，OpenClaw 直接回复当前 session 的状态卡片：用的什么模型、context 用了多少、上次回复消耗了多少 token（和费用，API key 模式下）。在 Control UI 和 Telegram 里都能用。

想看全局用量明细，用 CLI：

  1. openclaw status --usage   # 按 provider 分组显示用量和配额

这三个组合比「设预算上限」更灵活——不是限制你，是让你能看到钱花在哪里。

Tip 02｜子 Agent 默认继承主 Agent 的模型——这是隐性成本

sessions_spawn 出来的子 Agent，如果没有指定模型，默认用的是主 Agent 的模型。

主 Agent 跑 Opus，spawn 出 5 个子 Agent 并行处理数据，全部用 Opus。账单可以很轻松地翻倍。

修法是在配置里单独设子 Agent 的默认模型：

  1. {
  2.   agents:{
  3.     defaults:{
  4.       subagents:{
  5.         model:"anthropic/claude-sonnet-4-6",// 子 Agent 默认用 Sonnet
  6.         runTimeoutSeconds:120,
  7. }
  8. }
  9. }
  10. }

分级建议：

  • 数据抓取 / 格式转换 → openai/gpt-4o-mini 或 google/gemini-flash-2.5
  • 分析 / 摘要 → anthropic/claude-sonnet-4-6
  • 最终决策 / 复杂推理 → 主 Agent 自己来，不要 spawn

v2026.2.21 新增： channels.modelByChannel 可以给不同频道单独指定模型：

  1. {
  2.   channels:{
  3.     modelByChannel:{
  4.       telegram:"anthropic/claude-sonnet-4-6",// Telegram 私聊用 Sonnet
  5.       slack:"openai/gpt-4o-mini",// Slack 消息量大，用便宜的
  6. }
  7. }
  8. }

Telegram 私聊你会注意回复质量，Slack 群里的自动摘要换 gpt-4o-mini / Gemini 2.5 Flash完全够用。成本差距会比你想象的明显。

Tip 03｜maxConcurrent：两个不同的字段，别混淆

这两个字段名字像，用途完全不同：

  • agents.defaults.subagents.maxConcurrent（默认 8）：所有 Agent 的子任务共用一个全局并发通道，这个字段控制通道的上限
  • cron.maxConcurrentRuns（默认 2）：整个系统最多同时并行几个 Cron job

两个都是全局的，只是管的东西不同。前者管「子 Agent 跑多少个」，后者管「Cron job 跑多少个」。

早上 9 点有 3 个 Cron job 同时触发，系统默认最多跑 2 个，第三个等前面一个结束。这是合理的保护，防止瞬间并发成本飙升。

需要调整的时候：

  1. {
  2.   agents:{
  3.     defaults:{
  4.       subagents:{ maxConcurrent:3}// 全局子 Agent 并发上限，默认 8
  5. }
  6. },
  7.   cron:{ maxConcurrentRuns:1}// 同一时间只跑一个 Cron job
  8. }

Cron job 密集的情况，建议把 maxConcurrentRuns 控制在 1-2，成本更可预测。

Tip 04｜识别 token 消耗黑洞

账单超了不是模型问题，通常是这四种情况之一：

1. Context 里装了不该装的

Agent 拿到任务「总结这份日志文件」，把整个 1MB 日志读进来，但它实际只需要最后 50 行。

修法：在 task 里明确「只读 ~/.openclaw/logs/app.log 的最后 100 行，忽略 DEBUG 级别」。

2. 子 Agent 反复抓取同样的数据

主 Agent spawn 了 4 个子任务，每个都独立调了同一个 API 拉同一份数据。4 份相同的内容塞进 4 个 context。

修法：让一个子 Agent 先拉数据写文件，其他子 Agent 读文件。

3. Prompt 写得模糊，Agent 反复确认

task 写「帮我分析一下情况」，Agent 第一步就问「请问要分析什么」。每一轮确认都在消耗 token。这是最便宜的优化——把任务写清楚。

4. 截图分辨率没控制

browser_snapshot 截图默认 1200px 边长，全部进 context。调小到 800px，视觉任务够用，token 减少约 40%：

  1. {
  2.   agents:{
  3.     defaults:{
  4.       imageMaxDimensionPx:800
  5. }
  6. }
  7. }

两个高级选项（长 session 专用）：

如果 session 经常跑到 context 上限被截断，Opus 和 Sonnet 有 1M context beta 可以开（v2026.2.17）：

  1. {
  2.   agents:{
  3.     defaults:{
  4.       models:{
  5. "anthropic/claude-opus-4-6":{
  6. params:{ context1m:true}
  7. }
  8. }
  9. }
  10. }
  11. }

代价是每次调用的 context 窗口更大，成本也更高。长文档分析、长周期 session 才适合开。

另一个是 agents.list[].params.cacheRetention:"long"，对特定 Agent 开启长缓存，减少重复前缀的 token 开销。适合 task 描述很长、前缀每次都一样的 workflow。

Tip 05｜账单超预期的四个经典原因

快速检查清单，出账单异常先过一遍：

子 Agent 进入循环：task 没有终止条件，Agent 反复执行同一步骤。用 sessions_history 看 transcript，工具调用有没有明显重复。

Cron 任务失败重试：默认行为下，失败任务会重试。一个 job 失败了 3 次，就是 3 倍的 token 消耗。查 ~/.openclaw/cron/runs/ 下的日志，看有没有 retry 记录。

Browser 在反爬页面卡住：Agent 反复 browser_snapshot 但拿到的是 CAPTCHA 或登录页，每次 snapshot 消耗 token，没有产出。检查截图内容本身，而不只是看任务有没有完成。

大文件整个读进 context：Agent 处理文档，没有指定范围，把 100KB 全塞进去。固定方式是在 task 里明确范围和格式要求。

碰到账单异常，先对着这四条查一遍，大多数时候能找到原因。

第二章：故障模式（Tips 06–10）

不是报错就是不工作，后者才难处理。

最危险的故障是静默失败：Agent 跑了，Gateway 显示 completed，但什么都没发生。

Tip 06｜runTimeoutSeconds：子 Agent 超时之后发生了什么

超时停止不等于主 Agent 知道。

子 Agent 超了 runTimeoutSeconds，它停下来，但主 Agent 默认不会收到明确的错误信号。如果主 Agent 在等子 Agent 的结果，可能收到空值然后继续跑，输出废话——或者直接卡住等。

需要在 task 里预先写清楚超时的处理策略：

策略 A（允许不完整结果继续）：

  1. 如果某个子任务超时或没有返回结果，跳过，在最终输出里注明「[数据获取超时]」，
  2. 继续汇总其他已完成的结果。

策略 B（任意子任务失败就终止）：

  1. 所有子任务必须完成。任意一个超时或失败，停止整个任务，
  2. 推送「[任务失败：原因]」到Telegram。

早报用策略 A——少一条数据不影响整体价值。PR Review Bot 用策略 B——不完整的 review 比没有 review 更危险。

不写策略，Agent 会自己决定怎么处理超时。你不会满意它的默认选择。

Tip 07｜检测 Agent 原地循环

循环的外在表现：任务跑了很久，账单涨了，但没有产出。或者超时停了，但你不知道它在做什么。

诊断方法是直接让 Agent 查 transcript：

  1. 用sessions_history看一下[session_id]，
  2. 告诉我这个任务实际执行了哪些工具调用，是不是有重复的步骤。

如果发现循环，原因通常是两种：task 没有清晰的终止条件，或者 Agent 遇到错误之后不知道该停还是该继续。

预防方法 1：在 task 里写终止条件

  1. 完成以下任意一个条件就停止输出结果：
  2. ①找到目标数据并完成格式化输出
  3. ②尝试超过5次仍未获取有效数据
  4. ③已执行超过15个工具调用步骤

预防方法 2：设 runTimeoutSeconds 兜底

循环最多跑到超时截止，不会无限消耗。

Tip 08｜Fallback 模型：主模型挂了自动切换

字段是 model.fallbacks，真实存在：

  1. {
  2.   agents:{
  3.     defaults:{
  4.       model:{
  5.         primary:"anthropic/claude-sonnet-4-6",
  6.         fallbacks:["openai/gpt-4o","google/gemini-2.5-pro"],
  7. }
  8. }
  9. }
  10. }

Anthropic API 出故障，自动切到 GPT-4o；GPT-4o 也挂了，切 Gemini Pro。整个过程不需要你介入。

两个要注意的地方：

第一，不同模型的输出格式有差异。如果 SOUL.md 里要求严格的 Markdown 格式，切换后格式可能错乱。做过真实测试才能放心。

第二，fallback 模型的成本可能比主模型高——Gemini Pro 不一定比 Sonnet 便宜。Outage 期间消费的那几个小时，账单反映的是 fallback 模型的价格。

Tip 09｜Webhook 重复触发与幂等性

GitHub 和大多数外部服务，在 Webhook 没有收到 200 响应时会重试。同一个事件，可能触发 2-3 次。

最简单的去重方法是让 Agent 自己维护已处理记录：

  1. 收到Webhook之后，先检查~/.openclaw/data/processed-events.md 里
  2. 有没有这个事件的ID（PR编号或commit hash）。
  3. 有就跳过，没有就处理，处理完把ID写进文件（每行一个）。

不够优雅，但够用，而且完全不依赖外部存储。

如果事件量大，记得定期清理 processed-events.md，只保留最近 7 天的记录，防止文件无限增长。

Tip 10｜静默失败：跑了但没用

这是最难发现的故障模式。

sessions_list 显示 completed，Cron job 按时触发，一切看起来正常——但通知没到，数据没更新，什么输出都没有。

三种典型情况：

情况 1：Heartbeat 推送目标不稳定。配置了 target:"last"，昨天最后用的是群组，所以 Heartbeat 发到了群里；你以为会发到私聊，但没有。或者升级后 target 的解析行为变了，推送静默进了一个没人看的频道。

情况 2：子 Agent 输出了结果，主 Agent 解析失败（格式不匹配、字段名变了），把结果丢弃，继续输出了一份空摘要。

情况 3：Browser 截图成功，但截到的是登录页或者 CAPTCHA，Agent 误以为拿到了数据，汇总了一份没有实质内容的报告。

检测方法是成功心跳文件。在每个重要的 Cron job 末尾，让 Agent 往专用文件里写时间戳：

  1. 任务完成后，在~/.openclaw/data/heartbeat-[job-name].txt 里
  2. 写入当前时间（格式：YYYY-MM-DD HH:MM）。

再配一个监控 Cron job，每天检查这些文件，超过 25 小时没更新就推告警到 Telegram。

第三章：可观测性（Tips 11–15）

你的 Agent 在做什么，你得主动知道，而不是等出了问题再查。

Tip 11｜日志：在哪里，看什么

两个主要位置：

  • Session 日志： ~/.openclaw/agents/<agentId>/， .jsonl 格式
  • Cron 运行日志： ~/.openclaw/cron/runs/<jobId>.jsonl

Cron 日志默认限制 2MB / 2000 行，可以根据需要调大：

  1. {
  2.   cron:{
  3.     runLog:{
  4.       maxBytes:"5mb",
  5.       keepLines:5000,
  6. }
  7. }
  8. }

实际上你大多数时候不需要自己解析 .jsonl。直接让 Agent 帮你看：

  1. 用sessions_history查一下昨天凌晨1点那次竞品监控任务，
  2. 告诉我它实际做了什么、有没有遇到错误、最终输出了什么。

它比你自己翻文件快得多，也更容易发现问题。

Tip 12｜健康检查 Agent：每天早上告诉你 Gateway 状态

在 Cron 里加一个 8 点整的健康检查，Gateway 每天自己汇报一次状态：

  1. {
  2.   schedule:"0 8 * * *",
  3.   timezone:"Asia/Shanghai",
  4.   sessionTarget:"isolated",
  5.   payload:{
  6.     kind:"agentTurn",
  7.     message:"检查 Gateway 当前状态并汇报：1) ~/.openclaw/ 磁盘使用情况 2) 昨日所有 Cron 任务是否正常执行 3) 当前活跃 session 数量 4) 各频道连接状态。有异常直接说明，没有异常就一句话「一切正常」。",
  8. },
  9.   delivery:{ mode:"announce", channel:"telegram"},
  10. }

这个 job 有一个副作用：如果你哪天没收到它，就是一个信号——Gateway 出了问题，或者 Cron 调度停了。

本身作为监控失效，也是一种告警。

Tip 13｜sessions_history 不只是查进度

大多数人只在「这个任务跑到哪了」的时候用 sessions_history。它能做的更多：

发现 prompt 问题：Agent 反复在 transcript 里问类似的问题，说明 task 或 SOUL.md 没写清楚，它每次都在补充上下文。这种情况下优化 prompt 比换模型更有效。

分析决策质量：它做了什么工具调用、按什么顺序、最后输出了什么——这些比「有没有完成任务」更有诊断价值。

找静默失败的根因：从 transcript 最后几步倒查，一般能找到它在哪里误判了任务完成。

用法直接问 Agent：

  1. 用sessions_history查一下昨天的 PR ReviewBot任务，
  2. 告诉我它实际做了哪些步骤，有没有跳过某个PR，最终输出是什么。

Tip 14｜异常告警

配一个每小时跑一次的监控 Cron job，task 里写三个检查条件：

  1. 每小时检查以下三条：
  2. 1.过去1小时内是否有runTimeoutSeconds超时的session？
  3. 2.是否有某个Cron job连续失败超过2次？
  4. 3.~/.openclaw/目录磁盘使用是否超过总量的80%？
  5. 有任意一条触发，推Telegram并说清楚是哪个问题、大概是什么原因。
  6. 没有问题，不要发任何消息。

最后那条「没有问题不要发消息」是关键。

每小时推一条「一切正常」，两周之后你会开始忽略所有通知。真出了问题，那条告警淹没在正常消息里你看不到。

Tip 15｜Canvas 仪表盘：把运行状态做成一页

Agent 可以生成一个 Canvas 页面，把关键指标集中展示：

  1. 每天早上更新运行状态仪表盘：
  2. -本周Cron任务执行次数和成功率（表格）
  3. -本月token消耗按天趋势（折线图）
  4. -当前活跃session列表
  5. -最近3次失败任务的摘要
  6. 输出到Canvas，用A2UI渲染。

macOS 菜单栏打开 Canvas、iOS 节点打开 Canvas，都能直接看，不需要登录 Control UI。

适合把 OpenClaw 当「基础设施」跑的情况——你不需要每次主动去查，仪表盘在那里，有空瞄一眼。

第四章：状态管理（Tips 16–19）

多个 workflow 长期跑，数据要有地方存，状态要能跨 session 传。

没有规范，两个月后你自己也不知道哪个文件是干什么的。

Tip 16｜文件系统：Agent 持久化存储的规范

推荐的目录结构：

  1. ~/.openclaw/
  2. ├── openclaw.json                  ←配置文件
  3. ├── SOUL.md                        ←Agent规则
  4. ├── data/←所有 workflow 数据
  5. │├── competitor-snapshot.md
  6. │├── daily-report-history.md
  7. │├── pr-review-log.md
  8. │└── heartbeat-[jobname].txt ←心跳文件（Tip10）
  9. └── workspace/←工作区（代码、生成文档）

三个规范，不执行会出问题：

规范 1：数据文件带「最后更新时间」

文件第一行写 <!-- Updated: 2026-02-08 08:50 -->。Agent 读文件时能判断数据是不是过期的，不会拿着 3 天前的竞品数据当今天的来用。

规范 2：重要数据不要直接覆盖

让 Agent 先写临时文件（ competitor-snapshot-tmp.md），内容确认没问题再重命名替换。Agent 中途崩了，原始数据还在。

规范 3： workspace/ 和 data/ 用途分开

data/ 是 workflow 产生的运行数据， workspace/ 是你的工作文件。不要混放，维护清单的时候你会感谢这个规范。

Tip 17｜跨 session 传递状态

三种方式：

方式
适合
注意
文件读写
日常数据持久化
并发写同一文件会冲突（见 Tip 19）

两个 Agent 实时协作

大多数情况用文件就够了。文件简单、可查、出了问题容易 debug。

只有在两个 Agent 需要近实时通信时（比如一个 Agent 生成内容，另一个立即做格式转换并推送），才考虑 sessions_send。不要过度设计。

Tip 18｜HEARTBEAT.md：定期状态汇报的正确配置

Heartbeat 通过 HEARTBEAT.md 控制汇报内容和频率。

v2026.2.25 变更：旧的 Heartbeat DM boolean toggle 被替换为更明确的 heartbeat.directPolicy 字段：

  1. {
  2.   agents:{
  3.     defaults:{
  4.       heartbeat:{
  5.         every:"30m",
  6.         target:"last",
  7.         directPolicy:"allow",// 默认值，允许推 DM；改成 "block" 则禁止推私聊
  8. }
  9. }
  10. }
  11. }

也可以针对单个 Agent 单独配置： agents.list[N].heartbeat.directPolicy。

默认是 allow（允许推 DM）。如果你想限制 Heartbeat 只推频道不推私聊，显式设成 "block"。

一个实用建议：如果你的 Heartbeat target 设的是 "last"（最后一个活跃频道），可能不稳定——今天最后用的是私聊，明天最后用的是群组，推送目标每次都不一样。生产环境建议改成固定频道：

  1. {
  2.   agents:{
  3.     defaults:{
  4.       heartbeat:{
  5.         every:"30m",
  6.         target:"telegram",// 固定推 Telegram
  7.         to:"-100123456789",// 你的频道 ID
  8.         directPolicy:"block",// 不推私聊，只推频道
  9. }
  10. }
  11. }
  12. }

Tip 19｜防止多个 Agent 写同一个文件

什么时候会发生：竞品监控同时跑了 4 个子 Agent，每个都想更新 competitor-snapshot.md，并发写入，文件内容损坏。

OpenClaw 没有原生文件锁。解法是调架构：

  1. 子Agent A →写 snapshot-a.md
  2. 子Agent B →写 snapshot-b.md
  3. 子Agent C →写 snapshot-c.md
  4. 子Agent D →写 snapshot-d.md
  5. （以上并行）
  7. 主Agent→读取全部临时文件，合并，写入 competitor-snapshot.md
  8. （串行写入最终文件）

主 Agent 串行写入最终文件，没有竞争。

不要让多个子 Agent 直接写同一个目标文件。自己实现文件锁更容易出问题，调整架构比修锁逻辑省事得多。

第五章：长期运维（Tips 20–25）

配好了只是开始。怎么维护半年不出问题。

Tip 20｜用 git 管理 ~/.openclaw/ 配置

  1. cd ~/.openclaw
  2. git init

.gitignore 必须排除：

  1. .env
  2. logs/
  3. sessions/
  4. agents/
  5. cron/
  6. *.token
  7. credentials/
  8. auth-profiles.json

提交进 git 的： openclaw.json、 SOUL.md、 AGENTS.md、 data/ 下的模板文件（不是运行数据）。

这个习惯在什么时候救你：上周改了某个字段，今天 Cron 跑出了奇怪的结果， git diff 立刻看到改了什么。重大变更前打个 tag，出问题 git checkout 回之前的状态。

配置出了问题，第一个问题不是「哪里错了」，是「最近改了什么」。git 回答这个问题。

Tip 21｜月度 / 季度维护清单

每月第一个周一（10 分钟）：

  1. openclaw sessions cleanup       # 清 session 数据
  2. openclaw security audit         # 安全配置有没有问题

顺手看一眼本月 token 消耗和上月对比， ~/.openclaw/logs/ 目录大小有没有异常增长。

升级之前，先 dry-run（v2026.2.22 新增）：

  1. openclaw update --dry-run   # 看清楚升级了什么再决定要不要升

这是 v2026.2.22 加的，专门防止升上去才发现有 breaking change。

每季度（30 分钟）：

  • 有没有 Cron job 已经不跑了、没用了，直接删
  • SOUL.md 里的规则还符合实际吗（规则会随使用习惯过时）
  • 用的模型有没有新版本值得切
  • openclaw update--dry-run → 确认无异常 → openclaw update--channel stable → openclaw doctor

可选：开启自动更新（v2026.2.22，默认关闭）：

  1. {
  2.   update:{
  3.     channel:"stable",// 跟哪个 channel（stable / beta / dev）
  4. auto:{
  5.       enabled:true,
  6.       stableDelayHours:6,// 新版本发布后等 6 小时再自动升
  7. }
  8. }
  9. }

生产环境慎用——breaking change 在自动升级里是静默的。建议只在测试机开。

Tip 22｜从测试到生产的三个阶段

不要跳阶段。

阶段 1：本机测试（1-2 周）

  1. {
  2.   gateway:{ bind:"127.0.0.1"},// 只本地能访问
  3. }

sandbox 先关掉，方便调试。目标是 Cron 和 Webhook 逻辑跑通、没有明显的 token 泄漏。

阶段 2：内网稳定性验证（2-4 周）

迁移到专用机器（树莓派、Mac Mini、VPS），Tailscale serve 模式，只有你的设备能访问。开启 sandbox.mode:"non-main"。

这个阶段暴露配置问题——长期跑才会出现的内存增长、磁盘占用、Cron 时序问题，在这里出比在生产环境出代价低。

阶段 3：生产就绪

  • Tailscale funnel + 密码保护（需要公网访问时）
  • openclaw security audit 全部通过
  • 健康检查 Agent 上线（Tip 12）
  • 告警 Cron 上线（Tip 14）
  • ~/.openclaw/ 接入 git（Tip 20）

大多数配置问题在阶段 2 就能暴露。直接跳到阶段 3，出了问题排查成本更高。

Tip 23｜出问题时的排查顺序

按顺序来，找到问题层停下来：

  1. Step1: openclaw doctor
  2. →配置有没有错误或警告
  4. Step2: sessions_list
  5. →有没有卡住的 session、异常的状态
  7. Step3: sessions_history [session_id]
  8. →Agent在哪一步出的问题、做了什么工具调用
  10. Step4:~/.openclaw/cron/runs/
  11. →Cron job 的运行日志，有没有报错
  13. Step5: openclaw security audit
  14. →安全配置有没有在升级后被改掉

不要一次性把五步全跑完再看输出——你会淹没在信息里。找到问题层就停。

v2026.2.24 新增审计项：

openclaw security audit 现在会检测 Gateway 是否有多用户访问特征（ security.trust_model.multi_user_heuristic），发现时给出 WARNING 并建议：

  • sandbox.mode:"all"（所有 session 全部隔离，不只是 non-main）
  • 减少 SOUL.md 里的个人信息
  • 启用 dmPolicy:"pairing" 或 "allowlist"

只有你自己用的情况下，这条告警可以忽略。

Tip 24｜生产配置基线

可以直接复制的最小生产配置：

  1. {
  2.   agents:{
  3.     defaults:{
  4.       subagents:{
  5.         model:"anthropic/claude-sonnet-4-6",// 子 Agent 默认 Sonnet
  6.         runTimeoutSeconds:120,// 子任务 2 分钟超时
  7.         maxConcurrent:3,// 最多 3 个并行子任务
  8. },
  9.       imageMaxDimensionPx:800,// 截图 token 控制
  10.       sandbox:{ mode:"non-main"},// 非主 session 全部隔离
  11.       session:{
  12.         maintenance:{
  13.           mode:"enforce",
  14.           maxDiskBytes:"2gb",// 单 session 磁盘上限
  15. }
  16. },
  17. }
  18. },
  19.   cron:{
  20.     maxConcurrentRuns:2,// 同时最多 2 个 Cron job
  21.     sessionRetention:"24h",// Cron session 数据 24h 自动清理
  22. },
  23.   channels:{
  24.     telegram:{
  25.       dmPolicy:"pairing",// 未知来源需要配对验证
  26. }
  27. },
  28.   gateway:{
  29.     token:"用 openssl rand -hex 32 生成",// 替换成实际 token
  30. },
  31. }

每个字段为什么这么设，前面 23 条都讲过了。

Tip 25｜OpenClaw 的边界在哪里

不是功能介绍，是一个判断框架。

当前配置够支撑什么

1-3 个 Cron workflow，5-10 个并发子 Agent——完全没问题。这是绝大多数个人使用场景的全部需求。

5 个以上 Cron job 在同一时间窗口触发，把 cron.maxConcurrentRuns 控制好，防止突发成本。

什么时候需要 OpenClaw 之外的东西

跨 session 需要强一致性存储：引入 SQLite。Agent 能直接操作 SQLite 文件，不需要额外的服务。

需要向量搜索 / 语义 recall：OpenClaw 的 memory 系统支持接外部嵌入服务，查一下 openclaw memory 的配置文档。

多个 Gateway 实例协同：这是当前架构的真正边界。如果你到了这个阶段，说明你的场景已经不是「个人助手」了。

速查卡

  1. 第一章成本控制（01–05）→没有 runBudget /模型继承陷阱/ maxConcurrent 两个字段/ token 黑洞
  2. 第二章故障模式（06–10）→超时处理策略/循环检测/Fallback/Webhook幂等性/静默失败
  3. 第三章可观测性（11–15）→日志位置/健康检查Cron/ sessions_history /告警规则/Canvas
  4. 第四章状态管理（16–19）→文件目录规范/跨 session 三种方式/ HEARTBEAT 变更/并发写入
  5. 第五章长期运维（20–25）→ git 管理配置/月季维护清单/部署三阶段/排查顺序/配置基线
